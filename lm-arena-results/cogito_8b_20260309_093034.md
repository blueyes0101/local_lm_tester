## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
Here's a technical article about the differences between SQL and NoSQL databases:

SQL and NoSQL databases represent two distinct paradigms in data management, each with its own strengths and use cases. SQL databases, such as MySQL and PostgreSQL, are built on the relational model where data is stored in tables with well-defined schemas and relationships between tables are managed through complex queries. In contrast, NoSQL databases, including MongoDB and Cassandra, use various data models like document, key-value, wide-column, and graph databases, allowing for more flexible and schema-less data storage. This fundamental difference in data structure leads to distinct approaches in how data is queried and manipulated.

When it comes to scalability, SQL databases traditionally excel at handling complex queries and transactions but often struggle with horizontal scaling due to their rigid schema design. NoSQL databases, on the other hand, are designed for horizontal scalability, making them particularly suitable for high-traffic applications where rapid data growth and distributed processing are required. This scalability advantage, combined with their flexible schema, makes NoSQL databases ideal for applications like real-time analytics, social media platforms, and IoT systems where data structure and schema may evolve frequently.

The choice between SQL and NoSQL databases ultimately depends on the specific requirements of your application. SQL databases are preferred when you need ACID compliance, complex transactions, and predictable query performance, typically in enterprise applications, financial systems, or applications requiring complex joins and aggregations. NoSQL databases shine in scenarios where you need rapid scalability, flexible schema design, and high throughput with eventual consistency, such as in big data analytics, real-time web applications, and mobile backend services.

**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here's the iptables configuration with explanations for each rule:

```bash
# 1. Drop all incoming traffic by default
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT

# 2. Accept SSH (port 22) only from 10.0.0.0/24
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT

# 3. Accept HTTP (80) and HTTPS (443) from anywhere
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 4. Allow all outgoing traffic
iptables -A OUTPUT -j ACCEPT

# 5. Make the rules persistent
iptables-save > /etc/iptables/rules.v4
```

Let me break down each line:

1. `iptables -P INPUT DROP`: Sets the default policy for the INPUT chain to DROP, meaning no incoming packets will be accepted unless explicitly allowed by a rule.

2. `iptables -P FORWARD DROP`: Sets the default policy for the FORWARD chain to DROP, preventing any packets from being forwarded unless explicitly allowed.

3. `iptables -P OUTPUT ACCEPT`: Sets the default policy for the OUTPUT chain to ACCEPT, allowing all outgoing packets.

4. `iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT`: This line creates a new rule in the INPUT chain that:
   - Matches TCP packets (`-p tcp`)
   - Going to port 22 (SSH) (`--dport 22`)
   - Coming from the 10.0.0.0/24 network (`-s 10.0.0.0/24`)
   - Accepts these packets (`-j ACCEPT`)

5. `iptables -A INPUT -p tcp --dport 80 -j ACCEPT`: Accepts HTTP traffic (port 80) from any IP address.

6. `iptables -A INPUT -p tcp --dport 443 -j ACCEPT`: Accepts HTTPS traffic (port 443) from any IP address.

7. `iptables -A OUTPUT -j ACCEPT`: Allows all outgoing traffic by default.

8. `iptables-save > /etc/iptables/rules.v4`: Saves the current iptables configuration to a file, making it persistent across reboots. On Ubuntu, the configuration is stored in `/etc/iptables/rules.v4`.

Note: The `-A` flag means "append" the rule to the end of the chain, and `-j` stands for "jump to" the specified target (in this case, ACCEPT).

To apply these rules on a new system, you can use `iptables-restore` instead of `iptables-save`, or install the `iptables-persistent` package on Ubuntu which will automatically load the saved rules on boot.

- Tests the behavior when the API returns an error

**A:**
Let me explain the differences between Mock, Stub, and Fake, and then provide a Python example:

**Mock, Stub, and Fake:**

1. **Mock**: A complete fake object that implements all methods and can verify how it was used. Mocks record the interactions between the code and the mock object.

2. **Stub**: A simplified mock that only implements the methods needed for the test, without recording interactions. Stubs are used when you only care about the outcome of a method call, not how it was used.

3. **Fake**: A real implementation of an object or service, but one that doesn't interact with the real system. Fakes can be used instead of real services to make tests faster and more reliable.

**Example:**

Here's a Python example using `unittest.mock` that demonstrates mocking an external HTTP API call:

```python
import unittest
import requests
from unittest.mock import patch, Mock
from unittest.mock import MagicMock

class WeatherAPI:
    def get_weather(self, city):
        response = requests.get(f'http://api.weather.com/weather/{city}')
        return response.json()

class WeatherService:
    def __init__(self):
        self.weather_api = WeatherAPI()

    def get_temperature(self, city):
        try:
            weather_data = self.weather_api.get_weather(city)
            return weather_data['temperature']
        except requests.exceptions.RequestException as e:
            print(f"Error fetching weather: {e}")
            return None

class TestWeatherService(unittest.TestCase):
    def test_success(self):
        # Mock the requests.get method
        with patch('requests.get') as mock_get:
            mock_response = MagicMock()
            mock_response.status_code = 200
            mock_response.json.return_value = {'temperature': 25}
            mock_get.return_value = mock_response

            # Create service instance
            service = WeatherService()

            # Test the service
            temperature = service.get_temperature('London')

            # Assert the mock was called correctly
            mock_get.assert_called_once_with(f'http://api.weather.com/weather/London')

            # Assert the return value is correct
            self.assertEqual(temperature, 25)

    def test_api_error(self):
        # Mock the requests.get method to raise an exception
        with patch('requests.get') as mock_get:
            mock_get.side_effect = requests.exceptions.RequestException("API Error")

            # Create service instance
            service = WeatherService()

            # Test the service
            temperature = service.get_temperature('London')

            # Assert the return value is None
            self.assertIsNone(temperature)

    def test_connection_error(self):
        # Mock the requests.get method to raise a connection error
        with patch('requests.get') as mock_get:
            mock_get.side_effect = requests.exceptions.ConnectionError("Connection Error")

            # Create service instance
            service = WeatherService()

            # Test the service
            temperature = service.get_temperature('London')

            # Assert the return value is None
            self.assertIsNone(temperature)

if __name__ == '__main__':
    unittest.main()
```

This example demonstrates:

1. Using `Mock` to verify how the `WeatherAPI` class interacts with the external API.
2. Handling different types of API errors (success, API error, connection error).
3. Using `unittest.mock.patch` to replace the `requests.get` method with a mock version.
4. Verifying that the mock was called with the correct arguments using `assert_called_once_with`.
5. Testing the expected return values in different scenarios.

The tests ensure that the `WeatherService` class behaves correctly when interacting with the external API, and that it handles errors appropriately.

**A:**
Hermes mesaj kütüphanesi ile publisher/subscriber pattern'in nasıl çalıştığını açıklayayım:

1. Publisher Nasıl Mesaj Yayınlar?
```python
from hermes import Hermes
from hermes import Publisher
from hermes import BrokerConfig

# Publisher oluşturur
publisher = Publisher('publisher', 
                      BrokerConfig(broker='amqp://guest:guest@localhost:5672//', 
                                   heartbeat=60))
publisher.connect()
```

2. Subscriber Nasıl Dinler?
```python
from hermes import Subscriber

# Subscriber oluşturur
subscriber = Subscriber('subscriber', 
                        BrokerConfig(broker='amqp://guest:guest@localhost:5672//', 
                                     heartbeat=60))
subscriber.connect()
```

3. Bu Pattern'in Doğrudan Fonksiyon Çağrısına Göre Avantajları:

- **Mekanizasyon**:
  - Publisher/Subscriber modeli, kodun daha yapısal ve anlaşılır olmasını sağlar.
  - Kapsamlı işlemler yerine modüler parçalar halinde yazılabilir.

- **Gelişmiş İşlemler**:
  - Mesajları işlemek için farklı subscriber'lar farklı şekilde işleyebilir.
  - Mesajlar kaydedilebilir, işlenebilir veya başka sistemlere iletilebilir.

- **Güçlü Test Edimi**:
  - Publisher ve subscriber'lar ayrı ayrı test edilebilir.
  - Mesajların doğru şekilde iletildiğinden emin olunabilir.

- **Mümkün Hataları En Aza İndirme**:
  - Publisher ve subscriber'lar arasında bağımsız çalışan yapılar oluşturulur.
  - Sistemde bir hata oluşursa, sadece ilgili parça etkilenir.

Örnek Kullanım:
```python
# Publisher'da mesaj yayınlayan kod
@publisher.handler('mesaj_tipi')
def mesaj_yayinla(mesaj):
    publisher.publish('mesaj_kanalı', mesaj)

# Subscriber'da mesaj dinleyen kod
@subscriber.handler('mesaj_tipi')
def mesaj_al(mesaj):
    print(f"Alınan mesaj: {mesaj}")
```

Bu model, büyük ölçekli sistemlerde performans ve verimlilik artışı sağlar. Mesajları işlemek için farklı subscriber'lar farklı şekilde işleyebilir ve sistemde bir hata oluşursa sadece ilgili parça etkilenir.
