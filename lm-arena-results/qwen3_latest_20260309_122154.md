## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
SQL and NoSQL databases differ fundamentally in their approach to data storage, query mechanisms, and scalability. SQL databases, such as MySQL and PostgreSQL, are relational and rely on structured query language (SQL) to manage data stored in tables with predefined schemas. They enforce ACID (Atomicity, Consistency, Isolation, Durability) properties, ensuring data integrity for complex transactions. In contrast, NoSQL databases—like MongoDB or Cassandra—are non-relational and schema-less, allowing flexible, document-based or key-value data structures. This flexibility makes NoSQL suitable for handling unstructured or semi-structured data, though it often sacrifices strict consistency for scalability and performance.  

Scalability is another critical distinction. SQL databases typically scale vertically by upgrading hardware, which can become cost-prohibitive for large-scale applications. NoSQL databases, however, are designed for horizontal scaling, distributing data across multiple servers to handle massive datasets and high traffic loads. This makes NoSQL particularly effective for big data and real-time applications, where sharding and replication are common strategies. While SQL databases can achieve horizontal scalability with tools like sharding, the process is more complex and less seamless compared to their NoSQL counterparts.  

Use cases further highlight these differences. SQL databases excel in scenarios requiring complex queries, transactions, and strong consistency, such as financial systems or e-commerce platforms. NoSQL databases are ideal for applications with rapidly evolving data models, like social media platforms or IoT systems, where flexibility and horizontal scalability are prioritized. Developers often choose SQL for structured data and relational integrity, while NoSQL is preferred for unstructured data and high-throughput environments. Hybrid approaches, combining both database types, are also common to leverage their respective strengths. Ultimately, the choice depends on the specific requirements of the application, balancing consistency, scalability, and data structure needs.

## Question 5
**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here's how to configure `iptables` on an Ubuntu server according to your requirements, along with explanations for each line:

---

### **Step 1: Flush Existing Rules**
```bash
sudo iptables -F
```
**Explanation**:  
This command removes all existing rules from the `iptables` table to start fresh.

---

### **Step 2: Set Default Policies**
```bash
sudo iptables -P INPUT DROP
sudo iptables -P OUTPUT ACCEPT
sudo iptables -P FORWARD ACCEPT
```
**Explanation**:  
- `-P INPUT DROP`: All **incoming** traffic is **dropped by default**.  
- `-P OUTPUT ACCEPT`: All **outgoing** traffic is **allowed by default**.  
- `-P FORWARD ACCEPT`: Forwarding traffic (e.g., for NAT) is **allowed by default** (though not needed here).  

---

### **Step 3: Allow SSH (Port 22) from 10.0.0.0/24**
```bash
sudo iptables -A INPUT -s 10.0.0.0/24 -p tcp --dport 22 -j ACCEPT
```
**Explanation**:  
- `-A INPUT`: Append the rule to the `INPUT` chain.  
- `-s 10.0.0.0/24`: Match traffic **originating from** the subnet `10.0.0.0/24`.  
- `-p tcp`: Specify the protocol as **TCP**.  
- `--dport 22`: Match traffic **destined for port 22** (SSH).  
- `-j ACCEPT`: Allow (accept) this traffic.  

---

### **Step 4: Allow HTTP (Port 80) and HTTPS (Port 443) from Anywhere**
```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```
**Explanation**:  
- `-p tcp`: Use **TCP** protocol.  
- `--dport 80`: Match **HTTP** traffic (port 80).  
- `--dport 443`: Match **HTTPS** traffic (port 443).  
- `-j ACCEPT`: Allow these ports for **any source IP**.  

---

### **Step 5: Persist Rules**
```bash
sudo apt install iptables-persistent -y
sudo netfilter-persistent save
```
**Explanation**:  
- `iptables-persistent`: Installs tools to save `iptables` rules to disk.  
- `netfilter-persistent save`: Saves the current rules to `/etc/iptables/rules.v4` so they persist after reboot.  

---

### **Final Notes**
- **Security**: The default `DROP` policy ensures only explicitly allowed traffic is permitted.  
- **Outgoing Traffic**: All outgoing traffic is allowed (`OUTPUT` chain set to `ACCEPT`).  
- **Persistence**: Rules are saved to ensure they survive reboots.  

---

### **Verification**
To check the rules:
```bash
sudo iptables -L -v
```
This lists all rules and their statistics.

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
### Difference Between Mock, Stub, and Fake in Software Testing

1. **Stub**:  
   A **Stub** is a test double that provides predefined, static responses to method calls. Its primary purpose is to supply known data to the unit under test, allowing tests to focus on the logic of the unit rather than external dependencies. Stubs do not track interactions or verify method calls.  
   *Example*: A stub might return a fixed dataset when a method is called, simulating a database query.

2. **Fake**:  
   A **Fake** is a real, working object that simulates the behavior of a real object but is simplified for testing. Fakes are often implemented with real code but are lightweight or use in-memory data to avoid external dependencies.  
   *Example*: A fake database might be an in-memory database used during tests instead of a real database.

3. **Mock**:  
   A **Mock** is a test double that not only provides predefined responses but also **verifies interactions** (e.g., whether a method was called with specific arguments, how many times, etc.). Mocks are used to ensure that the code under test behaves correctly by checking its interactions with dependencies.  
   *Example*: A mock might verify that a method was called with the correct parameters to ensure proper business logic.

---

### Python Example Using `unittest.mock`

```python
import requests
import requests.exceptions
from unittest import TestCase
from unittest.mock import patch, MagicMock

def fetch_data(user_id):
    url = f"https://api.example.com/data/{user_id}"
    response = requests.get(url)
    response.raise_for_status()
    return response.json()

class TestFetchData(TestCase):
    @patch('requests.get')
    def test_fetch_data_success(self, mock_get):
        # Simulate a successful API response
        mock_response = MagicMock()
        mock_response.status_code = 200
        mock_response.json.return_value = {'data': 'success'}
        mock_get.return_value = mock_response

        result = fetch_data(123)
        self.assertEqual(result, {'data': 'success'})
        mock_get.assert_called_once_with('https://api.example.com/data/123')

    @patch('requests.get')
    def test_fetch_data_error(self, mock_get):
        # Simulate an error response (e.g., 404)
        mock_response = MagicMock()
        mock_response.status_code = 404
        mock_get.return_value = mock_response

        with self.assertRaises(requests.exceptions.HTTPError):
            fetch_data(456)

        mock_get.assert_called_once_with('https://api.example.com/data/456')
```

---

### Key Points in the Example

1. **Mocking the API Call**:  
   The `@patch('requests.get')` decorator replaces the `requests.get` method with a mock object, allowing us to simulate HTTP responses.

2. **Verifying the URL**:  
   `mock_get.assert_called_once_with(...)` ensures the `requests.get` method was called with the correct URL.

3. **Testing Success and Error Scenarios**:  
   - **Success**: A mock response with `status_code = 200` is used to simulate a valid API call.  
   - **Error**: A mock response with `status_code = 404` triggers an `HTTPError`, which is caught and verified using `self.assertRaises`.

This example demonstrates how to use **mocks** (to simulate API behavior and verify interactions) and **stubs** (to provide predefined responses) in unit tests.

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
### 1. **Publisher Nasıl Mesaj Yayınlar?**  
Hermes kütüphanesiyle publisher, belirli bir **topic** (konu) üzerinden mesaj yayınlar. Publisher, mesajı bir topic'e göndererek diğer subscriber'lar tarafından dinlenebilir hale getirir. Bu işlem genellikle `publish()` veya `send()` gibi bir fonksiyon üzerinden yapılır.  
**Örnek:**  
```python
from hermes import Hermes

# Hermes bağlantısı kurulumu
hermes = Hermes("localhost:11111")

# Mesaj yayın
topic = "test_topic"
message = "Merhaba, Hermes!"
hermes.publish(topic, message)
```
- `publish()` fonksiyonu, belirtilen `topic` adında bir konuya mesaj gönderir.  
- Bu mesaj, tüm bu topic'e abone olan subscriber'lar tarafından alınabilir.  

---

### 2. **Subscriber Nasıl Dinler?**  
Subscriber, belirli bir topic'e abone olur ve mesajları dinler. Hermes'te bu işlem genellikle `subscribe()` fonksiyonu ile yapılır. Subscriber, mesaj gelmesi durumunda bir callback fonksiyonu çağrılır.  
**Örnek:**  
```python
from hermes import Hermes

# Hermes bağlantısı kurulumu
hermes = Hermes("localhost:11111")

# Mesaj dinleme
topic = "test_topic"
def on_message(message):
    print("Alınan mesaj:", message)

hermes.subscribe(topic, on_message)
```
- `subscribe()` fonksiyonu, belirtilen `topic` adında bir konuya abone olur.  
- `on_message` callback fonksiyonu, mesaj gelmesi durumunda çağrılır ve mesaj işlenir.  

---

### 3. **Bu Pattern'in Doğrudan Fonksiyon Çağrısına Göre Avantajı Nedir?**  
Publisher/Subscriber pattern'i doğrudan fonksiyon çağrısına göre avantajları şunlardır:  

#### ✅ **Bağımsızlık ve Esneklik:**  
- Publisher, mesajı doğrudan bir subscriber'a değil, bir **mesaj ortamı** (örneğin Hermes) üzerinden gönderir.  
- Bu, sistemlerin bağımsız çalışmasına ve yeni subscriber'lar eklenmesine olanak tanır.  

#### ✅ **Dağıtık Sistemlerde Uyumluluk:**  
- Mesaj ortamı üzerinden iletişim, dağıtık sistemlerde (örneğin microservices) daha kolay entegrasyon sağlar.  
- Bir publisher, bir subscriber'a doğrudan erişmek yerine, mesaj ortamı üzerinden iletişim kurar.  

#### ✅ **Yüksek Ölçeklenebilirlik:**  
- Birden fazla subscriber, aynı topic'e abone olabilir.  
- Bu, yük paylaşımı ve paralel işleme imkanı sağlar.  

#### ✅ **Kurulum ve Yönetim Kolaylığı:**  
- Mesaj ortamı, mesajların güvenli ve düzenli bir şekilde taşınmasını sağlar.  
- Publisher ve subscriber'lar, mesaj ortamı üzerinden bağımsız olarak çalışabilir.  

---

### Özet:  
- **Publisher:** `publish(topic, message)` ile mesaj gönderir.  
- **Subscriber:** `subscribe(topic, callback)` ile mesaj dinler.  
- **Avantaj:** Doğrudan fonksiyon çağrısı yerine mesaj ortamı üzerinden iletişim, sistemlerin bağımsız çalışmasına, esnekliğe ve ölçeklenebilirliğe olanak tanır.
