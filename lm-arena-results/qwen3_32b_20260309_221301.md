---

### **1. Sorunun Nedeni**
Nginx servisini başlatamamanızın nedeni, **80 numaralı portun zaten başka bir uygulama (Apache2) tarafından kullanılıyor olması**.  
`journalctl` çıktısındaki hata mesajı net:  
```

```
Bu, Nginx'in 80 portuna bağlanamadığını gösterir.

```
LISTEN 0 128 0.0.0.0:80 users:(("apache2",pid=1247))
```

---

### **2. Çözüm Adımları**

#### **A. Apache2'yi Durdurun**
80 portunu serbest bırakmak için Apache2'yi durdurun:  
```bash
sudo systemctl stop apache2
```

#### **B. Portun Serbest Olduğunu Kontrol Edin**
Apache2 durduktan sonra 80 portunun kullanılıp kullanılmadığını tekrar kontrol edin:  
```bash
sudo ss -tlnp | grep :80
```
**Beklenen çıktı:**  
Eğer Apache2 doğru şekilde durdurulduysa, bu komut **hiçbir şey döndürmeyebilir** (yani 80 portu serbest olur).

---

#### **C. Nginx'i Başlatın**
Port serbest olduktan sonra Nginx'i başlatın:  
```bash
sudo systemctl start nginx
```

#### **D. Nginx'in Çalıştığını Doğrulayın**
Nginx'in çalışıp çalışmadığını kontrol etmek için:  
```bash
sudo systemctl status nginx
```

**A:**
### Problems Identified:

#### 1. **Security Vulnerability (SQL Injection)**
- **Issue**: The `username` variable is directly interpolated into the SQL query using an f-string (`f"SELECT * FROM orders WHERE username='{username}'"`). This allows an attacker to inject malicious SQL (e.g., `'; DROP TABLE orders;--`) and compromise the database.
- **Fix**: Use parameterized queries (SQLite's `?` placeholder) to separate SQL logic from user input.

#### 2. **Resource Leak (Unclosed Database Connection)**
- **Issue**: The `conn` object is never explicitly closed. If the function raises an exception or the caller forgets to close it, the connection remains open, leading to resource exhaustion.
- **Fix**: Use a context manager (`with sqlite3.connect(...) as conn`) or explicitly call `conn.close()` in a `finally` block.

#### 3. **No Error Handling**
- **Issue**: The code does not handle exceptions (e.g., database errors, invalid `db_path`).
- **Fix**: Wrap database operations in a `try-except` block to catch and handle errors gracefully.

#### 4. **Performance Risk (Fetching All Rows at Once)**
- **Issue**: `fetchall()` loads all results into memory at once. For large datasets, this can cause high memory usage.
- **Fix**: Use `fetchone()` in a loop for streaming results or ensure the query is optimized to return a manageable dataset.

#### 5. **Poor Architecture (Tight Coupling)**
- **Issue**: The function is tightly coupled to the database implementation. If the database schema or backend changes, this code must be rewritten.
- **Fix**: Abstract database operations into a repository or DAO (Data Access Object) class to decouple logic from implementation.
