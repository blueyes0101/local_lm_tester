## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
**SQL vs. NoSQL Databases: Key Differences for Developers**

SQL databases, also known as relational databases, store data in structured tables with predefined schemas. Each table consists of rows and columns, and relationships between tables are enforced through foreign keys and joins. This rigid structure ensures strong consistency and makes SQL databases ideal for applications where data integrity and complex queries are paramount, such as financial systems or enterprise resource planning (ERP) platforms. Developers benefit from mature query languages like SQL and powerful transactional guarantees, but must design their schemas upfront to accommodate future changes.

In contrast, NoSQL databases embrace flexible data models that can store documents, key‑value pairs, wide columns, or graphs. These databases often sacrifice strict consistency for higher availability and partition tolerance, following the BASE (Basically Available, Soft state, Eventual consistency) principles. Because NoSQL systems can scale horizontally across commodity servers, they excel in handling large volumes of unstructured or semi‑structured data, such as user‑generated content, real‑time analytics, or IoT telemetry. Developers can add new fields or change document structures without downtime, making schema evolution more agile.

Choosing between SQL and NoSQL depends on the specific requirements of the application. For transactional workloads demanding ACID compliance and complex joins, SQL remains the go‑to choice. For high‑velocity, large‑scale data ingestion and retrieval where flexibility is key, NoSQL offers superior scalability and performance. Understanding these trade‑offs—data structure rigidity versus schema agility, vertical versus horizontal scaling, and consistency versus availability—enables developers to select the right database technology for their project’s goals.

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
In software testing, **Mock**, **Stub**, and **Fake** are terms used to describe different types of test doubles, each serving a distinct purpose:

- **Mock**: A mock object is a test double that records how it was called (methods, arguments, etc.) and can be programmed to expect certain calls. It is primarily used to verify that the system under test interacts correctly with its dependencies. Mocks are usually "strict" — if a method is not called as expected, the test fails.

- **Stub**: A stub is a simpler test double that provides predefined responses to method calls. It doesn't verify anything; it just returns data to allow the system under test to run. Stubs are used to simulate the behavior of real objects without enforcing any expectations.

- **Fake**: A fake is a working implementation of a dependency, but one that is not meant for production. It behaves like the real thing but is easier to control or reset between tests. Fakes are useful when you need a real-like object but want to avoid side effects.

---

Here’s a Python example using `unittest.mock` that demonstrates all three concepts:

```python
import unittest
from unittest.mock import Mock, Stub, patch
import requests

# The real function that makes an HTTP call
def fetch_data_from_api(url):
    response = requests.get(url)
    response.raise_for_status()
    return response.json()

# Test class
class TestApiCall(unittest.TestCase):
    @patch('requests.get')
    def test_mock_verifies_call(self, mock_get):
        # Create a mock object
        mock_response = Mock()
        mock_response.json.return_value = {"data": "success"}
        mock_get.return_value = mock_response

        # Call the function under test
        result = fetch_data_from_api("https://api.example.com/data")

        # Verify the mock was called with the correct URL
        mock_get.assert_called_once_with("https://api.example.com/data")
        self.assertEqual(result, {"data": "success"})

    @patch('requests.get')
    def test_stub_returns_error(self, mock_get):
        # Create a stub that always returns an error response
        stub_response = Stub()
        stub_response.raise_for_status.side_effect = requests.HTTPError("404 Not Found")
        mock_get.return_value = stub_response

        # Expect an HTTPError when calling the function
        with self.assertRaises(requests.HTTPError):
            fetch_data_from_api("https://api.example.com/invalid")

if __name__ == '__main__':
    unittest.main()
```

### Explanation:
- **Mock**: In `test_mock_verifies_call`, `Mock` is used to simulate the `requests.get` call. We verify that the method was called with the correct URL using `assert_called_once_with`.
- **Stub**: In `test_stub_returns_error`, `Stub` is used to override `raise_for_status` to raise an `HTTPError`, simulating an API error without making a real HTTP request.
- **Behavior verification**: The first test ensures the mock was used correctly, while the second test confirms the system handles an API error gracefully.

This example shows how to use mocks for verification, stubs for controlled responses, and how to combine them to test both happy and error paths.

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
### Hermes Mesaj Kütüphanesi ile Publisher/Subscriber Pattern

#### 1. Publisher Nasıl Mesaj Yayınlar?
**Publisher** (Yayıncı) kısmı, bir mesajı **Hermes Mesaj Kütüphanesi**'ne gönderir. Bu kütüphane, mesajları bir listeye (topikler) takip eden tüm **Subscriber**'lerinin bilgisini saklar.  
Örnek olarak, bir haber sitesi yazarları (Publisher) mesajlarını (örneğin: "Yeni ürünler") kullanıcıların (Subscriber) ilgisine göndermek için kullanır.  
**Nasıl?**  
- Publisher, mesajı bir **topik** (kategori) ile ilişkilendirir.  
- Mesaj kütüphanesi, mesajı topikten kaydeden tüm Subscriber'leri bulur.  
- Subscriber'ler, mesajın konusunu görebilir ve ona göre işlem yapabilir.  

#### 2. Subscriber Nasıl Dinler?
**Subscriber** (İçerik alanı) mesaj kütüphanesinden mesajlar alır.  
**Nasıl?**  
- Subscriber, bir veya birden fazla topikten (mesaj kategorisi) izin alır.  
- Mesaj kütüphanesi, Subscriber'ye mesajı gönderdikten sonra, Subscriber'nin **hizmet (callback)** fonksiyonuna çağrı yapar.  
- Örneğin: Bir abone, "Yeni ürünler" topi için mesaj almak için bir fonksiyon kullanır.  

#### 3. Avantajı Ne Zaman?  
**Doğrudan fonksiyon çağrısına göre** (örneğin: `publish()` ve `subscribe()` fonksiyonlarını aşağıdaki gibi) Hermes mesaj kütüphanesi birkaç avantaj sunar:  

- **Yeniden Kullanım Özgürlüğü (Loose Coupling):**  
  Publisher ve Subscriber, birbirine bağımlı olmaz. Publisher, Subscriber'nin içeriğini bilmeden mesajları gönderir.  
  Örnek: Bir abone mesajı yazıyorsa, kendi kodunu değiştirmadan (örneğin, bir fonksiyonun adını değiştirmek) yeni abone ekleyebilir.  

- **Yeniden Kullanım Kolaylığı (Scalability):**  
  Yeni Subscriber'ler eklenirken, kod değişikliği yapılmaz. Mesaj kütüphanesi otomatik olarak mesajı her şeyine gönderir.  

- **Asınma ve Yük Kontrolü (Asynchronous & Decoupled):**  
  Mesajlar asenkron olarak gönderilir. Publisher, Subscriber'ın yanıtını beklememek için hızlı bir yapıdır.  

- **Topik Temelinde Güvenlik:**  
  Mesaj kütüphanesi, mesajları topiklere göre filtreler. Subscriber, kendi ilgili topiklerle ilgilidir.  

---

### Örnek Kod Örneği (Python)
```python
from collections import defaultdict

class MessageBroker:
    def __init__(self):
        self.subscribers = defaultdict(list)  # Topik: [Subscriber-Fonksiyonları]

    def subscribe(self, topic, callback):
        self.subscribers[topic].append(callback)  # Subscriber'yi topikle bağla

    def publish(self, topic, message):
        for callback in self.subscribers[topic]:
            callback(message)  # Tüm abone fonksiyonlarına mesaj gönder

# Publisher (Yayıncı)
def publisher(broker, topic, message):
    print(f"Yayınlanan mesaj: {message} (Topik: {topic})")
    broker.publish(topic, message)  # Broker üzerinden mesaj gönder

# Subscriber (İçerik alanı) - İşlem fonksiyonu
def subscriber_alon(subscriber_id, message):
    print(f"Subscriber {subscriber_id} alıyor: {message}")

# Kullanım Örneği
broker = MessageBroker()
broker.subscribe("yeni_ürünler", lambda msg: subscriber_alon(1, msg))  # Subscriber'ı "yeni_ürünler" topi için abone

# Mesaj yayınla (Publisher)
publisher(broker, "yeni_ürünler", "Yeni ürünler geliştirildi!")  # Çıktı: "Yayınlanan mesaj: Yeni ürünler geliştirildi! (Topik: yeni_ürünler)"
```

**Açıklama:**  
- `MessageBroker`, mesajları yönetecek kütüphane.  
- `subscriber_alon`, Subscriber'nin mesaj alanı.  
- `publisher`, Publisher'ın mesajı kütüphaneye göndermesi.  
- `subscriber`, mesaj alanının fonksiyonu.  

Bu yapı, Publisher/Subscriber patterninin temelini oluşturur.
