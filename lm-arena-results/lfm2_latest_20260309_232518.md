## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
**SQL vs. NoSQL: Choosing the Right Database for Modern Applications**

SQL databases, such as MySQL, PostgreSQL, and Oracle, store data in structured tables with predefined schemas. Each record is organized into rows and columns, and relationships between entities are enforced through foreign keys and joins. This rigid structure guarantees data integrity and makes complex queries straightforward using SQL language. In contrast, NoSQL databases—like MongoDB, Cassandra, and Redis—offer flexible, schema-less storage models. They can store data as documents, key-value pairs, wide-column stores, or graphs, allowing developers to adapt the data model on the fly without costly migrations.

Scalability is another key differentiator. SQL databases traditionally scale vertically, meaning you increase the capacity of a single server by adding more CPU, RAM, or storage. While this approach can be effective for moderate workloads, it has limits and can become expensive. NoSQL systems are designed for horizontal scalability, distributing data across multiple servers or clusters. This enables them to handle massive volumes of read and write operations with lower latency, making them well-suited for cloud-native and big data applications.

When deciding between SQL and NoSQL, developers should consider the nature of their data and application requirements. SQL excels in scenarios requiring strong consistency, complex transactions, and relational data modeling—think financial systems, inventory management, or enterprise reporting. NoSQL shines in use cases with rapidly evolving data structures, high throughput demands, or unstructured data—such as real-time analytics, content management systems, or IoT data ingestion. Understanding these differences helps teams align database choices with performance, maintainability, and future growth goals.

## Question 5
**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Below is a **complete `iptables` configuration** that implements the rules you described, followed by a **line‑by‑line explanation** of what each rule does.

---

## 1️⃣ The configuration script (saved as a shell script)

```bash
#!/bin/bash
# ------------------------------------------------------------
# iptables configuration for an Ubuntu server
# ------------------------------------------------------------

# 1️⃣  Flush any existing rules (optional – be careful!)
iptables -F
iptables -X

# 2️⃣  Set the default policy for the INPUT chain to DROP
#    This means “reject everything that reaches the server unless an explicit rule allows it”
iptables -P INPUT DROP

# 3️⃣  Allow related / established connections (important for outgoing traffic)
#    This rule matches packets that are related to outgoing connections,
#    so they are automatically allowed even though the default policy is DROP.
iptables -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT

# 4️⃣  Accept SSH (port 22) only from the 10.0.0.0/24 network
#    The --dport flag restricts the rule to the destination port.
iptables -A INPUT -p tcp -s 10.0.0.0/24 --dport 22 -j ACCEPT

# 5️⃣  Accept HTTP (port 80) and HTTPS (port 443) from anywhere
#    We use a single rule with the -p tcp option and a list of destination ports.
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 6️⃣  Allow all outgoing traffic (no restriction on what leaves the server)
#    The OUTPUT chain is never restricted by default, but we explicitly allow it
#    to be clear and to avoid accidental drops.
iptables -A OUTPUT -j ACCEPT

# 7️⃣  Save the rules so they survive a reboot
#    On Ubuntu the default persistence mechanism is `iptables-persistent`.
#    The script will write the current rule set to the persistent store.
iptables-save > /etc/iptables/rules.v4

# 8️⃣  (Optional) Verify that the rules were applied correctly
iptables -L -v -n
```

> **Tip:** Run the script once after installing `iptables-persistent` (or `iptables-persistence` on some distros) to make the rules permanent.  
> On Ubuntu you can also enable persistence automatically with `iptables-persistent`:

```bash
sudo apt install iptables-persistent
sudo netfilter-persistent save
```

---

## 2️⃣ Line‑by‑line explanation

| # | Command | What it does (in plain English) |
|---|---------|---------------------------------|
| **1** | `iptables -F` | **Flush** (delete) all existing **INPUT**, **OUTPUT**, and **FORWARD** chains. This clears any old rules you may have had. |
| **2** | `iptables -X` | Deletes any **custom user‑defined chains** (if you created any). Keeps only the default chains (`INPUT`, `OUTPUT`, `FORWARD`). |
| **3** | `iptables -P INPUT DROP` | Sets the **default policy** for the **INPUT** chain to **DROP**. Every packet that reaches the server *without* a matching rule will be rejected. This is the “drop all incoming” rule. |
| **4** | `iptables -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT` | Adds a rule that **accepts** packets that are **related** or **already established** (i.e., traffic that originated from the server and is responding to an outgoing request). This is essential for protocols like SSH, web servers, or any service that needs to reply to a client. Without this rule, the server would block its own responses. |
| **5** | `iptables -A INPUT -p tcp -s 10.0.0.0/24 --dport 22 -j ACCEPT` | Allows **incoming SSH connections** only from the **10.0.0.0/24** network. The `-p tcp` flag restricts the rule to the TCP protocol (the most common for SSH). The `-s` field defines the source IP range, and `--dport 22` limits the rule to port 22 (SSH). |
| **6** | `iptables -A INPUT -p tcp --dport 80 -j ACCEPT` | Allows **incoming HTTP traffic** (port 80) from any source. This is the default for web servers. |
| **7** | `iptables -A INPUT -p tcp --dport 443 -j ACCEPT` | Allows **incoming HTTPS traffic** (port 443) from any source. This is required for secure web traffic. |
| **8** | `iptables -A OUTPUT -j ACCEPT` | Explicitly permits **all outgoing traffic**. By default the OUTPUT chain is already open, but this rule makes the intent clear and ensures no accidental drops. |
| **9** | `iptables-save > /etc/iptables/rules.v4` | Writes the current rule set to the file `/etc/iptables/rules.v4`. On Ubuntu (and Debian) this file is read at boot by `iptables-persistent`, which restores the rules automatically. |
| **10** (optional) | `iptables -L -v -n` | Lists all rules with verbose output, so you can double‑check that everything is in place. |

---

## 3️⃣ Why this order matters

1. **Default DROP** – Starting with a clean slate and a strict default policy prevents accidental exposure of services.
2. **Allow related traffic** – Without this, the server would block its own replies, breaking SSH, web servers, etc.
3. **Specific allow rules** – After the default is set, we only permit the traffic we explicitly want.
4. **Persistence** – Saving the rules ensures they survive reboots.

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
In software testing, **Mock**, **Stub**, and **Fake** are terms used to describe different types of test doubles, each with distinct purposes:

- **Mock**: A mock object is a test double that records how it was called (methods, arguments, etc.) and allows you to verify these calls after execution. It is typically used to ensure that certain interactions happen as expected. Mocks are usually "strict" and will raise an error if expectations are not met.

- **Stub**: A stub object provides predefined responses to method calls without recording or verifying how it was called. It is used to simulate the behavior of real objects in a controlled way, often to isolate the system under test from external dependencies.

- **Fake**: A fake object is a working implementation of a dependency, but one that is simplified or designed for testing purposes. It behaves like the real object but is easier to control or reset. Unlike stubs, fakes may have their own logic and state.

---

Here’s a Python example using `unittest.mock` to demonstrate these concepts:

```python
import unittest
from unittest.mock import Mock, patch

# Simulated external HTTP API function
def fetch_data_from_api(url):
    # In a real scenario, this would make an HTTP request
    pass

# Function under test
def get_data(url):
    response = fetch_data_from_api(url)
    if response.get("status") == 200:
        return response.get("data")
    else:
        raise Exception("API Error")

class TestApiCalls(unittest.TestCase):
    @patch('__main__.fetch_data_from_api')
    def test_mock_verification(self, mock_api):
        # Setup mock to return a successful response
        mock_api.return_value = {"status": 200, "data": "test data"}
        
        # Call the function under test
        result = get_data("http://example.com/api")
        
        # Verify the mock was called with the correct arguments
        mock_api.assert_called_once_with("http://example.com/api")
        
        # Verify the result
        self.assertEqual(result, "test data")

    @patch('__main__.fetch_data_from_api')
    def test_error_handling(self, mock_api):
        # Setup mock to simulate an API error
        mock_api.return_value = {"status": 500, "error": "Internal Server Error"}
        
        # Expect an exception when the API returns an error
        with self.assertRaises(Exception) as context:
            get_data("http://example.com/api")
        
        # Verify the mock was called
        mock_api.assert_called_once_with("http://example.com/api")
        
        # Verify the error message
        self.assertEqual(str(context.exception), "API Error")

if __name__ == '__main__':
    unittest.main()
```

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
### 1. Publisher Nasıl Mesaj Yayınlar?

Hermes mesaj kütüphanesinde **Publisher**, mesajları birleştirir ve bu mesajları **Subscriber**'lara gönderir. Publisher, mesajları belirli bir kategoride (örneğin "Haberler", "İletişim") saklayarak gönderir. Her bir mesajın içeriği, belirlenmiş bir **Topik** veya **Konu** kategoriye ait olabilir. Örneğin, bir haber mesajı "Güncel haberler" kategorisinde yayılır ve bu kategoride ilgilenen tüm abonelere gönderilir.

**Örnek Kod:**
```python
from kazoo.client import KazooClient
from kazoo.exceptions import KazooError

class Publisher:
    def __init__(self, zk_hosts, topic):
        self.zk = KazooClient(hosts=zk_hosts)
        self.zk.start()
        self.topic = topic
        # Topikunu Hermes üzerinde yüklenmesini sağlar
        self.zk.create_zone(topic)

    def publish(self, message):
        # Topik üzerinde mesajı yayınlar
        self.zk.publish(self.topic, message)
        print(f"[Publisher] Topik '{self.topic}' için mesaj yayıldı: {message}")

# Kullanım Örneği
publisher = Publisher("localhost:2181", "Haberler")
publisher.publish("Güncel haberler: 2023 yılında 10. Mart.")
```

---

### 2. Subscriber Nasıl Dinler?

**Subscriber**, belirlenmiş bir **Topic** üzerinde Hermes mesajlarını dinler. Topikten mesaj alındığında, abonelere mesajın içeriğini işler ve ilgili işlemleri gerçekleştirir (örneğin, mesajı gösterme veya daha fazla işlem yapma).

**Örnek Kod:**
```python
class Subscriber:
    def __init__(self, zk_hosts, topic, callback):
        self.zk = KazooClient(hosts=zk_hosts)
        self.zk.start()
        self.topic = topic
        self.callback = callback  # Mesaj dinleme işlemi için işlem
        # Topik üzerinde abone olun
        self.zk.add_subscribe(self.topic, self.on_message)

    def on_message(self, topic, message):
        # Mesajı dinler ve işlemleri yap
        print(f"[Subscriber] Topik '{topic}'den mesaj aldı: {message}")
        self.callback(message)  # İlgili işlem için işlemi çağrır

# Kullanım Örneği
def handle_news(message):
    print(f"[Yönetim] Yeni haber: {message}")

subscriber = Subscriber("localhost:2181", "Haberler", handle_news)
subscriber.zk.start()  # Mesaj dinleme işlemi başlat
```

---

### 3. Bu Pattern'in Avantajı

Hermes mesaj kütüphanesindeki **Publisher/Subscriber patterni**, aşağıdaki avantajları sunar:

1. **Yapılan ve Decentralizasyonlu:** Publisher ve Subscriber ayrı ayrı geliştirilebilir. Her biri başkalarına doğrudan bağlanmaz; Hermes mesaj kütüphanesi tarafından ortak bir kanallara bağlanır.
2. **Yüksek Esneklik:** Yeni aboneler eklenebilir veya kesilmelere gerek yok. Topikler ve mesajlar sıklıkla yayınlanır.
3. **Yeniden Kullanıma Öne Çıkan:** Mesajların sıkça güncellenebilir (örneğin, "Haberler" topi). Abonelere gerçek zamanlı mesajlar aktarılır.
4. **Yönetim Kolaylığı:** Topiklerle mesajları kategorize edilir. Abonelere mesajların gönderilmesi, hangi kategoride ilgili olduğuna bağlı olarak otomatik olarak yönlendirilir.

**Örnek Senaryo:**  
Bir haber platformu olsaydı. Publisher (Haber Verisi Tabanı) "Güncel Haberler" topi üzerinde mesajlar yayar. Subscriber (Haber Tabanı, Uygulama Geliştiricisi) bu topi takip eder. Uygulama, mesajları dinler ve güncelleri gösterir. Bu yapı, platformun yeni abonelere hızlıca entegre olmasını sağlar.
