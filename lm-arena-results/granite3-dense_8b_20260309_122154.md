**A:**
Title: SQL vs NoSQL Databases: A Comparative Analysis

SQL (Structured Query Language) and NoSQL (Not Only SQL) databases are two distinct categories of database management systems, each with its unique characteristics and use cases.

SQL databases, also known as relational databases, store data in a structured format using tables consisting of rows and columns. This structure allows for complex queries and transactions, making SQL databases ideal for applications requiring complex joins and transactions, such as financial systems and data warehousing. However, SQL databases can struggle with scalability due to their rigid schema, which can limit their ability to handle large volumes of unstructured data.

On the other hand, NoSQL databases are designed to handle unstructured, semi-structured, and structured data. They offer flexible schemas, allowing for easy scalability and high performance. NoSQL databases are often used in big data and real-time web applications, where data is less structured and the need for scalability is paramount. However, they may not support complex queries and transactions as efficiently as SQL databases.

In terms of use cases, SQL databases are typically used in scenarios where data consistency, integrity, and complex queries are crucial. They are well-suited for applications like enterprise resource planning (ERP), customer relationship management (CRM), and data warehousing. In contrast, NoSQL databases are often used in scenarios where scalability, flexibility, and high performance are prioritized. They are commonly used in big data applications, real-time analytics, and content management systems.

In conclusion, the choice between SQL and NoSQL databases depends on the specific requirements of the application. SQL databases are suitable for structured data and complex queries, while NoSQL databases are ideal for unstructured data and high scalability.

## Question 5
**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
1. Drop all incoming traffic by default:
```
iptables -P INPUT DROP
```
This command sets the default policy for the INPUT chain to DROP, meaning all incoming traffic will be dropped unless explicitly allowed.

2. Accept SSH (port 22) only from 10.0.0.0/24:
```
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT
```
This command adds a rule to the INPUT chain that allows incoming TCP traffic on port 22 (SSH) from the 10.0.0.0/24 network.

3. Accept HTTP (80) and HTTPS (443) from anywhere:
```
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```
These commands add rules to the INPUT chain that allow incoming TCP traffic on ports 80 (HTTP) and 443 (HTTPS) from any source.

4. Allow all outgoing traffic:
```
iptables -P OUTPUT ACCEPT
```
This command sets the default policy for the OUTPUT chain to ACCEPT, allowing all outgoing traffic.

5. Make the rules persistent:
```
iptables-save > /etc/iptables/rules.v4
```
This command saves the current iptables rules to the file /etc/iptables/rules.v4, making them persistent across reboots.

To load the rules at boot, you can use the following command:
```
iptables-restore < /etc/iptables/rules.v4
```

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
In software testing, Mock, Stub, and Fake are used to replace real objects with simulated ones for better control and isolation of the system under test.

- Mock: A mock object is a simulated object that mimics the behavior of a real object in a controlled way. It is used to isolate the system under test from its dependencies and to verify its interactions with those dependencies.
- Stub: A stub is a simulated object that provides canned responses to specific inputs. It is used to replace real objects that are difficult to control or that have side effects.
- Fake: A fake is a simulated object that mimics the behavior of a real object, but it is not designed to be used for testing. It is used to provide a simple and fast replacement for a real object in a production environment.

Here's an example of using unittest.mock to mock an external HTTP API call, verify the mock was called with the correct arguments, and test the behavior when the API returns an error:
```
import unittest
from unittest.mock import patch
import requests

def get_data():
    response = requests.get('https://api.example.com/data')
    if response.status_code == 200:
        return response.json()
    else:
        raise Exception('Error getting data')

class TestGetData(unittest.TestCase):
    @patch('requests.get')
    def test_get_data(self, mock_get):
        mock_get.return_value.status_code = 200
        mock_get.return_value.json.return_value = {'key': 'value'}
        self.assertEqual(get_data(), {'key': 'value'})

        mock_get.return_value.status_code = 404
        with self.assertRaises(Exception):
            get_data()

        mock_get.assert_called_with('https://api.example.com/data')
        mock_get.assert_called_once()

if __name__ == '__main__':
    unittest.main()
```
In this example, we have a function `get_data()` that makes an HTTP GET request to an external API to retrieve data. We want to test this function in isolation, so we use unittest.mock to mock the `requests.get()` function and replace it with a mock object.

In the test case, we first set up the mock object to return a successful response with a status code of 200 and a JSON payload of `{'key': 'value'}`. We then call `get_data()` and assert that it returns the expected payload.

Next, we set up the mock object to return an error response with a status code of 404. We then call `get_data()` and assert that it raises an exception.

Finally, we verify that the mock object was called with the correct arguments and that it was called only once.

```python
from hermes import Publisher

```

```python
from hermes import Subscriber

subscriber = Subscriber()
subscriber.subscribe("hello", callback=lambda msg: print(f"Received message: {msg}"))
```

**Q:** Review the following Python code and identify ALL problems (security, performance, architecture). Then rewrite it correctly:

import sqlite3

def get_user_orders(username, db_path):
    conn = sqlite3.connect(db_path)
    query = f"SELECT * FROM orders WHERE username='{username}'"
    result = conn.execute(query)
    return result.fetchall()

Please answer in English.

**A:**
1. Security: The code is vulnerable to SQL injection attacks. The username input is directly concatenated into the SQL query, which can be exploited by an attacker to modify the query.

2. Performance: The code does not use parameterized queries or prepared statements, which can lead to slower performance when dealing with a large number of orders or users.

3. Architecture: The code does not follow the principle of separation of concerns. The function is responsible for both connecting to the database and executing the query, which can make the code harder to maintain and test.

Here's the corrected code:

```python
import sqlite3
from sqlite3 import Error

def get_user_orders(username, db_path):
    conn = None
    try:
        conn = sqlite3.connect(db_path)
        cursor = conn.cursor()
        query = "SELECT * FROM orders WHERE username=?"
        cursor.execute(query, (username,))
        result = cursor.fetchall()
    except Error as e:
        print(e)
    finally:
        if conn:
            conn.close()
    return result
```

This version of the code uses a parameterized query to prevent SQL injection attacks and improves performance. It also separates the concerns of connecting to the database and executing the query, making the code more maintainable and testable.
