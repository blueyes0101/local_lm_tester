## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
**SQL vs. NoSQL Databases: Key Differences for Developers**

SQL (Structured Query Language) databases, such as PostgreSQL or MySQL, rely on a rigid, tabular schema to organize data. They enforce strict data structures with predefined columns, data types, and relationships (e.g., primary/foreign keys), ensuring consistency and integrity through ACID (Atomicity, Consistency, Isolation, Durability) compliance. This schema-driven approach excels in complex queries involving joins, aggregations, and transactions—ideal for applications requiring precise, structured data like financial systems or inventory management. However, this rigidity can become a bottleneck as data grows, as schema modifications often require downtime and careful planning.

NoSQL databases, in contrast, prioritize flexibility and scalability by abandoning fixed schemas in favor of dynamic, document-oriented (e.g., MongoDB), key-value (e.g., Redis), column-family (e.g., Cassandra), or graph (e.g., Neo4j) models. This schema-less nature allows rapid iteration and adaptation to evolving data formats, making NoSQL a natural fit for unstructured or semi-structured data like JSON, logs, or social media feeds. While NoSQL sacrifices some ACID guarantees for eventual consistency, it shines in horizontally scalable architectures, distributing data across clusters to handle massive volumes of low-latency reads/writes—critical for modern web apps, IoT platforms, or real-time analytics.

Choosing between SQL and NoSQL hinges on the application’s requirements. SQL databases dominate in scenarios demanding strong consistency, relational integrity, and multi-user transactions (e.g., banking or ERP systems), while NoSQL excels in high-velocity, distributed environments where flexibility and horizontal scaling are paramount (e.g., content management, real-time collaboration, or big data pipelines). Hybrid approaches, like multi-model databases (e.g., Couchbase), are also emerging to bridge the gap, offering developers the best of both worlds when schema constraints and scalability needs coexist. Ultimately, the decision should align with performance, maintainability, and the nature of the data itself.

     ```bash
     sudo ufw status
     ```

       ```bash
       sudo ufw allow 80/tcp
       sudo ufw reload
       ```

#### **6. Nginx Loglarını Kontrol Edin**

     ```bash
     sudo tail -n 50 /var/log/nginx/error.log
     ```

1. **Apache2'yi durdurun** (`sudo systemctl stop apache2`).
2. **Nginx'i başlatın** (`sudo systemctl start nginx`).
3. **Port 80'de kalan süreçleri kontrol edin** (`sudo lsof -i :80`).
4. **Nginx config dosyasını doğrulayın** (`sudo nginx -t`).
5. **Firewall ayarlarını kontrol edin** (`sudo ufw status`).

## Question 5
**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here’s how to configure `iptables` on an Ubuntu server with the specified rules, along with an explanation of each step:

---

### **1. Flush Existing Rules (Optional but Recommended)**
Before applying new rules, clear existing ones to avoid conflicts:
```bash
sudo iptables -F
sudo iptables -X
sudo iptables -Z
```
- `-F`: Flush all rules in all tables.
- `-X`: Delete all custom chains.
- `-Z`: Reset packet counters.

---

### **2. Set Default Policies**
#### **Drop all incoming traffic by default**
```bash
sudo iptables -P INPUT DROP
```
- `-P INPUT DROP`: Set the default policy for the `INPUT` chain to `DROP` (reject all incoming traffic).

#### **Allow loopback traffic (optional but recommended)**
```bash
sudo iptables -A INPUT -i lo -j ACCEPT
```
- `-A INPUT`: Append a rule to the `INPUT` chain.
- `-i lo`: Match traffic coming from the loopback interface (localhost).
- `-j ACCEPT`: Accept this traffic.

---

### **3. Allow SSH (Port 22) Only from `10.0.0.0/24`**
```bash
sudo iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT
```
- `-A INPUT`: Append to the `INPUT` chain.
- `-p tcp`: Match TCP traffic.
- `--dport 22`: Destination port 22 (SSH).
- `-s 10.0.0.0/24`: Source IP range `10.0.0.0/24`.
- `-j ACCEPT`: Accept this traffic.

---

### **4. Allow HTTP (Port 80) and HTTPS (Port 443) from Anywhere**
```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```
- `-A INPUT`: Append to the `INPUT` chain.
- `-p tcp`: Match TCP traffic.
- `--dport 80`: Destination port 80 (HTTP).
- `--dport 443`: Destination port 443 (HTTPS).
- `-j ACCEPT`: Accept this traffic.

---

### **5. Allow All Outgoing Traffic**
```bash
sudo iptables -P OUTPUT ACCEPT
```
- `-P OUTPUT ACCEPT`: Set the default policy for the `OUTPUT` chain to `ACCEPT` (allow all outgoing traffic).

---

### **6. Allow Established/Related Connections**
```bash
sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```
- `-A INPUT`: Append to the `INPUT` chain.
- `-m conntrack`: Match connection tracking.
- `--ctstate ESTABLISHED,RELATED`: Allow established or related connections (e.g., responses to outgoing requests).
- `-j ACCEPT`: Accept this traffic.

---

### **7. Save Rules Persistently (Ubuntu 18.04+)**
Ubuntu uses `netfilter-persistent` to save `iptables` rules. Install and configure it:
```bash
sudo apt update
sudo apt install iptables-persistent -y
```
Save the rules:
```bash
sudo netfilter-persistent save
```
(Or manually save with `sudo iptables-save > /etc/iptables/rules.v4`.)

---

### **8. Verify Rules**
Check the current rules:
```bash
sudo iptables -L -n -v
```
- `-L`: List rules.
- `-n`: Show numeric output (no DNS resolution).
- `-v`: Verbose (show packet/byte counts).

---

### **Explanation of Key Concepts**
1. **Default Policies**:
   - `DROP` for `INPUT`: Reject all incoming traffic unless explicitly allowed.
   - `ACCEPT` for `OUTPUT`: Allow all outgoing traffic by default.

2. **Chains**:
   - `INPUT`: Incoming traffic to the server.
   - `OUTPUT`: Outgoing traffic from the server.
   - `FORWARD`: Traffic routed through the server (not used here).

3. **Matching**:
   - `-p tcp`: Protocol (TCP).
   - `--dport`: Destination port.
   - `-s`: Source IP range.
   - `-m conntrack`: Connection tracking (for stateful rules).

4. **Jump (`-j`) Actions**:
   - `ACCEPT`: Allow traffic.
   - `DROP`: Silently discard traffic (no response sent).

5. **Persistence**:
   - Ubuntu stores rules in `/etc/iptables/rules.v4` (or `/etc/ip6tables/rules.v6` for IPv6).

---

### **Final Notes**
- **Security**: Ensure SSH (`port 22`) is only accessible from trusted networks (`10.0.0.0/24`).
- **Testing**: Test SSH access from `10.0.0.0/24` and verify HTTP/HTTPS work.
- **Firewall Tools**: For a more user-friendly interface, consider `ufw` (Uncomplicated Firewall):
  ```bash
  sudo ufw allow 22 from 10.0.0.0/24
  sudo ufw allow 80,443
  sudo ufw enable
  ```

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
### **Mock, Stub, and Fake in Software Testing**

In software testing, **Mock**, **Stub**, and **Fake** are techniques used to isolate the unit under test by replacing dependencies with controlled alternatives.

1. **Stub**
   - A **Stub** is a simplified replacement for a dependency that returns **predefined responses** (e.g., hardcoded values).
   - It does **not** simulate behavior—it just provides a fixed answer.
   - Example: A stub for a database might always return `{"id": 1, "name": "Test"}`.

2. **Mock**
   - A **Mock** is an **interactive fake object** that records interactions (e.g., method calls, arguments) and can enforce expectations.
   - It allows **verification** (e.g., "Was `fetch_data()` called with `param='x'`?").
   - Example: A mock HTTP client that tracks if `GET /api/data` was called with `params={"key": "value"}`.

3. **Fake**
   - A **Fake** is a **simplified implementation** of a real dependency (e.g., an in-memory database instead of a real SQL database).
   - It behaves **closer to the real thing** but is easier to control in tests.
   - Example: A fake `UserService` that stores data in a Python dict instead of a database.

---

### **Python Example Using `unittest.mock`**
Here’s a test that:
1. **Mocks an HTTP API call** (using `unittest.mock.patch`).
2. **Verifies the mock was called with correct arguments**.
3. **Tests error handling** when the API fails.

```python
import unittest
from unittest.mock import patch, MagicMock
import requests  # Assume this is the real HTTP client

# Function under test (e.g., fetches data from an API)
def fetch_user_data(user_id):
    response = requests.get(f"https://api.example.com/users/{user_id}")
    response.raise_for_status()  # Raises HTTPError on 4xx/5xx
    return response.json()

class TestFetchUserData(unittest.TestCase):
    @patch("requests.get")  # Mock the `requests.get` call
    def test_fetch_user_data_success(self, mock_get):
        # Setup mock response (success case)
        mock_response = MagicMock()
        mock_response.json.return_value = {"id": 1, "name": "Alice"}
        mock_response.raise_for_status.return_value = None  # No error
        mock_get.return_value = mock_response

        # Call the function
        result = fetch_user_data(1)

        # Verify the mock was called with the correct URL
        mock_get.assert_called_once_with("https://api.example.com/users/1")
        self.assertEqual(result, {"id": 1, "name": "Alice"})

    @patch("requests.get")
    def test_fetch_user_data_error(self, mock_get):
        # Setup mock response (error case)
        mock_response = MagicMock()
        mock_response.raise_for_status.side_effect = requests.exceptions.HTTPError("404 Not Found")
        mock_get.return_value = mock_response

        # Call the function (expect it to raise an exception)
        with self.assertRaises(requests.exceptions.HTTPError):
            fetch_user_data(999)

        # Verify the mock was called
```

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
**Hermes Mesaj Kütüphanesi ile Publisher/Subscriber Pattern (Yayıncı/Abone Modeli)**

Hermes, C++'da yazılmış bir mesaj geçiş kütüphanesi olup, asenkron mesajlaşma için **Publisher-Subscriber (Pub/Sub)** modelini destekler. Bu modelde, yayıncı (publisher) mesajları yayınlar ve aboneler (subscribers) bu mesajları dinleyerek işler. Hermes, bu modeli **hızlı, hafif ve düşük gecikmeli** bir şekilde sağlar.

---

### **1. Publisher Nasıl Mesaj Yayınlar?**
Publisher, Hermes üzerinden mesajları bir **toplu (bus)** veya **konu (topic)** üzerine yayınlar. Hermes'de bu işlem **asenkron** olarak gerçekleşir (yani yayıncı mesajı gönderdikten sonra beklemez).

#### **Kod Örneği (C++)**
```cpp
#include <hermes/hermes.h>

int main() {
    // Hermes bus'u başlat
    HermesBus bus;

    // Bir mesaj oluştur
    HermesMessage msg;
    msg.setData("Merhaba Hermes! Bu bir test mesajı.");

    // Mesajı yayınla (bus'a gönder)
    bus.publish(msg);

    // Bus'u kapat (aslında bu örnekte sadece gösterim amaçlı)
    bus.shutdown();

    return 0;
}
```
- **`bus.publish(msg)`** ile mesaj yayınlanır.
- Hermes, mesajı **yönlendirme (routing)** yaparak abonelere ulaştırır.

---

### **2. Subscriber Nasıl Dinler?**
Subscriber, belirli bir **toplu (bus)** veya **konu (topic)** üzerinde abone olur ve gelen mesajları dinler.

#### **Kod Örneği (C++)**
```cpp
#include <hermes/hermes.h>

void messageCallback(const HermesMessage& msg) {
    std::cout << "Alınan mesaj: " << msg.getData() << std::endl;
}

int main() {
    // Hermes bus'u başlat
    HermesBus bus;

    // Abonelik oluştur ve mesajları dinle
    bus.subscribe("mesaj_konusu", messageCallback);

    std::cout << "Mesajları dinlemeyi bekliyor..." << std::endl;

    // Bus'u kapat (aslında bu örnekte sadece gösterim amaçlı)
    bus.shutdown();

    return 0;
}
```
- **`bus.subscribe("mesaj_konusu", callback)`** ile abone olur.
- Gelen mesajlar **`messageCallback`** fonksiyonuna iletilir.

---

### **3. Publisher-Subscriber Modelinin Avantajları (Doğrudan Fonksiyon Çağrısına Göre)**
| **Avantaj** | **Açıklama** |
|-------------|-------------|
| **Bağımsızlık** | Publisher ve subscriber birbirinden bağımsız çalışır. Birinin devamlılığı diğerini etkilemez. |
| **Esnek Yönlendirme** | Birden fazla abone aynı mesajı alabilir (örn. loglama, işleme, depolama). |
| **Yük Dağılımı** | Mesajlar farklı sistemlere/servislere dağılabilir (örn. bir mesajı bir API'ye ve bir veritabanına göndermek). |
| **Asenkron İşlem** | Publisher mesajı gönderdikten sonra beklemez, aboneler mesajı aldıktan sonra işler. |
| **Ölüm Dirençli** | Bir abone devre dışı kalırsa, diğer aboneler etkilenmez. |
| **İleri Teknoloji Destek** | Hermes, **hızlı mesaj geçişi** ve **düşük gecikme** sağlar (örn. IoT, gerçek zamanlı sistemler için ideal). |

#### **Örnek Senaryo**
- **Doğrudan Fonksiyon Çağrısı:** `fonksiyonA(fonksiyonB());` → Fonksiyonlar sırayla çalışır, birinin hata vermesi diğerini etkiler.
- **Pub/Sub:** `fonksiyonA()` mesaj yayınlar, `fonksiyonB()` abone olur → İki işlem bağımsız çalışır, birinin arızası diğerini etkilemez.

---
### **Sonuç**
Hermes ile **Publisher-Subscriber**, mesaj geçişini **esnek, hızlı ve ölçeklenebilir** hale getirir. Özellikle **dağıtılmış sistemler, IoT, gerçek zamanlı uygulamalar** için ideal bir çözümdür.

Eğer **Python** için Hermes kullanıyorsanız, genellikle **Hermes C++ kütüphanesi** üzerinden Python bağlayıcıları (örn. `pybind11`) kullanılır. Eğer tamamen Python'da Pub/Sub istiyorsanız, **ZeroMQ, RabbitMQ veya Python'ın `asyncio`** gibi alternatifler de vardır.
