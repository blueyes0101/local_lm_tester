3. To implement refresh token rotation with a Redis-based token store, you can store the refresh tokens as hashes in Redis. The key of the hash would be the user ID, and the field-value pair would be the refresh token and its metadata like expiry time. The TTL strategy would be to set an expiry time for each refresh token. You can also use a LRU (Least Recently Used) strategy to remove tokens when Redis runs out of memory.

5. When a user changes their password, you should invalidate all their existing sessions. To do this, you can store a version number or a timestamp in each refresh token in the Redis store. During the refresh token validation, check if the version number or timestamp in the incoming token matches the one in the Redis store. If not, reject the token.

6. Token blacklisting and token rotation are two different strategies to handle revoked tokens. Token blacklisting involves maintaining a list of revoked tokens and checking each incoming token against this list. If the token is in the list, it's revoked. Token rotation, on the other hand, involves issuing a new token and invalidating the old one. Token blacklisting is useful when you have a short-lived token and a high volume of requests. Token rotation is useful when you want to increase the security of your system and reduce the impact of a compromised token. Token rotation is more secure but requires more overhead.

**A:**
1. Custom hook useSearch(query: string)

```javascript
import { useState, useEffect, useRef } from 'react';
import axios from 'axios';

const useSearch = (query) => {
  const [loading, setLoading] = useState(false);
  const [results, setResults] = useState([]);
  const [error, setError] = useState(false);
  const [cache, setCache] = useState({});
  const source = useRef(axios.CancelToken.source());

  useEffect(() => {
    const timer = setTimeout(() => {
      if (query && (!cache[query] || Date.now() - cache[query].timestamp > 300000)) {
        setLoading(true);
        setError(false);
        source.current = axios.CancelToken.source();
        axios.get(`/api/search?q=${query}`, {
          cancelToken: source.current.token
        })
          .then(response => {
            const result = response.data;
            setResults(result);
            setCache(prevCache => ({ ...prevCache, [query]: { timestamp: Date.now(), data: result } }));
            setLoading(false);
          })
          .catch(error => {
            if (axios.isCancel(error)) {
              console.log('Request cancelled');
            } else {
              setError(true);
              setLoading(false);
            }
          });
      } else if (cache[query]) {
        setResults(cache[query].data);
      }
    }, 300);

    return () => {
      clearTimeout(timer);
      source.current.cancel();
    };
  }, [query, cache]);

  return { loading, results, error };
};

export default useSearch;
```

## Question 9
**Q:** You are building a collaborative document editor (like Google Docs, simplified):
- Multiple users can edit the same document simultaneously
- Each user sees others' cursors in realtime
- Changes are broadcast to all connected clients
- If a user disconnects and reconnects, they get the latest document state

Implement the server side in Node.js using the ws library:
1. Connection handling — how do you associate a WebSocket connection with a user and document?
2. Message protocol — define the message types (JSON) for: join document, text change, cursor move, user left
3. Broadcasting — when user A makes a change, how do you send it to all other users in the same document but NOT back to user A?
4. Reconnection — how do you handle a user rejoining after disconnect?
5. What happens if two users edit the exact same position simultaneously? You don't need to implement OT/CRDT — just explain the problem and name the solution approaches.

Write the core server implementation (connection map, message routing, broadcast logic).

**A:**
1. Connection Handling:
We can use a JavaScript object or Map data structure to map each WebSocket connection to a unique user id and document id. When a new WebSocket connection is established, we can assign a unique id to the user and then ask the user to join a document by sending a 'join document' message. The server can then store this information in the map.

```javascript
let connections = new Map(); // key: ws connection, value: { userId, documentId }
```

2. Message Protocol:
We will define the following JSON message types:

- `join document`: Sent by a user to join a document. Payload includes the document id.
```javascript
{
  type: 'join document',
  documentId: string,
  userId: string
}
```

- `text change`: Sent by a user to change text. Payload includes the document id, new text, start position, and end position.
```javascript
{
  type: 'text change',
  documentId: string,
  userId: string,
  text: string,
  start: number,
  end: number
}
```

- `cursor move`: Sent by a user to move their cursor. Payload includes the document id and new cursor position.
```javascript
{
  type: 'cursor move',
  documentId: string,
  userId: string,
  position: number
}
```

- `user left`: Sent by the server to notify a user that another user has left the document. Payload includes the user id of the user who left.
```javascript
{
  type: 'user left',
  userId: string
}
```

3. Broadcasting:
When a user makes a change, we can broadcast the message to all connected users in the same document except for the user who made the change. This can be done by iterating over the connections map and checking the document id for each connection.

```javascript
connections.forEach((connectionInfo, connection) => {
  if (connectionInfo.documentId === documentId && connectionInfo.userId !== userId) {
    connection.send(JSON.stringify(message));
  }
});
```

4. Reconnection:
When a user disconnects and then reconnects, we can check if they have the latest document state by comparing the version number of the document on the server with the version number on the client. If the versions are different, we can send the latest document state to the user.

5. Simultaneous Edits:
If two users edit the exact same position simultaneously, it can lead to a "conflict" where one user's changes overwrite the other user's changes. To solve this problem, we can use a technique called Operational Transformation (OT) or Conflict-free Replicated Data Type (CRDT). OT allows two users to make simultaneous changes to a document without causing conflicts, by transforming each user's changes based on the other user's changes. CRDT is a data structure that allows multiple users to make simultaneous changes to a document without causing conflicts, by using a mathematical approach to merge changes.

Here's a simple implementation of the server:

```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

let connections = new Map();

wss.on('connection', (ws) => {
  ws.on('message', (message) => {
    const msg = JSON.parse(message);
    switch (msg.type) {
      case 'join document':
        connections.set(ws, { userId: msg.userId, documentId: msg.documentId });
        break;
      case 'text change':
      case 'cursor move':
        broadcast(msg, ws);
        break;
    }
  });

  ws.on('close', () => {
    const connectionInfo = connections.get(ws);
    if (connectionInfo) {
      connections.delete(ws);
      broadcast({ type: 'user left', userId: connectionInfo.userId }, null, connectionInfo.documentId);
    }
  });
});

function broadcast(message, sender, documentId) {
  connections.forEach((connectionInfo, connection) => {
    if (connection !== sender && (!documentId || connectionInfo.documentId === documentId)) {
      connection.send(JSON.stringify(message));
    }
  });
}
```

const DataLoader = require('dataloader');

const batchAuthors = async (keys, dataSources) => {
