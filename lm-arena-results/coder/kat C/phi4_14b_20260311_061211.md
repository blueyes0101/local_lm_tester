```typescript
app.patch('/workspaces/:workspaceId/tasks/:taskId', (req, res, next) => {
  const { workspaceId, taskId } = req.params;

  // Validate workspaceId and taskId
  if (!Number.isInteger(parseInt(workspaceId)) || !Number.isInteger(parseInt(taskId))) {
    return res.status(400).json({ error: 'Invalid workspaceId or taskId' });
  }

  const workspace = workspaces[parseInt(workspaceId)];
  const task = workspace?.tasks[parseInt(taskId)];

  if (!workspace || !task) {
    return res.status(404).json({ error: 'Workspace or task not found' });

  if (assignee_id !== null && !users[assignee_id]?.workspaceId === parseInt(workspaceId)) {

  } catch (error) {
    return next(error);
  }

  // Check for open subtasks before cancelling
  if (status === 'cancelled' && task.subtasks.some(subtask => subtask.status !== 'done')) {
```

**A:**
To implement the requested authentication features using a Redis-based token store, we need to design a system that efficiently handles refresh tokens, session management, and token security. Let's break down each requirement:

### 1. POST /auth/refresh

**Objective:** Use the refresh token from the cookie to issue a new access token and rotate the refresh token.

**Implementation Steps:**

- Retrieve the refresh token from the cookie.
- Check if the refresh token exists in Redis and is valid.
- If valid, generate a new access token and a new refresh token.
- Store the new refresh token in Redis with the user's ID and invalidate the old refresh token.
- Send the new access token and refresh token back to the client.

### 2. POST /auth/logout

**Objective:** Invalidate the refresh token and clear the cookie.

**Implementation Steps:**

- Retrieve the refresh token from the cookie.
- Delete the refresh token from Redis to invalidate it.
- Clear the refresh token cookie on the client side.

### 3. Redis Data Structure and TTL Strategy

**Data Structure:**

- Use a Redis hash to store refresh tokens. The hash key can be a namespace like `refresh_tokens`.
- Each entry in the hash will have:
  - `refresh_token_id`: The refresh token itself.
  - `user_id`: The ID of the user associated with the token.
  - `expires_at`: Timestamp when the token should expire.

**TTL Strategy:**

- Set a TTL (Time to Live) for each refresh token entry, e.g., 30 days.
- This ensures that tokens automatically expire if not used within the TTL period.

### 4. Handling Refresh Token Reuse (Token Reuse Detection)

**Objective:** Prevent a refresh token from being used more than once.

**Implementation Steps:**

- When a refresh token is used, immediately mark it as used by setting a flag or deleting it from Redis.
- If a refresh token is reused, it will not be found in the expected state, indicating a potential attack.

### 5. Invalidate All Sessions on Password Change

**Objective:** Invalidate all existing sessions when a user changes their password.

**Implementation Steps:**

- Store a user-specific "session version" or "token version" in Redis.
- Each refresh token should include this version.
- When a password is changed, increment the user's session version.
- Any refresh token with an outdated version should be considered invalid.

### 6. Token Blacklisting vs. Token Rotation

**Token Blacklisting:**

- Involves maintaining a list of tokens that are explicitly invalidated and should no longer be used.
- Useful for scenarios where you need to immediately invalidate tokens without waiting for expiration.

**Token Rotation:**

- Involves issuing a new token and invalidating the old one upon each use.
- Enhances security by limiting the lifespan of tokens and reducing the risk of token reuse.

**When to Use Each:**

- **Token Blacklisting:** Use when you need to invalidate tokens immediately, such as in response to a security breach or when a user logs out.
- **Token Rotation:** Use for ongoing session management to enhance security by regularly updating tokens.

```typescript
interface DataTableState<T> {
  sortedColumn: keyof T | null;
  sortDirection: 'asc' | 'desc' | 'none';
  selectedRows: T[];
  currentPage: number;
  pageSize: number;
}

const DataTable = <T,>({
  columns,
  data,
  pageSizeOptions = [10, 25, 50],
  defaultPageSize = 10,
  controlledPagination,
  controlledSorting,
  controlledSelection,
  onSortChange,
  onRowSelect,
  onPageSizeChange,
  ...props
}: DataTableProps<T>) => {
  const [state, setState] = useState<DataTableState<T>>({
    sortedColumn: null,
    sortDirection: 'none',
    selectedRows: [],
    currentPage: 1,
    pageSize: defaultPageSize,
  });

  const { sortedColumn, sortDirection, selectedRows, currentPage, pageSize } = state;

  const handleSort = useCallback(
    (columnKey: keyof T) => {
      let newDirection = 'asc';
      if (sortedColumn === columnKey) {
        newDirection =
          sortDirection === 'asc' ? 'desc' : sortDirection === 'desc' ? 'none' : 'asc';
      }

      setState((prevState) => ({
        ...prevState,
        sortedColumn: newDirection === 'none' ? null : columnKey,
        sortDirection: newDirection,
      }));

      if (onSortChange) {
        onSortChange(columnKey, newDirection);
      }
    },
    [sortedColumn, sortDirection, onSortChange]
  );

  const handleSelectRow = useCallback(
    (row: T) => {
      setState((prevState) => {
        const newSelectedRows = prevState.selectedRows.includes(row)
          ? prevState.selectedRows.filter((r) => r !== row)
          : [...prevState.selectedRows, row];

        if (onRowSelect) {
          onRowSelect(newSelectedRows);
        }

        return {
          ...prevState,
          selectedRows: newSelectedRows,
        };
      });
    },
    [onRowSelect]
  );

  const handleSelectAll = useCallback(() => {
    setState((prevState) => {
      const newSelectedRows = selectedRows.length === data.length
        ? []
        : data;

      if (onRowSelect) {
        onRowSelect(newSelectedRows);
      }

      return {
        ...prevState,
        selectedRows: newSelectedRows,
      };
    });
  }, [data, selectedRows, onRowSelect]);

  const handlePageChange = useCallback(
    (page: number) => {
      setState((prevState) => ({ ...prevState, currentPage: page }));
    },

  }, [sortedData, currentPage, pageSize]);

      if (props.rowClick) {
        props.rowClick(row);
      }
    },

  );

  return (
    <div>
      <table>
        <thead>

                )}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {paginatedData.map((row) => (
            <tr key={row[columns[0].key]} onClick={() => handleRowClick(row)}>
              <td>
                <input
                  type="checkbox"
                  checked={selectedRows.includes(row)}

              </td>
              {columns.map((column) => (
                <td key={column.key} width={column.width}>
                  {column.renderCell ? column.renderCell(row) : row[column.key]}
                </td>
              ))}
            </tr>
          ))}
        </tbody>
      </table>
      <div>
        <button onClick={() => handlePageChange(currentPage - 1)} disabled={currentPage === 1}>
          Previous
        </button>
        <span>
          Page {currentPage} of {Math.ceil(sortedData.length / pageSize)}
        </span>
        <button onClick={() => handlePageChange(currentPage + 1)} disabled={currentPage * pageSize >= sortedData.length}>
          Next
        </button>
      </div>
      <div>
        Page Size:
        {pageSizeOptions.map((size) => (
```

### 1. Custom Hook: `useSearch`

```jsx
import { useState, useEffect, useCallback } from 'react';
import { useHistory, useLocation } from 'react-router-dom';
import axios from 'axios';

const useSearch = (query) => {
  const [results, setResults] = useState(null);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);
  const [debouncedQuery, setDebouncedQuery] = useState(query);
  const [cancelTokenSource, setCancelTokenSource] = useState(null);

  const history = useHistory();
  const location = useLocation();

  const cache = new Map();

  useEffect(() => {
    const handleLocationChange = () => {
      const params = new URLSearchParams(location.search);
      const newQuery = params.get('q') || '';
      if (query !== newQuery) {
        setDebouncedQuery(newQuery);
      }
    };

    window.addEventListener('popstate', handleLocationChange);
    return () => window.removeEventListener('popstate', handleLocationChange);
  }, [location, query]);

  useEffect(() => {
    if (!debouncedQuery) {
      setResults(null);
      setError(null);
      setLoading(false);
      return;
    }

    const cachedResult = cache.get(debouncedQuery);
    if (cachedResult) {
      setResults(cachedResult);
      return;
    }

    const delayDebounce = setTimeout(() => {
      setLoading(true);
      setError(null);

      const cancelToken = axios.CancelToken.source();
      setCancelTokenSource(cancelToken);

      axios
        .get(`/api/search?q=${encodeURIComponent(debouncedQuery)}`, {
          cancelToken: cancelToken.token,
        })
        .then((response) => {
          const data = response.data;
          cache.set(debouncedQuery, data);
          setResults(data);
          setLoading(false);
        })
        .catch((err) => {
          if (axios.isCancel(err)) return;
          setError(err);
          setLoading(false);
        });

      return () => {
        cancelToken.cancel();
      };
    }, 300);

    return () => clearTimeout(delayDebounce);
  }, [debouncedQuery]);

  const retry = useCallback(() => {
    setResults(null);
    setError(null);
    setLoading(true);
    axios
      .get(`/api/search?q=${encodeURIComponent(debouncedQuery)}`)
      .then((response) => {
        const data = response.data;
        cache.set(debouncedQuery, data);
        setResults(data);
        setLoading(false);
      })
      .catch((err) => {
        setError(err);
        setLoading(false);
      });
  }, [debouncedQuery]);

  useEffect(() => {
    if (query !== debouncedQuery) {
      setDebouncedQuery(query);
    }

    const params = new URLSearchParams(location.search);
    if (!params.has('q') || params.get('q') !== query) {
      params.set('q', query);
      history.push({ search: params.toString() });
    }
  }, [query, location, history]);

  return { results, loading, error, retry };
};
```

```javascript
const WebSocket = require('ws');
const server = new WebSocket.Server({ port: 8080 });

const connections = {}; // Maps documentId to an array of user connections
const userToDocument = {}; // Maps userId to documentId

function joinDocument(userId, documentId) {
    if (!connections[documentId]) {
        connections[documentId] = [];
    }
    connections[documentId].push({ userId, ws: null });
    userToDocument[userId] = documentId;

function leaveDocument(userId) {
    const documentId = userToDocument[userId];

}
```

### 5. Choosing GraphQL over REST

**When to choose GraphQL:**

- **Complex Data Requirements:** When the client needs to fetch complex, nested data structures in a single request.
- **Reduced Overhead:** When you want to avoid multiple round trips to the server for related data.
- **Flexible Queries:** When clients need the flexibility to specify exactly what data they need.

**When NOT to choose GraphQL:**

- **Simple APIs:** When the API is simple and doesn't require complex queries or nested data fetching.
- **Caching Needs:** If caching strategies are a priority, REST with HTTP caching can be more straightforward.
- **Highly Secure APIs:** GraphQL can expose too much of the API surface, potentially leading to security issues if not properly managed.

GraphQL is powerful for complex, dynamic data fetching but may introduce complexity that isn't necessary for simpler APIs.

**A:**
### 1. Transport Mechanism

**WebSocket** is the most suitable transport mechanism for real-time notifications in this context. Here's why:

- **Real-time Communication**: WebSocket provides full-duplex communication channels over a single TCP connection, allowing real-time updates.
- **Low Latency**: Unlike polling, which can introduce latency and unnecessary network traffic, WebSocket maintains an open connection, reducing latency.
- **Efficiency**: WebSocket is more efficient than Server-Sent Events (SSE) for bidirectional communication, as SSE is primarily for server-to-client updates.
- **Scalability**: While WebSockets can be more challenging to scale than polling or SSE, modern architectures (e.g., using message brokers and load balancers) can handle this efficiently.

### 2. Database Schema for Notifications

Here's a proposed schema for storing notifications:


### 3. Handling Offline Users

To handle users who are offline:

- **Local Storage**: Use IndexedDB or localStorage to store notifications locally when a user is offline.
- **Queue System**: Implement a queue system (e.g., using Redis) to temporarily store notifications for offline users.
- **Sync on Reconnect**: When the user comes back online, sync the local storage or queue with the server to update the notification status.

### 4. Deadline Check Implementation

For checking deadlines every hour, an **event-driven approach** using a message queue (e.g., RabbitMQ, Kafka) is effective:

- **Scheduler**: Use a cron job to trigger a job every hour that publishes a message to a queue.
- **Worker**: A worker service listens to this queue and processes the message by checking all tasks for upcoming deadlines.
- **Notification**: If a deadline is within 24 hours, create a notification for the assigned user.

By implementing these strategies, the system can efficiently manage real-time notifications for a large number of concurrent users.
