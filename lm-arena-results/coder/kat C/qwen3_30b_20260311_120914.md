### 1. Prisma Query for 10 Most Recent Published Posts (No N+1)
```prisma
const posts = await prisma.post.findMany({
  take: 10,
  where: { published: true },
  orderBy: { createdAt: 'desc' },
  include: {
    author: { select: { email: true } }, // Only fetch email (not full user)
    tags: { select: { name: true } },    // Only fetch tag names
    comments: { select: { _count: true } } // Get count, not comments
  }
});
```

**Why this avoids N+1:**
- `include` fetches all required relations in **one query** (no additional DB calls).
- `select: { _count: true }` uses Prisma's built-in aggregate count (no need to load all comments).
- **Generated SQL** (simplified):
  ```sql
  SELECT 
    p.*, 
    a.email, 
    ARRAY_AGG(t.name) AS tags, 
    (SELECT COUNT(*) FROM comment c WHERE c.postId = p.id) AS comments_count
  FROM post p
  JOIN user a ON p.authorId = a.id
  JOIN post_tags pt ON p.id = pt.postId
  JOIN tag t ON pt.tagId = t.id
  WHERE p.published = true
  ORDER BY p.createdAt DESC
  LIMIT 10;
  ```

### 2. Safe Migration: Add Unique Non-Nullable `slug` to `Post` (10M Rows)
**Step-by-Step (Zero Downtime):**
1. **Add column as `NULL`** (avoids table lock during initial add):
   ```sql
   ALTER TABLE "Post" ADD COLUMN "slug" TEXT NULL;
   ```
   *Why?* PostgreSQL allows adding `NULL` columns without table-wide write lock.

2. **Populate existing rows in batches** (via background worker):
   ```prisma
   // Worker script (run in chunks, e.g., 10k rows at a time)
   const posts = await prisma.post.findMany({
     where: { slug: null },
     take: 10000
   });
   for (const post of posts) {
     await prisma.post.update({
       where: { id: post.id },
       data: { slug: slugify(post.title) } // e.g., "my-post-title"
     });
   }
   ```
   *Why?* Avoids blocking the table for 10M rows.

3. **Make column `NOT NULL`** (after all rows populated):
   ```sql
   ALTER TABLE "Post" ALTER COLUMN "slug" SET NOT NULL;
   ```

4. **Add unique constraint** (final step):
   ```sql
   ALTER TABLE "Post" ADD CONSTRAINT "Post_slug_unique" UNIQUE ("slug");
   ```

**Critical Notes:**
- **Never** run `ALTER TABLE ... ADD COLUMN ... NOT NULL` directly on 10M rows (locks table for hours).
- **Slug generation** must be deterministic (e.g., `slugify(title)` with conflict resolution).
- **Application change**: Update new posts to generate `slug` immediately (no nulls).

### 3. Backend Crashes (Exit Code 137): Meaning & Diagnosis

**Meaning:**  
Exit code `137` = **`SIGKILL`** (process killed by the OS due to **out-of-memory** (OOM).

**Diagnosis Steps:**
1. **Check Container Memory Limits:**
   ```bash
   docker inspect <container> | grep -i memory
   ```
   - If `Memory: 512m`, but your app needs 1GB → **increase limit**.

2. **Monitor Memory Usage:**
   ```bash
   docker stats <container>  # Check real-time memory
   ```

3. **Check Logs for OOM Errors:**
   ```bash
   docker logs <container> | grep -i "Out of memory"
   ```

4. **Profile Memory Usage:**
   - In Node.js: `node --inspect-brk` + Chrome DevTools Memory tab
   - Use `heapdump` package to capture memory snapshots.

5. **Common Causes:**
   - Memory leak in code (e.g., unbounded event listeners)
   - Large dataset processed without streaming
   - **Too many concurrent requests** (scale up container memory or add load balancing)

**Fix:**  
- Increase container memory limit in `docker-compose.yml`:
  ```yaml
  backend:
    deploy:
      resources:
        limits:
          memory: 1G
  ```
- Optimize code (e.g., paginate large queries, use streams).

### 4. Zero-Downtime Deployment (Additive DB Schema Change)

**Assumptions:**  
- New column is `NULL` (no data migration needed)
- Backend supports old/new schema simultaneously
- Load balancer (e.g., Nginx, AWS ALB) with health checks

**Deployment Sequence:**

1. **Apply DB Migration (Pre-Deployment):**  
   ```bash
   # Run migration on production DB (safe since column is NULL)
   psql -h db-host -U user -d app_db -c "ALTER TABLE users ADD COLUMN new_field TEXT NULL;"
   ```

2. **Deploy New Backend Version (Rolling Update):**  
   - **Step 1:** Start new containers with the new code (using the *same* DB schema).
   - **Step 2:** Wait for health checks to pass (`/health` endpoint).
   - **Step 3:** Remove old containers *only after* new ones are healthy (load balancer routes traffic to new instances first).

   *Example using Docker Swarm:*
   ```bash
   docker service update --image myapp:2.1 --force myapp_service
   ```

3. **Verify & Finalize:**  
   - Monitor logs for errors in new instances.
   - Confirm all requests are routed to new containers.
   - **Do NOT delete old containers until the new version is verified** (rollback path).

**Why This Works:**  
- The new column is `NULL`, so old code (running on old containers) still works.
- Load balancer drains old containers *after* new ones are healthy → **zero downtime**.
- DB migration runs *before* new code deploys → no schema mismatch.

> ⚠️ **Critical:** Never run `ALTER TABLE` with `NOT NULL` during deployment! This would block writes until data is populated (causing downtime).

**Q:** You are asked to add real-time notifications to the existing project management tool. Users should be notified when: a task is assigned to them, a comment is added to their task, a deadline is approaching (within 24h).

Design the notification system:
1. What transport mechanism would you use and why? (WebSocket, SSE, polling, push notifications — compare and justify)

3. How do you handle a user who is offline — do they get missed notifications when they come back online?
4. The deadline check needs to run every hour for all tasks. How do you implement this? (cron job, queue, event-driven?) — show the implementation approach.
5. How do you scale this if there are 100,000 concurrent users?

**A:**
Here's a comprehensive design for the real-time notification system, optimized for performance, reliability, and scalability:

---

### **1. Transport Mechanism: WebSocket (with SSE fallback)**
**Why WebSocket over alternatives?**
| **Mechanism** | **Pros**                          | **Cons**                                  | **Verdict** |
|---------------|-----------------------------------|-------------------------------------------|-------------|
| **WebSocket** | Bidirectional, low latency, efficient (1 connection per client), supports persistent state | Requires client-side implementation | ✅ **Best choice** |
| SSE           | Simple server-to-client streaming | No client-to-server communication, limited browser support (older browsers) | ❌ Not ideal for interactive UI |
| Polling       | Simple, works everywhere          | High server load (1000s of requests/sec), high latency, wasteful | ❌ Inefficient |
| Push Notifications | Mobile-only, requires app install | Not applicable for web UI, complex setup (FCM/APNs) | ❌ Wrong context |

**Justification:**  
- **Real-time need:** Users require *immediate* notifications (e.g., task assignment), making polling/SSE inadequate.  
- **Efficiency:** WebSocket uses a single persistent connection (vs. polling’s 50+ requests/sec per user). For 100k users, WebSocket reduces server load by **>99%** vs. polling.  
- **Web context:** Project management tools are primarily web-based (not mobile-only), so WebSocket is the standard.  
- **Fallback:** Use SSE for browsers without WebSocket support (e.g., old IE), but prioritize WebSocket.

---

### **2. Database Schema**
**Table: `notifications`**
```sql
CREATE TABLE notifications (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL,          -- FK to users.id
  type VARCHAR(20) NOT NULL,      -- 'task_assigned', 'comment_added', 'deadline_approaching'
  task_id UUID NOT NULL,          -- FK to tasks.id
  content TEXT,                   -- For comments: "User X commented: 'Check this'"
  triggered_at TIMESTAMPTZ NOT NULL, -- When notification was triggered (e.g., task assigned)
  expires_at TIMESTAMPTZ,         -- For deadlines: deadline + 24h (null for assignments/comments)
  is_read BOOLEAN DEFAULT false,
  created_at TIMESTAMPTZ DEFAULT NOW(),
  updated_at TIMESTAMPTZ DEFAULT NOW()
);

-- Indexes for performance
CREATE INDEX idx_user_read ON notifications (user_id, is_read);
CREATE INDEX idx_expires ON notifications (expires_at) WHERE expires_at IS NOT NULL;
```

**Key Design Choices:**  
- **`expires_at`**: For deadlines, stores the time when the notification should be *automatically cleared* (after 24h).  
- **`type`**: Avoids complex joins; client uses this to render UI.  
- **Indexing**: `user_id + is_read` enables fast fetching of unread notifications on login.  
- **No `sent` flag**: Notifications are *stored* first; delivery happens via WebSocket/SSE.

---

### **3. Handling Offline Users**
**Mechanism:** **Store all notifications in DB → Deliver on reconnect.**  
- **How it works:**  
  1. When a notification is triggered (e.g., task assigned), it’s **immediately written to `notifications`**.  
  2. User’s client disconnects → WebSocket closes → **No delivery loss**.  
  3. On reconnection:  
     - Client sends `GET /notifications?unread=true` to fetch *all unread notifications*.  
     - Server returns list (max 50 per request, paginated).  
     - Client marks them as read *after display* (or on `is_read` update via API).  
- **Why not push to mobile?**  
  Web apps don’t have mobile push capabilities. Storing in DB is the only reliable way.  
- **Edge case:** If user has 10k unread notifications, server sends in batches (e.g., 50 at a time).
