6. Token Blacklisting vs. Rotation:

Token Blacklisting:
- Maintains a blacklist of invalid tokens
- Tokens can be manually blacklisted (e.g., logout, password change)
- Works with both short-lived and long-lived tokens
- Can lead to memory issues with many tokens
- Good for scenarios where you need fine-grained control over token invalidation

Token Rotation:
- Automatically replaces old tokens with new ones
- Provides better security against token theft
- Tokens have built-in expiration
- No need to maintain a blacklist
- More suitable for long-lived tokens
- Better performance (no additional lookup needed)

When to use each:
- Use Blacklisting when you need to invalidate specific tokens immediately (e.g., logout, compromised tokens)
- Use Rotation for general session management with built-in expiration and automatic refresh
- Can use both in combination: rotation for normal session management and blacklisting for special cases

The combination of both approaches provides the most robust security:
- Rotation handles normal token lifecycle
- Blacklisting handles exceptional cases
- Both can use Redis with appropriate TTLs to manage memory usage

```tsx
  header: string;
  width?: number;
  sortable?: boolean;
  renderCell?: (data: T, rowIdx: number) => React.ReactNode;
}

interface DataTableProps<T> {
  data: T[];
  columns: Column<T>[];
  pageSize?: number;
  className?: string;
}

// Main Component
const DataTable: React.FC<DataTableProps<any>> = ({
  data,
  columns,
  pageSize = 10,
  className = '',
}) => {
  // State
  const [sortColumn, setSortColumn] = useState<Column<any> | null>(null);
  const [sortOrder, setSortOrder] = useState<'asc' | 'desc' | null>(null);
  const [selectedRows, setSelectedRows] = useState<string[]>([]);
  const [currentPage, setCurrentPage] = useState(1);

  // Derived state
  const sortedData = useMemo(() => {
    if (!sortColumn) return data;

    const column = columns.find(c => c.key === sortColumn.key);
    if (!column) return data;

    return [...data].sort((a, b) => {
      const aValue = (a as any)[column.key];
      const bValue = (b as any)[column.key];

      if (sortOrder === 'asc') {
        return aValue > bValue ? 1 : -1;
      } else {
        return aValue < bValue ? 1 : -1;
      }
    });
  }, [data, sortColumn, sortOrder]);

  const paginatedData = useMemo(() => {
    const start = (currentPage - 1) * pageSize;
    return sortedData.slice(start, start + pageSize);
  }, [sortedData, currentPage, pageSize]);

  // Handlers
  const handleSort = (column: Column<any>) => {
    if (!column.sortable) return;

    if (sortColumn?.key === column.key) {
      setSortOrder(sortOrder === 'asc' ? 'desc' : null);
    } else {
      setSortColumn(column);
      setSortOrder('asc');
    }
  };

  const handleRowSelect = (rowIdx: number) => {
    const rowId = uuidv4();
    const newSelection = [...selectedRows];
    const isRowSelected = newSelection.includes(rowId);

    if (isRowSelected) {
      newSelection.splice(newSelection.indexOf(rowId), 1);
    } else {
      newSelection.push(rowId);
    }

    setSelectedRows(newSelection);
  };

    const newSelection = checked

              <th

                onClick={() => handleSort(col)}
                style={{ width: col.width }}
              >
                {col.header}
                {sortColumn?.key === col.key && (
                  <span>
                    {sortOrder === 'asc' ? ' ↑' : sortOrder === 'desc' ? ' ↓' : ''}
                  </span>
                )}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {paginatedData.map((row, idx) => (

                <input

                  checked={selectedRows.includes(uuidv4())}

              </td>

          Next
```

**A:**
Here's a basic implementation of a collaborative document editor server in Node.js using the `ws` library. I'll address each requirement and provide explanations.

```javascript
const WebSocket = require('ws');
const wss = new WebSocket.Server({ port: 8080 });

// Connection state management
const connections = new Map(); // user -> {socket, document}
const documents = new Map();   // documentId -> {users, cursorPositions, content}

// Message types
const MSG_TYPES = {
    JOIN: 'join',
    TEXT_CHANGE: 'textChange',
    CURSOR_MOVE: 'cursorMove',
    USER_LEFT: 'userLeft',
    NEW_CONTENT: 'newContent'
};

// Connection handling
wss.on('connection', (ws) => {
    ws.on('message', (message) => {
        try {
            const msg = JSON.parse(message);
            handleMessage(ws, msg);
        } catch (err) {
            console.error('Error parsing message:', err);
        }
    });

    ws.on('close', () => {
        const user = connections.get(ws);
        if (user) {
            leaveDocument(user);
        }
    });
});

function handleMessage(ws, message) {
    switch (message.type) {
        case MSG_TYPES.JOIN:
            joinDocument(ws, message);
            break;
        case MSG_TYPES.TEXT_CHANGE:
            textChange(ws, message);
            break;
        case MSG_TYPES.CURSOR_MOVE:
            cursorMove(ws, message);
            break;
        default:
            console.error('Unknown message type:', message.type);
    }
}

function joinDocument(ws, message) {
    const { user, documentId } = message;
    const userDoc = documents.get(documentId);
    if (!userDoc) {
        userDoc = {
            users: new Set(),
            cursorPositions: new Map(),
            content: ''
        };
        documents.set(documentId, userDoc);
    }

    const userState = {
        socket: ws,
        documentId
    };
    connections.set(ws, userState);
    userDoc.users.add(user);
    userDoc.cursorPositions.set(user, { x: 0, y: 0 });

    // Send current document content to new user
    ws.send(JSON.stringify({
        type: MSG_TYPES.NEW_CONTENT,
        content: userDoc.content
    }));

    // Notify other users of new user joining
    broadcastToDocument(documentId, {
        type: MSG_TYPES.USER_LEFT,
        user: 'system',
        message: `${user} joined`
    });
}

function leaveDocument(userState) {
    const { user, documentId } = userState;
    const userDoc = documents.get(documentId);
    if (userDoc) {
        userDoc.users.delete(user);
        userDoc.cursorPositions.delete(user);
        connections.delete(userState.socket);
        
        // Notify other users
        broadcastToDocument(documentId, {
            type: MSG_TYPES.USER_LEFT,
            user: user,
            message: 'left'
        });
    }
}

function textChange(ws, message) {
    const { user, documentId, position, text } = message;
    const userDoc = documents.get(documentId);
    if (userDoc) {
        // Apply change to document content
        let newContent = userDoc.content;
        newContent = newContent.substring(0, position) + text + newContent.substring(position);
        userDoc.content = newContent;

        // Broadcast change to other users
        broadcastToDocument(documentId, {
            type: MSG_TYPES.TEXT_CHANGE,
            user,
            position,
            text
        });
    }
}

function cursorMove(ws, message) {
    const { user, documentId, x, y } = message;
    const userDoc = documents.get(documentId);
    if (userDoc) {
        userDoc.cursorPositions.set(user, { x, y });
        
        // Broadcast cursor movement
        broadcastToDocument(documentId, {
            type: MSG_TYPES.CURSOR_MOVE,
            user,
            x,
            y
        });
    }
}

function broadcastToDocument(documentId, message) {
    const userDoc = documents.get(documentId);
    if (userDoc) {
        userDoc.users.forEach(user => {
            const ws = connections.get(user).socket;
            if (ws && ws.readyState === WebSocket.OPEN) {
                ws.send(JSON.stringify(message));
            }
        });
    }
}
```

This implementation addresses:

1. **Connection handling**:
   - Uses `connections` map to track user-to-WebSocket associations
   - Uses `documents` map to track document state and connected users

2. **Message protocol**:
   - Defines JSON message types for joining, text changes, cursor movements, and user disconnections
   - Each message includes a `type` field and relevant data

3. **Broadcasting**:
   - Uses `broadcastToDocument` function to send messages to all users in the same document except the sender
   - Checks WebSocket state before sending to avoid sending to disconnected clients

4. **Reconnection**:
   - When a user reconnects, they need to send a JOIN message with their document ID
   - The server sends the latest document content back to the reconnected user

5. **Conflict Resolution**:
   - The current implementation doesn't handle concurrent edits properly
   - Real solutions would involve:
     - Operational Transformation (OT) to resolve concurrent changes
     - Conflict-free Replicated Data Types (CRDTs) for eventual consistency
     - Last-write-wins with conflict markers

To handle concurrent edits, you would need to implement one of the above approaches. OT would track operations and transform them to maintain consistency, while CRDTs would allow each user to have their own version and merge them automatically.

Let me break down the design for a real-time notification system for the project management tool.

1. Transport Mechanism:
I recommend using WebSocket combined with Push Notifications as a fallback:

- WebSocket for Real-Time Communication:
  - Bidirectional communication allows immediate delivery of notifications
  - Low latency and efficient for real-time updates
  - Can handle multiple concurrent connections per user
  - Ideal for browser-based applications

- Push Notifications (Firebase Cloud Messaging, Apple Push Notification Service) as fallback:
  - Delivers notifications when the user's app is in the background or closed
  - Works even if the user isn't actively using the application
  - Can target specific devices or user groups

Why not polling/SSE:
- Polling would be inefficient and lead to higher server load
- SSE is unidirectional, not ideal for real-time bidirectional communication
- WebSocket with push notifications offers the best balance of real-time capabilities and offline support

2. Database Schema:
```sql
CREATE TABLE notifications (
    id SERIAL PRIMARY KEY,
    user_id INTEGER REFERENCES users(id),
    type VARCHAR(50) NOT NULL, -- 'task_assignment', 'comment', 'deadline_reminder'

    content TEXT,
    is_read BOOLEAN DEFAULT FALSE,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
```

3. Handling Offline Users:
