## Question 12
**Q:** You are asked to add real-time notifications to the existing project management tool. Users should be notified when: a task is assigned to them, a comment is added to their task, a deadline is approaching (within 24h).

Design the notification system:
1. What transport mechanism would you use and why? (WebSocket, SSE, polling, push notifications — compare and justify)
2. How do you store notifications? Design the database schema.
3. How do you handle a user who is offline — do they get missed notifications when they come back online?
4. The deadline check needs to run every hour for all tasks. How do you implement this? (cron job, queue, event-driven?) — show the implementation approach.
5. How do you scale this if there are 100,000 concurrent users?

**A:**
To add real-time notifications to the project management tool, here's a structured approach addressing each requirement:

### 1. Transport Mechanism
**WebSocket** is the chosen transport mechanism. It's ideal for real-time communication, providing two-way, event-driven communication over the web, making it efficient for handling multiple concurrent users without the overhead of polling.

### 2. Database Schema
A Notifications table will be created to store all relevant information:

- `notification_id` (Primary Key, UUID): Unique identifier for each notification.
- `user_id` (Foreign Key): References the user who received the notification.
- `task_id` (Foreign Key): References the task associated with the notification.
- `type` ( ENUM): Type of notification (e.g., 'task_assigned', 'comment_added', 'deadline_reminder').
- `message` ( TEXT): Detailed message of the notification.
- `created_at` ( TIMESTAMP): When the notification was created.
- `read_at` ( TIMESTAMP): When the notification was read, nullable.
- `comment_id` (Foreign Key, Nullable): References a comment if applicable.

### 3. Handling Offline Users
Notifications for offline users are stored in the database. When they come online, the WebSocket server sends all unread notifications. Additionally, the application will check for unread notifications upon the user's next login to ensure they don't miss any updates.

### 4. Implementing Deadline Checks
A cron job will be scheduled to run every hour. This job will query all tasks with deadlines within the next 24 hours and trigger notifications through the WebSocket server or via email/push notifications.

### 5. Scaling for 100,000 Users
To handle scalability:
- **WebSocket Scaling**: Use a load balancer with multiple WebSocket servers and cluster them for session management.
- **Message Broker**: Implement a message broker (like Redis or RabbitMQ) to manage WebSocket connections efficiently.
- **Horizontal Scaling**: Add more backend servers and scale the database as needed.
- **Efficient Transport**: Optimize WebSocket connections and ensure the frontend efficiently handles notifications to reduce server load.

### Summary
- **Transport**: WebSocket for real-time communication.
- **Storage**: Notifications table in the database.
- **Offline Handling**: Store notifications and send upon reconnection.
- **Deadline Checks**: Cron job for hourly task deadline checks.
- **Scalability**: Use load balancing, clustering, and efficient transport mechanisms to support 100,000 users.

This approach ensures real-time notifications are efficiently handled, even at scale.
