import { taskValidationSchema } from '../validation/taskValidation';
import { notFound } from 'express-middleware-utils';
import { errorHandler } from '../middlewares/errorHandler';
import { TaskModel } from '../models/taskModel'; // Assuming you have a Task model set up
import { WorkspaceModel } from '../models/workspaceModel'; // Assuming you have a Workspace model set up

export const updateTask = async (req: Request, res: Response) => {
  try {
    const { workspaceId, taskId } = req.params;
    const { error } = taskValidationSchema.parse(req.body);
    if (error) {
      return res.status(400).json({ message: error.message });
    }

    const task = await TaskModel.findById(taskId);
    if (!task) {
      return notFound('Task not found');
    }

    const workspace = await WorkspaceModel.findById(workspaceId);
    if (!workspace) {
      return notFound('Workspace not found');
    }

    if (!workspace.members.includes(req.body.assignee_id as number)) {
      return res.status(403).json({ message: 'Assignee is not a member of the workspace' });
    }

    const subtasksCount = await TaskModel.countDocuments({ workspaceId, parentTaskId: taskId });
    if (task.status === 'cancelled' && subtasksCount > 0) {
      return res.status(409).json({ message: 'Cannot set status to "cancelled" when there are open subtasks' });
    }

    await task.updateOne(req.body);
    res.status(200).json({ message: 'Task updated successfully' });
  } catch (err) {
    errorHandler(err, req, res, next);
  }
};

**Q:** Continuing from the previous auth system:

1. POST /auth/refresh — use the refresh token from cookie to issue a new access token. Rotate the refresh token (issue new one, invalidate old one).
2. POST /auth/logout — invalidate the refresh token, clear the cookie

**A:**
1. **POST /auth/refresh**: To implement this endpoint, you would need to:

   - Extract the refresh token from the HTTP cookie.
   - Validate the refresh token (ensure it's not expired, not revoked, etc.).
   - If valid, generate a new access token and a new refresh token.
   - Store the new refresh token in your Redis store with an appropriate TTL (Token Time To Live).
   - Invalidate the old refresh token by removing it from the Redis store.
   - Return the new access token to the client.

   Here's a simplified example of how the Redis data structure might look after the refresh token has been rotated:

   - `old_refresh_token`: expires in 1 minute (for invalidation)
   - `new_refresh_token`: expires in the usual duration for a refresh token (e.g., 2 weeks)

   The TTL strategy here is to have a very short TTL for the old refresh token to ensure it's quickly invalidated. The new refresh token has a longer TTL appropriate for its intended lifespan.

2. **POST /auth/logout**: To implement the logout endpoint, you would:

   - Extract the refresh token from the HTTP cookie.
   - Validate the refresh token (if necessary).
   - Invalidate the refresh token by removing it from the Redis store.
   - Clear the cookie from the client's browser to prevent further use of the refresh token.

3. **Implement refresh token rotation with a Redis-based token store**: You can use a Redis `Hash` to store the user's tokens with keys as the token identifiers and fields for the access token and refresh token. For each request that uses an access token, you would also retrieve the associated refresh token from the Redis store and check if it has been rotated. If it has, use the new refresh token for subsequent operations.

   Example Redis command to store tokens:

   ```redis
   HMSET user:101 access_token:new_access_token refresh_token:new_refresh_token
   ```

   And to retrieve and rotate the token (pseudo-code):

   ```python
   # Assuming 'redis' is a connected Redis client
   user_id = extract_user_id_from_access_token()
   tokens = redis.hgetall(f"user:{user_id}")
   old_refresh_token = tokens['refresh_token']

   # Rotate the refresh token
   new_refresh_token = generate_new_refresh_token()
   redis.set(f"refresh_token:{old_refresh_token}", new_refresh_token, ex=REFRESH_TOKEN_EXPIRY)
   redis.set(f"refresh_token:{new_refresh_token}", new_refresh_token, ex=REFRESH_TOKEN_EXPIRY)
   redis.hset(f"user:{user_id}", 'refresh_token', new_refresh_token)

   # Invalidate the old refresh token
   redis.set(f"refresh_token:{old_refresh_token}", '', ex=1)  # 1 minute TTL to expire and invalidate
   ```

4. **Handle the case where a refresh token is used twice**: To detect token reuse, you can:

   - Implement a check within your token validation logic to see if the refresh token has been previously validated or used.
   - Store a usage count or a timestamp of the last use in Redis along with the token.
   - If the same refresh token is presented for validation again within a short time frame or has been used the maximum allowed number of times, flag it as potentially reused.

   Example Redis command to store token usage information:

   ```redis
   HMSET refresh_token:123 usage_count:1 last_used:UNIX_TIMESTAMP
   ```

   When validating the token, you would check the `usage_count` and `last_used` fields to determine if the token has been reused.

5. **A user changes their password**: To invalidate all existing sessions, you would:

   - When a user changes their password, generate a new set of tokens for the user.
   - Store these new tokens in Redis with the old tokens being invalidated by setting them to an empty string or to a special 'revoked' status with a very short TTL.
   - Optionally, send a notification to the user's devices (if applicable) advising them of the session update.

   Example Redis command to invalidate old tokens:

   ```redis
   HMSET user:101 access_token: new_access_token refresh_token: new_refresh_token status:revoked
   ```

6. **Difference between token blacklisting and token rotation**:

   - **Token Blacklisting**: This is the process of marking a token as invalid or "blacklisted" once it's been compromised or used beyond its intended lifecycle. Blacklisted tokens are checked against a list, and any request attempting to use a blacklisted token is denied. This is a reactive measure, typically used in cases of token compromise.

   - **Token Rotation**: This is a proactive security measure where new tokens are issued as part of the normal lifecycle of a token. The new tokens invalidate the old tokens, which are then allowed to expire. This approach helps to mitigate the risk of token leakage or long-term exposure.

   When to use each:

   - Use **token rotation** to manage the normal lifecycle of tokens and to ensure that tokens are regularly refreshed and updated.
   - Use **token blacklisting** when there is a suspicion that a token has been compromised or is being misused, even before its regular expiration. This allows for immediate action to prevent unauthorized access.

Implementing both strategies provides a robust security posture for your authentication system.

2. The complete component implementation (controlled version):

```typescript
import React, { useMemo, useState, useEffect } from 'react';
import { DataTableProps, DataTableStateProps } from './DataTableProps';

const DataTableComponent: React.FC<DataTableProps<DataTableRow>> & DataTableState = (props) => {
  const { columns, data, sortColumn, onSort, rowSelection, pagination, isControlled, controlledPaginationData } = props;
  const [stateProps, setStateProps] = useState<DataTableStateProps<DataTableRow>>();

  useEffect(() => {
    if (isControlled) {
      setStateProps(controlledPaginationData);
    } else {
      const sortedData = data.sort((a, b) => {
        if (!sortColumn || !stateProps.sortColumn) return 0;
        if (a[stateProps.sortColumn] < b[stateProps.sortColumn]) return -1;
        if (a[stateProps.sortColumn] > b[stateProps.sortColumn]) return 1;
        return 0;
      });

      const selectedRowIds = rowSelection?.onSelect
        ? Object.keys(rowSelection.enabled ? {} : undefined)
        : [];

      const totalRows = data.length;
      setStateProps({ sortedData, selectedRowIds, currentPage: 1, totalRows, pageSize: 10 });
    }
  }, [data, sortColumn, stateProps.sortColumn, rowSelection, isControlled, controlledPaginationData]);

  const { sortedData, selectedRowIds, currentPage, totalRows, pageSize } = stateProps;

  const handleSort = (column: string) => {
    if (sortColumn === column) {
      onSort(column, stateProps.sortColumn === column ? 'none' : stateColumn.sortOrder);
    } else {
      onSort(column, 'asc');
    }
  };

  const handleSelect = (isSelected: boolean, id: string) => {
    if (rowSelection && rowSelection.enabled) {
      const newSelectedRowIds = isSelected
        ? [...selectedRowIds, id]
        : selectedRowIds.filter((rowId) => rowId !== id);
      rowSelection.onSelect && rowSelection.onSelect(newSelectedRowIds);
      setStateProps({ ...stateProps, selectedRowIds: newSelectedRowIds });
    }
  };

  const handlePageChange = (pageNumber: number) => {
    setStateProps({
      ...stateProps,
      currentPage: pageNumber,
    });
  };

  const paginationOptions = useMemo(
    () => pagination?.pageSizeOptions.map((option) => option * option), // Square the values for spacing
    [pagination]
  );

  return (
    <div>
      <table>
        <thead>
          <tr>
            {columns.map((column) => (
              <th key={column.key} onClick={() => handleSort(column.key)}>
                {column.header}
                {column.sortable && (
                  <span>
                    {stateProps.sortColumn === column.key && stateProps.sortOrder !== 'none' && ' (' + stateProps.sortOrder + ')'}
                  </span>
                )}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {sortedData.slice((currentPage - 1) * pageSize, currentPage * pageSize).map((row) => (
            <tr key={row.id}>
              {columns.map((column) => {
                const value = column.renderCell ? column.renderCell(row) : row[column.key];
                return <td key={column.key + '-' + row.id}>{value}</td>;
              })}
              <td>
                <input
                  type="checkbox"
                  checked={selectedRowIds.includes(row.id)}
                  onChange={(e) => handleSelect(e.target.checked, row.id)}
                />
              </td>
            </tr>
          ))}
        </tbody>
      </table>
      <div>
        {paginationOptions && (
          <pagination
            currentPage={currentPage}
            totalPages={Math.ceil(totalRows / pageSize)}
            onPageChange={handlePageChange}
          />
        )}
      </div>
    </div>
  );
};

export default DataTableComponent;
```

3. **N+1 Problem Example and Solution:**

**Problem:**
Suppose we have 10 recipes, and we fetch 3 of them using the `recipes` query. For each recipe, we make a separate database call to get the author's information. If each call is represented as `N`, we end up with `10 * N` calls in total, which is inefficient.

**Solution:**
We can solve this using `DataLoader`. By loading all ratings in a batch, we can reduce the number of calls. The `batchRatings` DataLoader in the resolver example above aggregates all necessary user data into a single call, thus mitigating the N+1 problem.

5. **When to Choose GraphQL over REST and Vice Versa:**

**Choose GraphQL over REST when:**
- You want to fetch exactly the data you need in a single request.
- Your clients require a flexible and dynamic query pattern, where the shape of the data can vary greatly.
- You have a complex UI that needs to fetch different types of data (e.g., user data, comments, posts) in a single request.
- You want to reduce the number of network requests, which can improve performance, especially on mobile networks.
- You need strong typing in your API with schema validation.

**Choose REST over GraphQL when:**
- Your data structure is relatively static and doesn't change often.
- You prefer a more traditional web API that is broadly understood and supported by a vast ecosystem of tools and documentation.
- Your clients are simple and only require a fixed set of resources.
- You have legacy systems that are easier to adapt to REST than to implement a new query language like GraphQL.
- You want to leverage caching at the CDN level, which is more straightforward with REST endpoints.
- You're working with a team that has more experience with RESTful services.

In summary, GraphQL is often preferred for complex or flexible data requirements, while REST is suitable for simpler, more static data structures and when leveraging the existing REST ecosystem.

**Q:** You are asked to add real-time notifications to the existing project management tool. Users should be notified when: a task is assigned to them, a comment is added to their task, a deadline is approaching (within 24h).

Design the notification system:

**A:**
1. **Transport Mechanism for Real-Time Notifications:**

   - **WebSockets**: WebSockets provide a full-duplex communication channel over a single long-lived connection. It's ideal for real-time applications because it can push data from the server to the client instantly. WebSockets are widely supported by modern browsers and have server-side implementations in various languages (e.g., Node.js with `ws` library).
   
   - **Server-Sent Events (SSE)**: SSE is a server push technology in which the server can send new data to a web client anytime by initiating a new connection from the server to the client. SSE is simpler to implement than WebSockets because it only allows data to flow from the server to the client. It's supported by all modern browsers without the need for any special configuration.
   
   - **Polling**: This is the most straightforward approach where the client periodically sends AJAX requests to the server to check for new notifications. However, it's not as efficient as WebSockets or SSE because it involves HTTP requests and can introduce latency. It's also less scalable due to the load it puts on the server.
   
   - **Push Notifications**: This approach relies on a device's push notification service, which is supported by mobile devices. It's useful for mobile applications and can be used in conjunction with WebSockets or SSE for a more comprehensive solution.

   **Justification**: For a project management tool that requires immediate notifications, WebSockets would be the best choice due to its real-time, full-duplex communication capabilities. It can handle the necessary bidirectional communication and is widely supported. If the application also has a mobile component, push notifications could be used as a fallback or additional mechanism for mobile users.

2. **Database Schema for Storing Notifications:**

   A simple and scalable schema could look like this:

   - **Users Table**: `user_id`, `name`, `email`, etc.
   - **Tasks Table**: `task_id`, `assigned_to_user_id`, `description`, `deadline`, etc.
   - **Notifications Table**: `notification_id`, `user_id`, `task_id`, `type` (e.g., 'task_assigned', 'comment_added', 'deadline_approaching'), `message`, `read` (boolean), `created_at`, `updated_at`.
   - **Notification_Log Table**: `log_id`, `notification_id`, `timestamp`, `user_id` (to track offline notifications).

3. **Handling Users Who Are Offline:**

   When a user goes offline, the application should continue to receive events and store them in a notification log. When the user comes back online, the application should sync the log to retrieve any missed notifications. This can be achieved by:

   - Detecting the user's offline status (e.g., on `beforeunload` event in browsers).
   - Storing missed notifications in the `Notification_Log` table with the user's ID and timestamps.
   - When the user is back online, the client application can request all entries from the `Notification_Log` table that match the user's ID and marked as unread.

4. **Implementing Deadline Check:**

   The deadline check can be implemented using a combination of a scheduled task (cron job) and a message queue (e.g., RabbitMQ, AWS SQS). Here's an approach:

   - **Scheduled Task**: A cron job runs every hour to trigger a backend service that publishes a message to a queue with a list of task IDs that are approaching their deadlines.
   
   - **Message Queue**: The queue receives the task IDs and distributes them to worker processes that check the status of each task and notify the relevant users if necessary. This approach ensures that the deadline check is performed consistently and can be easily scaled.

5. **Scaling for 100,000 Concurrent Users:**

   To handle 100,000 concurrent users, the system must be designed for high availability and scalability:

   - **Load Balancers**: Distribute the load across multiple servers or cloud instances.
   - **Horizontal Scaling**: Use WebSockets or SSE with a load balancer that can handle many connections simultaneously.
   - **Database Scaling**: Use a distributed database system that can handle high read/write loads and scale horizontally (e.g., Amazon RDS with read replicas, Google Cloud Spanner).
   - **Caching**: Implement caching for frequent database queries using Redis or Memcached to reduce database load.
   - **Microservices Architecture**: Break down the application into microservices to isolate and scale different parts of the system independently.
   - **Queueing System**: Ensure that the message queue can handle a high volume of messages without bottlenecking (e.g., AWS SQS with fan-out pattern to multiple worker services).
   - **Monitoring and Alerting**: Use monitoring tools to keep track of the system's health and performance metrics, and set up alerting to respond to anomalies or failures quickly.
   - **Rate Limiting and Backpressure**: Implement rate limiting to prevent abuse and backpressure mechanisms to handle overload situations.

By considering these aspects, the notification system can be designed to be real-time, reliable, and scalable to accommodate a large number of users.
