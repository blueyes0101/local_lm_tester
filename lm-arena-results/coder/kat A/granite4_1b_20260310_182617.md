**A:**
1. **POST /auth/refresh**

When a user refreshes their access token, they make a POST request to the `/auth/refresh` endpoint with their refresh token. The server validates the refresh token against a secure Redis-based token store to ensure its validity. Once validated, it generates a new access token and issue a new refresh token. The existing refresh token is then rotated and invalidated, while the new token is stored in the Redis store with an appropriate Time To Live (TTL) strategy.

2. **POST /auth/logout**

When a user logs out, they make a POST request to the `/auth/logout` endpoint. This endpoint invalidates the refresh token by removing it from the Redis store and clears the corresponding cookie. The user is then redirected to the authentication page to re-authenticate.

3. **Implementing Refresh Token Rotation with Redis**

To store the refresh tokens in Redis, we can use a data structure like a hash to store each token's information, including the token itself, the user ID, and the expiration timestamp. Here's an example of how the Redis data structure could look like:

```
{
  "refresh_token_abc123": {
    "token": "refresh_token_abc123",
    "user_id": "user_id_123",
    "expires_at": "timestamp"
  }
}
```

To implement TTL (Time To Live) strategy, we can set the expiration time for each token in the hash, which will automatically invalidate the token after it expires. For example, we could set a TTL of 30 days for each refresh token.

4. **Handling Token Reuse**

To handle the case where a refresh token is used twice, we can implement token reuse detection by checking the status of the token in Redis before issuing a new access token. If the token already exists in the Redis store, it means the token has been used before, and an error should be thrown to prevent unauthorized access.

5. **Invalidating All Existing Sessions on Password Change**

When a user changes their password, we invalidate all existing sessions by generating a new refresh token for the user and storing it in the Redis store. This effectively creates a new session for the user, and all previous sessions are considered invalid.

6. **Token Blacklisting vs. Token Rotation**

Token blacklisting and token rotation are two different strategies for managing access tokens:

- **Token Blacklisting**: This involves maintaining a blacklist of invalid or revoked tokens. When a user logs in, the server checks the token against the blacklist before issuing a new access token. This approach can handle token reuse attacks, as the server can invalidate the token if it is found on the blacklist.

- **Token Rotation**: This involves periodically rotating the access and refresh tokens. Each time a token is rotated, the old token is invalidated, and a new one is issued. This approach provides better security, as each token has a limited lifetime, and the likelihood of a token being reused is reduced.

When to use each:

- **Token Blacklisting**: Use when you need to quickly invalidate access tokens without affecting the user's experience. It's a good choice when you have a small number of tokens to handle and need to maintain a relatively simple token management system.

- **Token Rotation**: Use when you have a large number of tokens, or when you need to implement a more robust token management system with a to-and-from strategy. Token rotation provides better security by reducing the risk of token reuse attacks and allows for easier of managing token lifetimes.
