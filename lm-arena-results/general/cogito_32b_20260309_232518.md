## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
SQL vs NoSQL: Choosing the Right Database for Your Application

The fundamental difference between SQL and NoSQL databases lies in their data structure and query capabilities. SQL databases use structured, tabular formats with predefined schemas, typically following the relational model. Data is stored in tables with rows and columns, and relationships between tables are maintained through primary and foreign keys. In contrast, NoSQL databases are schema-less and use various data models such as document (JSON/BSON), key-value pairs, column-family, or graph structures. This flexibility allows NoSQL databases to handle unstructured and semi-structured data more efficiently.

When it comes to scalability, NoSQL databases generally outperform SQL databases in horizontal scaling. NoSQL databases are designed to scale out by distributing data across multiple servers, making them ideal for handling large volumes of data and high traffic loads. They can easily add more nodes to the cluster to improve performance and availability. SQL databases, on the other hand, primarily scale vertically by adding more resources to a single server, which can become cost-prohibitive and may hit hardware limitations. However, modern SQL databases do offer some horizontal scaling capabilities, albeit with more complexity.

The choice between SQL and NoSQL often depends on specific use cases. SQL databases are best suited for applications requiring complex queries, transactions, and data integrity, such as financial systems, e-commerce platforms, and any application needing ACID compliance. Their structured nature makes them ideal for applications with well-defined data models that don't change frequently. NoSQL databases excel in applications requiring high scalability, flexible schema, and real-time data processing, such as social media platforms, IoT applications, and big data analytics. They are also preferred for handling unstructured data from various sources, like user-generated content or sensor data. The decision ultimately depends on the specific requirements of the project, including data structure, consistency needs, and scalability requirements.

**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here's how to configure iptables with the specified rules and their explanations:

1. First, flush existing rules and set policies:
```bash
iptables -F
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT
```
Explanation:
- `-F`: Flushes all existing rules
- `-P INPUT DROP`: Sets default policy to DROP incoming traffic
- `-P FORWARD DROP`: Sets default policy to DROP forwarded traffic
- `-P OUTPUT ACCEPT`: Sets default policy to ACCEPT outgoing traffic

2. Allow SSH (port 22) from 10.0.0.0/24:
```bash
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT
```
Explanation:
- `-A INPUT`: Append rule to INPUT chain
- `-p tcp`: Protocol is TCP
- `--dport 22`: Destination port is 22 (SSH)
- `-s 10.0.0.0/24`: Source IP range is 10.0.0.0/24
- `-j ACCEPT`: Accept matching packets

3. Allow HTTP and HTTPS (ports 80 and 443) from anywhere:
```bash
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```
Explanation:
- `-A INPUT`: Append rule to INPUT chain
- `-p tcp`: Protocol is TCP
- `--dport 80/443`: Destination ports are 80 (HTTP) and 443 (HTTPS)
- `-j ACCEPT`: Accept matching packets

4. Make rules persistent:

First, install iptables-persistent:
```bash
sudo apt-get install iptables-persistent
```

Save current rules:
```bash
sudo netfilter-persistent save
```

To ensure iptables rules are loaded on boot, install iptables-persistent and enable the service:
```bash
sudo systemctl enable netfilter-persistent
```

Complete script for all rules:
```bash
#!/bin/bash
iptables -F
iptables -P INPUT DROP
iptables -P FORWARD DROP
iptables -P OUTPUT ACCEPT
iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT
iptables -A INPUT -p tcp --dport 80 -j ACCEPT
iptables -A INPUT -p tcp --dport 443 -j ACCEPT
sudo netfilter-persistent save
```

Additional notes:
- The script should be run with sudo privileges
- To test changes, you can use `iptables -L -n -v` to list rules
- To remove all rules and reset to defaults, use `iptables -F` and `iptables -X`
- It's recommended to have a backup plan (like a direct connection) before implementing these rules to prevent locking yourself out

Remember to restart the service after making changes:
```bash
sudo systemctl restart netfilter-persistent
```

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
Let me explain the differences between Mock, Stub, and Fake, and then provide a Python example.

**Mock vs Stub vs Fake**

1. **Mock**
- An object that simulates the behavior of a real object
- Records interactions (calls, arguments, return values)
- Used to verify that the code under test behaves as expected
- Typically used for unit testing to isolate dependencies

2. **Stub**
- A simplified implementation that returns prearranged data
- Doesn't verify behavior, just provides test data
- Simpler than mocks, used when you only need to provide data
- Often used for integration tests

3. **Fake**
- A simplified implementation of an interface
- Behaves similarly to the real implementation but with fewer features
- Used when the real implementation is too slow or difficult to use
- Examples: In-memory database instead of real database

Here's a Python example using unittest.mock:

```python
import unittest
from unittest.mock import patch, MagicMock
import requests

class WeatherService:
    def get_temperature(self, city):
        response = requests.get(f"https://api.weather.com/temperature/{city}")
        if response.status_code != 200:
            raise Exception("API Error")
        return response.json()['temperature']

class TestWeatherService(unittest.TestCase):
    def setUp(self):
        self.weather_service = WeatherService()

    @patch('requests.get')
    def test_get_temperature_success(self, mock_get):
        # Arrange
        mock_response = MagicMock()
        mock_response.status_code = 200
        mock_response.json.return_value = {'temperature': 25}
        mock_get.return_value = mock_response

        # Act
        temperature = self.weather_service.get_temperature("London")

        # Assert
        self.assertEqual(temperature, 25)
        mock_get.assert_called_once_with("https://api.weather.com/temperature/London")

    @patch('requests.get')
    def test_get_temperature_api_error(self, mock_get):
        # Arrange
        mock_response = MagicMock()
        mock_response.status_code = 500
        mock_get.return_value = mock_response

        # Act & Assert
        with self.assertRaises(Exception) as context:
            self.weather_service.get_temperature("London")
        
        self.assertTrue("API Error" in str(context.exception))
        mock_get.assert_called_once_with("https://api.weather.com/temperature/London")

if __name__ == '__main__':
    unittest.main()
```

In this example:

1. We're testing a `WeatherService` class that makes an HTTP request to an external API.

2. We use `unittest.mock.patch` to mock the `requests.get` function.

3. In the first test (`test_get_temperature_success`):
   - We set up a mock response with a 200 status code and sample temperature data
   - We verify that the temperature is returned correctly
   - We verify that the mock was called with the correct URL

4. In the second test (`test_get_temperature_api_error`):
   - We set up a mock response with a 500 status code to simulate an API error
   - We verify that an exception is raised
   - We verify that the mock was called with the correct URL

This example demonstrates:
- Mocking an external HTTP API call
- Verifying mock calls and arguments
- Testing error conditions
- Using MagicMock to simulate HTTP response objects

The tests are isolated from the real API and can be run quickly and reliably.

**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
Merhaba! Hermes mesaj kütüphanesini kullanarak pub/sub pattern'ini anlatalım:

1. Publisher Mesaj Yayınlanması:

```python
from hermes_python.hermes import Hermes
from hermes_python.ontology import Message

class Publisher:
    def __init__(self):
        self.client = Hermes("localhost:1883")  # MQTT sunucusu bağlantı noktası
        
    def publish_message(self, intent_name, message_data):
        message = Message(intent_name, message_data)
        self.client.publish(intent_name, message)
        
# Kullanım örneği
publisher = Publisher()
publisher.publish_message("weather.forecast", {"city": "İstanbul", "temperature": "25°C"})
```

2. Subscriber Mesaj Dinleme:

```python
class Subscriber:
    def __init__(self):
        self.client = Hermes("localhost:1883")
        
    def subscribe(self, intent_name):
        self.client.subscribe(intent_name, self.message_handler)
        
    def message_handler(self, message):
        print(f"Gelen mesaj: {message.data}")
        # Burada gelen mesajı işleyin
        
# Kullanım örneği
subscriber = Subscriber()
subscriber.subscribe("weather.forecast")
```

3. Pub/Sub Pattern'in Avantajları:

- **Bağımsızlık**: Publisher ve subscriber doğrudan bağlantılı değil, mesaj kuyruğunu aracılığıyla iletişim kuruyor. Bu, sistem modülleri arasındaki bağımlılığı azaltır.

- **Elastisite**: Yeni subscriber'lar eklenip çıkarılabilir, bu da sistemin genişletilmesini sağlar.

- **Performance**: Mesaj kuyruğuna yazma işlemi hızlıdır ve sistemin performansını etkilemez.

- **Kesin Bir Kez Mesajlaşma**: Her subscriber mesajı tek bir kez alır, bu da veri bütünlüğünü sağlar.

- **Asenkron İşlem**: Publisher'lar mesajı yayınladıktan sonra hemen devam edebilir, subscriber'lar mesajı alıp işleyebilir.

Örnek bir uygulama:

```python
# Weather Service (Publisher)
class WeatherService:
    def __init__(self):
        self.publisher = Publisher()
        
    def update_weather(self, city, temperature):
        self.publisher.publish_message(
            "weather.update",
            {"city": city, "temperature": temperature}
        )

# Weather Display (Subscriber)
class WeatherDisplay:
    def __init__(self):
        self.subscriber = Subscriber()
        self.subscriber.subscribe("weather.update")
        
    def message_handler(self, message):
        city = message.data["city"]
        temp = message.data["temperature"]
        print(f"{city} hava durumu: {temp}")

# Kullanım
weather_service = WeatherService()
weather_display = WeatherDisplay()

weather_service.update_weather("İstanbul", "25°C")
```

Bu örnekte WeatherService, hava durumunu güncellediğinde mesaj yayınlar ve WeatherDisplay bunu dinler ve gösterir. Her iki bileşen de doğrudan bağlantılı değil, bu da sistem modüllerinin bağımsız olarak geliştirilmesini sağlar.

def get_user_orders(username, db_path):

Please answer in English.

**A:**
Here are the problems identified in the code:

1. SQL Injection Vulnerability:
- The code directly concatenates user input (username) into the SQL query, making it vulnerable to SQL injection attacks.

2. Resource Management:
- The database connection is not properly closed after use, which could lead to resource leaks.
- No context manager is used for database operations.

3. Error Handling:
- There's no error handling for database operations, which could cause the program to crash on failure.

4. Data Validation:
- No input validation for username or db_path parameters.

5. Performance:
- Selecting all columns (*) is not recommended as it might retrieve unnecessary data.
- No index optimization mentioned.

Here's the corrected version:

```python
import sqlite3
from typing import List, Tuple

def get_user_orders(username: str, db_path: str) -> List[Tuple]:
    """
    Retrieve user orders from the database.
    
    Args:
        username: The username to query
        db_path: Path to the SQLite database file
    
    Returns:
        List of tuples containing order data
    
    Raises:
        sqlite3.Error: If database operations fail
        ValueError: If input parameters are invalid
    """
    # Input validation
    if not username or not db_path:
        raise ValueError("Username and database path cannot be empty")
    
    # Use parameterized query to prevent SQL injection
    query = "SELECT order_id, order_date, order_status FROM orders WHERE username = ?"
    
    try:
        with sqlite3.connect(db_path) as conn:
            cursor = conn.cursor()
            cursor.execute(query, (username,))

    except sqlite3.Error as e:
        raise sqlite3.Error(f"Database error: {str(e)}")
```

Improvements made:

1. SQL Injection Protection:
- Used parameterized queries with placeholders (?)
- Avoided string concatenation for user input

2. Resource Management:
- Used context manager (with statement) for database connection
- Automatically closes the connection after use

3. Error Handling:
- Added try-except block to handle database errors
- Raising specific exceptions with meaningful messages

4. Input Validation:
- Added checks for empty input parameters

5. Type Hints:
- Added type hints for better code readability and IDE support

6. Documentation:
- Added docstring with function description, parameters, return value, and exceptions

7. Specific Columns:
- Selected only necessary columns instead of using *

8. Return Type:
- Specified return type as List[Tuple] for better type checking

This version is more secure, robust, and follows better Python practices.
