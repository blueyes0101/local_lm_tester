**SQL vs. NoSQL Databases: Key Differences for Developers**  

SQL (Structured Query Language) databases, such as MySQL, PostgreSQL, and Oracle, rely on a rigid, tabular structure with predefined schemas. Data is organized into rows and columns, enforced through data types and relationships via foreign keys. This structured approach ensures consistency and supports complex queries using SQL, making them ideal for applications requiring transactional integrity, like financial systems. In contrast, NoSQL databases (e.g., MongoDB, Cassandra, Redis) prioritize flexibility with unstructured or semi-structured data models, including documents, key-value pairs, or graphs. Developers must weigh trade-offs between scalability and consistency based on their use case.

NoSQL excels in scenarios requiring high write throughput, flexible schemas, or real-time analytics, like IoT data ingestion, social media platforms, or content management systems. For example, a real-time recommendation engine might use a document-store NoSQL database for fast, schema-free data access, while a stock trading platform would rely on SQL to enforce transactional accuracy. Understanding these trade-offs helps developers choose the right tool for their application’s needs.

## SQL vs. NoSQL: Choosing the Right Data Store for Your Application

At the core of the SQL vs. NoSQL debate lies the fundamental difference in **data structure**. SQL (Relational) databases enforce a rigid, schema-defined structure using tables with predefined columns and explicit relationships (foreign keys). This ensures strong data integrity and complex query capabilities (e.g., joins, transactions adhering to ACID principles) through a declarative language. In contrast, NoSQL databases embrace a schema-less or flexible schema approach. They store data in various models: key-value pairs (e.g., Redis), document formats (e.g., JSON in MongoDB), column-family stores (e.g., Cassandra), or graph structures (e.g., Neo4j). This flexibility allows for rapid iteration and handling of unstructured or semi-structured data, but at the cost of potentially weaker consistency guarantees and more complex application-level logic for relationships.

The **scalability** characteristics are a major driver for choosing between these paradigms. SQL databases traditionally scale vertically (upgrading to more powerful servers) to handle increased load, which can become cost-prohibitive and has inherent hardware limits. While modern SQL systems (like PostgreSQL with sharding tools or cloud-native options) support horizontal scaling (adding more servers), it's often more complex and less seamless. NoSQL databases, however, were designed from the ground up for **horizontal scalability**. They natively distribute data across clusters of commodity hardware using techniques like sharding and replication. This makes them exceptionally well-suited for handling massive data volumes and extremely high throughput, common in modern web-scale applications, often with eventual consistency models to optimize for write speed over immediate ACID guarantees.

The optimal choice hinges entirely on the **use case**. Opt for SQL when your data is highly structured, relationships between entities are critical (e.g., financial transactions, inventory management), and you require strong consistency, complex queries, and ACID transactions (e.g., banking systems, ERP applications). Choose NoSQL when you need to handle vast amounts of unstructured or rapidly changing data, require high read/write throughput with low latency (e.g., user sessions, real-time analytics), or need to scale horizontally with minimal operational overhead (e.g., content management, IoT telemetry, recommendation engines). The key is understanding that "schema flexibility" and "horizontal scaling" in NoSQL come with trade-offs in consistency and query complexity, while SQL's power in data integrity and complex querying has limitations in pure scale and schema agility. Always prioritize the specific data requirements and access patterns of your application.

Here is how you can configure `iptables` on an Ubuntu server with the specified rules, along with an explanation of each line:

```bash
# Drop all incoming traffic by default
* * * * * DROP

# Accept SSH (port 22) only from 10.0.0.0/24
-A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT

# Accept HTTP (80) and HTTPS (443) from anywhere
-A INPUT -p tcp --dport 80 -j ACCEPT
-A INPUT -p tcp --dport 443 -j ACCEPT

# Allow all outgoing traffic
* * * * * ACCEPT

# Make the rules persistent
iptables-save > /etc/iptables/rules.v4
```

Explanation of each line:

1. `* * * * DROP`: This rule sets a default policy for all incoming traffic. The `*` characters represent the table, chain, source, destination, and protocol, respectively. This line drops all incoming traffic by default.
2. `-A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT`: This rule allows incoming TCP traffic on port 22 (SSH) only from the IP range 10.0.0.0/24. The `-A INPUT` part specifies that we are appending a rule to the INPUT chain, `-p tcp` specifies TCP traffic, `--dport 22` specifies port 22, `-s 10.0.0.0/24` specifies the source IP range, and `-j ACCEPT` tells iptables to accept this traffic.
3. `-A INPUT -p tcp --dport 80 -j ACCEPT`: This rule allows incoming TCP traffic on port 80 (HTTP). It is similar to the previous rule, but for HTTP traffic.
4. `-A INPUT -p tcp --dport 443 -j ACCEPT`: This rule allows incoming TCP traffic on port 443 (HTTPS). Again, it is similar to the previous rules but for HTTPS.
5. `* * * * ACCEPT`: This rule allows all outgoing traffic. The `*` characters represent the table, chain, source, destination, and protocol, respectively. This line ensures that all outgoing traffic is allowed by default.
6. `iptables-save > /etc/iptables/rules.v4`: This command saves the current iptables rules to a file named `rules.v4` in the `/etc/iptables/` directory. This makes the rules persistent, so they are retained even after a restart or system reboot.

Remember to run the `iptables-save` command with administrative privileges, as it modifies the iptables configuration. Also, be cautious when configuring firewall rules, as incorrect rules can lock you out of your server. It is always a good idea to test these rules in a controlled environment before applying them to a production server.

### Explanation of Commands

#### Part 1: Prerequisites & Flushing
*   `sudo apt-get install iptables-persistent`: Installs the service required to save rules across reboots. Without this, rules vanish after a restart.
*   `sudo iptables -F`: **Flush** (delete) all existing rules in the chains. This prevents conflicting rules from old configurations from remaining.
*   `sudo iptables -X`: **Delete** any user-defined chains that might exist.

#### Part 2: Input Chain Rules
The firewall checks packets top-to-bottom. If a packet matches a rule, it executes the action (`-j`). If it reaches the end of the chain, the **Policy** determines the action.

1.  `sudo iptables -A INPUT -i lo -j ACCEPT`
    *   `-A INPUT`: Append (add) this rule to the input chain (incoming traffic).
    *   `-i lo`: Match packets coming from the **loopback interface**. This allows local services (like database connections via localhost) to communicate without being blocked.
    *   `-j ACCEPT`: Jump to the target `ACCEPT` (allow it).

2.  `sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT`
    *   This enables a **Stateful Firewall**.
    *   It allows incoming packets that are part of an already established connection or related to one (like DNS responses or the return traffic for your outgoing requests). Without this, a web server might hang because it cannot reply to clients on port 80/443 properly once the policy drops.

3.  `sudo iptables -A INPUT -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT`
    *   `-p tcp`: Match **TCP** protocol only (SSH uses TCP).
    *   `--dport 22`: Destination port 22 (**SSH**).
    *   `-s 10.0.0.0/24`: Source IP must be in the range **10.0.0.0 to 10.0.0.255**. Any SSH attempt from outside this subnet will fail this rule and eventually be dropped.

4.  `sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT` &
5.  `sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT`
    *   These rules allow **HTTP (80)** and **HTTPS (443)** traffic from any source IP (`-s` is not used here, meaning "anyone").

#### Part 3: Default Policies
These commands control what happens to packets that do not match the previous rules.

*   `sudo iptables -P OUTPUT ACCEPT`:
    *   `-P OUTPUT`: Set the **Policy** for the output chain (outgoing traffic).
    *   `ACCEPT`: Allow all outgoing requests. This allows the server to update packages, pull data from the internet, etc., without needing specific allow rules for every domain.

*   `sudo iptables -P INPUT DROP`:
    *   `-P INPUT`: Set the **Policy** for the input chain (incoming traffic).
    *   `DROP`: If a packet comes in that isn't allowed by a rule above (like SSH from 10.x or Web traffic), this is the default action. It silently discards the traffic.

#### Part 4: Persistence
*   `sudo netfilter-persistent save`: This command writes the current active configuration to `/etc/iptables/rules.v4`. When you reboot, this service automatically reads these rules and restores the firewall state. Without this step, your changes will disappear after a restart.

#### **d) Ölçeklenebilirlik**

#### **e) Geri Zamanlama ve Güvenilirlik**
- Hermes, mesajları **persist** (sürdürülebilir) bir şekilde saklar, bu da sistemin yeniden başlatılması durumunda mesajların kaybolmasını engeller.
- Mesajlar **iade edilebilir** (retriable) veya **süresiz bekletebilir** (dead-letter queue).

#### **f) Modüler Mimari**
- Yeni işlevsellikler eklenmek istenirse, sadece **yeni subscriber'lar** eklemek yeterli olur.
- Örnek: Bir sosyal medya uygulamasında, yeni bir bildirim türü eklendiğinde, sadece o türü dinleyen bir subscriber yazmak gerekir.
