## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
SQL and NoSQL databases are two major types of database management systems that developers use to store, manage, and retrieve structured data. Understanding the differences between these two approaches is crucial for selecting the right technology for your specific project requirements.

SQL databases, like MySQL and PostgreSQL, follow a structured query language standard that enforces a fixed schema for data organization. Each table in an SQL database has a predefined schema that dictates the type of data that can be stored in each column. This rigid structure ensures data consistency and allows for complex queries through SQL. However, this comes with limitations in scalability, particularly when dealing with large volumes of unstructured or semi-structured data. As the size and complexity of datasets grow, traditional SQL databases can struggle with performance and require more resources to manage, leading to issues like slow query execution times and increased disk space usage.

In contrast, NoSQL databases, including MongoDB and Cassandra, are designed to handle large volumes of diverse data types without strict schema constraints. This flexibility makes NoSQL ideal for handling unstructured and semi-structured data, such as JSON documents in MongoDB or key-value pairs in Redis. NoSQL databases scale horizontally by adding more servers to handle increased load, making them highly scalable for big data applications. However, this scalability often comes at the cost of data consistency and complex query capabilities. NoSQL databases prioritize speed and flexibility, making them suitable for applications requiring real-time data processing and large scale, like social media feeds, IoT devices, and content management systems.

Use cases for SQL databases typically include applications that require complex queries, ACID transactions (Atomicity, Consistency, Isolation, Durability), and strong consistency, such as banking systems and enterprise resource planning (ERP). NoSQL databases, on the other hand, excel in scenarios where high throughput and scalability are critical, like e-commerce platforms, social media, and real-time analytics. For instance, a social media company might use a NoSQL database to store and retrieve user data and interactions quickly, while a financial services company might prefer an SQL database for its robust transaction processing capabilities.

In summary, SQL databases offer strong data consistency and complex query capabilities, making them well-suited for applications with strict data integrity requirements and complex business logic. NoSQL databases, with their flexibility and scalability, are ideal for big data and real-time applications that require high throughput and can tolerate eventual consistency. Choosing the right database type depends on the specific needs of your project, including data structure, scalability requirements, and query complexity.

**A:**
To configure `iptables` on an Ubuntu server according to your requirements, you will need to execute several commands. Here are the commands and explanations for each line:

```bash
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT
sudo iptables -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT
```

5. `sudo iptables -A INPUT -p tcp --dport 80 -m comment --comment "Allow HTTP from anywhere" -j ACCEPT`: Accept HTTP (TCP port 80) traffic from anywhere in the network. The comment adds clarity about the purpose of the rule.

6. `sudo iptables -A INPUT -p tcp --dport 443 -m comment --comment "Allow HTTPS from anywhere" -j ACCEPT`: Accept HTTPS (TCP port 443) traffic from anywhere in the network. The comment adds clarity about the purpose of the rule.

7. `sudo iptables -A OUTPUT -j ACCEPT`: Accept all outgoing traffic.

8. `sudo iptables -P OUTPUT ACCEPT`: Set default policy for outgoing traffic to ACCEPT

**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
In software testing, the concepts of Mock, Stub, and Fake are used to simulate certain components, behaviors, or interactions to isolate and test the functionality of a specific module or service. Here is a brief explanation of each:

1. **Mock**: A mock object is a stand-in for another object with a specific behavior. Typically, mocks are used to simulate the behavior of external dependencies (like services or APIs) or internal components that are difficult to test due to their complexity or side effects.

2. **Stub**: A stub is a simplified version of a service or component that returns predetermined values or behaviors without simulating the interaction with the external world. Stubs are often used when you want to verify that a certain method is called or return a specific result without the overhead of mocking out dependencies.

3. **Fake**: A fake is an exact replica of the real object being tested. It is often used when you need to test the exact behavior of a real object, including its state and side effects. This is rarely used because testing real objects is usually not practical or necessary.

Now, let's write a Python example using `unittest.mock` to test an external HTTP API call.

```python
import requests
import unittest
from unittest.mock import patch, Mock

class TestExternalAPIIntegration(unittest.TestCase):
    def test_api_call(self):
        # Mock the requests.get() function
        with patch('requests.get') as mock_get:
            # Configure the mock to return a specific response
            mock_get.return_value.status_code = 200
            mock_get.return_value.json.return_value = {
                'key': 'value'
            }
            
            # Call the function that makes the external API call
            response = make_api_call()

            # Verify that requests.get() was called with the correct arguments
            mock_get.assert_called_with('https://example.com/api', params={})

            # Verify the response from the API call
            self.assertEqual(response, {
                'key': 'value'
            })

            # Test behavior when the API returns an error
            mock_get.return_value.status_code = 404
            mock_get.return_value.json.return_value = {'error': 'Not Found'}

            response_with_error = make_api_call()
            self.assertEqual(response_with_error, {'error': 'Not Found'})
            self.assertEqual(mock_get.call_count, 2)

def make_api_call():
    return requests.get('https://example.com/api', params={}).json()

if __name__ == '__main__':
    unittest.main()
```

In this example:
1. We import `patch` from `unittest.mock` to mock `requests.get()`.
2. We define a test case class `TestExternalAPIIntegration`.
3. Inside the `test_api_call` method, we mock `requests.get()` and configure it to return a specific response and status code.
4. We call the `make_api_call` function, which simulates the external API call.
5. We verify that `requests.get()` was called with the correct arguments and that the returned response matches the expected response.
6. We test the behavior when the API returns an error by changing the mock's status code and response content, ensuring the function handles errors correctly and that `requests.get()` is called multiple times as expected.

This example demonstrates how to use `unittest.mock` to test external dependencies in Python applications, focusing on mocking, verifying arguments, and handling error responses.

**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
1. Publisher Mesaj Yayınlayanı:
Publisher, mesajların yayılmasında etkili bir rol oynar. Mesajları oluşturmak ve diğerleri tarafından yayınlanan bir hizmet olarak yayınlayan bir uygulama veya uygulamaların bir örneğidir. Mesajlar genellikle bir konum, bir olay veya bir dizi parametre içerebilir.

Bir örnek olarak, bir hesap yönetim sisteminde giren para miktarını yayacak bir Publisher olabilir. Bu Publisher, para miktarını dinleyen diğer uygulamaları bildirmek için mesaj gönderecektir.

Aşağıdaki kod örneğinde bir Publisher oluşturabiliriz:
```python
from hermes import Publisher

publisher = Publisher('PublisherExample') 

message = {
    'amount': 100,
    'action': 'deposit'
}

publisher.publish(message)
```

2. Subscriber Mesaj Dinleyenı:
Subscriber, gönderilen mesajları alır ve onları işler. Mesajları genellikle bir dizi callback fonksiyonu aracılığıyla dinler. Bir subscriber, mesajların içeriğini okuyabilir ve gerekirse bazı işlemler yapabilir.

Bir örnek olarak, para miktarını alacak ve hesabınızı güncelleyecek bir Subscriber olabilir. 

Aşağıdaki kod örneğinde bir Subscriber oluşturabiliriz:
```python
from hermes import Subscriber

subscriber = Subscriber('SubscriberExample')

def process_deposit(amount):
    print(f"Received deposit of {amount}")

subscriber.subscribe('deposit', process_deposit)
```
3. Publisher Subscriber Pattern'ı'nın Avantajları:

Publisher Subscriber pattern'ı, kodun modülerliği ve yeniliklilikini artırır. Her uygulama veya hizmetin, diğerlerine mesaj göndermesi ve dinlemesi yerine, doğrudan bir şekilde birbirine fonksiyon çağırma yeteneğini taşır. Bu, daha az entegrasyon ve daha az kod gereksinimine sahiptir ve daha az hata ve daha az kod bakımına neden olabilir.

Kod örneği:

```python
from hermes import Publisher, Subscriber

publisher = Publisher('PublisherExample')
subscriber = Subscriber('SubscriberExample')

def process_deposit(amount):
    print(f"Received deposit of {amount}")

publisher.subscribe('deposit', process_deposit)
publisher.publish({
    'amount': 100,
    'action': 'deposit'
})
```

Bu örnekte, Publisher bir 'deposit' etiketiyle 'amount' parametresi olan bir mesaj oluşturur ve bu mesajı 'Subscriber''e yayar. 'Subscriber', mesaj alır ve 'process_deposit' fonksiyonunu çağırır.

**Q:** Review the following Python code and identify ALL problems (security, performance, architecture). Then rewrite it correctly:

def get_user_orders(username, db_path):
    conn = sqlite3.connect(db_path)

Please answer in English.

**A:**
The provided Python code has several issues that need to be addressed for both security and performance reasons. Below are the issues and the corrected code:

### Issues:

1. **Security**:
   - **Unencrypted Database**: Using `sqlite3.connect(db_path)` assumes that the database file is stored in plain text. To protect sensitive data, the database file should be encrypted.
   - **SQL Injection**: The string `username` is directly used in the SQL query without proper sanitization, which can lead to SQL injection attacks.

2. **Performance**:
   - **Fetching All Data**: The current code fetches all the rows as a list of tuples. This could be inefficient if the database table contains many rows.

3. **Architecture**:
   - **Database Connection**: The connection object is not being properly closed after usage. This could cause memory leaks and potential database resource issues.

### Corrected Code:
```python
import sqlite3
from sqlite3 import connect

def get_user_orders(username, db_path):
    # Use a secure connection with encryption (assuming SQLite database is encrypted)
    conn = connect(db_path, uri=True)
    # Sanitize input to prevent SQL injection
    sanitized_username = username.replace("'", "''")
    query = f"SELECT * FROM orders WHERE username='{sanitized_username}'"
    try:
        with conn.cursor() as cursor:
            # Execute the query using a context manager for better resource management
            cursor.execute(query)
            # Fetch the results
            result = cursor.fetchall()
            return result
    finally:
        # Ensure the connection is closed even if an error occurs
        if conn:
            conn.close()

# Example usage
db_path = 'path_to_your_encrypted_db.db'
username = 'user123'
orders = get_user_orders(username, db_path)
print(orders)
```

### Explanation:

1. **Security**:
   - The `connect(db_path, uri=True)` initializes a secure connection. This assumes that the database file path is correctly set up with encryption in SQLite settings.
   - The `username` is sanitized to ensure safe usage in the SQL query.

2. **Performance**:
   - The `with` statement is used for the cursor object. This ensures that the resources are properly managed and the connection is closed automatically, preventing issues with resource leaks.

3. **Architecture**:
   - The `finally` block guarantees that the database connection is closed, even in the case of an error or exception during the operation.

This corrected code addresses the issues by ensuring secure database interactions, optimizing resource usage, and maintaining good architectural practices.
