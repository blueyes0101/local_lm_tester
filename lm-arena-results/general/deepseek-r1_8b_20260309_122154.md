## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
## SQL vs. NoSQL: Understanding the Core Differences

When designing modern applications, choosing the right database is critical. The fundamental choice often comes down to SQL (Structured Query Language) vs. NoSQL (Not Only SQL) databases, primarily differing in how they handle data structure, scale, and the types of applications they best suit. SQL databases, like PostgreSQL or MySQL, are relational. They store data in predefined tables with rows and columns, enforced by rigid schemas. This structure ensures strong consistency, ACID compliance (Atomicity, Consistency, Isolation, Durability), and powerful, standardized querying capabilities via SQL. In contrast, NoSQL databases, including document (e.g., MongoDB), key-value (e.g., DynamoDB), graph (e.g., Neo4j), and wide-column stores (e.g., Cassandra), offer diverse data models. They typically use flexible, schema-less or schema-light structures (like JSON-like documents), making them highly adaptable to evolving data requirements and better suited for unstructured or semi-structured data.

Scalability presents another key divergence. Traditional SQL databases often scale vertically, meaning you increase the power (CPU, RAM, storage) of a single server. While effective for predictable workloads, vertical scaling has inherent limits and can become prohibitively expensive. Horizontal scaling, adding more servers, is generally more complex in most SQL systems, often requiring intricate sharding strategies or data distribution mechanisms. NoSQL databases, however, were often architected with horizontal scalability in mind from the start. Many NoSQL solutions inherently support distributed, decentralized data models that simplify adding more servers (scaling out) to handle increased load or data volume. This makes them particularly attractive for applications experiencing rapid growth or needing extreme write/read throughput across a global user base.

The appropriate use case largely dictates the choice. SQL databases excel in applications requiring complex transactions, multi-table joins, and strong consistency, such as financial systems, e-commerce platforms with inventory management, and any application demanding robust transactional integrity. NoSQL databases shine in scenarios with rapidly changing data models, large-scale distributed systems, high write throughput, and flexible querying needs. Common use cases include content management systems (CMS), real-time analytics, IoT data ingestion, mobile applications with user-generated content, and recommendation engines, where flexibility and horizontal scalability often take precedence over rigid schemas and complex joins. Therefore, selecting between SQL and NoSQL involves evaluating the trade-offs between structure, consistency, scalability, and the specific application requirements.

**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here's the step-by-step configuration of `iptables` on an Ubuntu server, with explanations for each rule:

---

### **1. Flush Existing Rules**
```bash
sudo iptables -F
sudo iptables -t nat -F
sudo iptables -t mangle -F
sudo iptables -t filter -F
```
- **Explanation**: Clears all existing rules from the `filter`, `nat`, and `mangle` tables to start fresh.

---

### **2. Set Default Policies**
```bash
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT
```
- **Explanation**:
  - `-P INPUT DROP`: Drops all incoming traffic by default (unless explicitly allowed).
  - `-P FORWARD DROP`: Drops all forwarded traffic (not needed for basic server setups).
  - `-P OUTPUT ACCEPT`: Allows all outgoing traffic (e.g., outbound connections).

---

### **3. Allow Established/Related Connections**
```bash
sudo iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT
```
- **Explanation**: Accepts incoming traffic that is part of an **existing connection** (e.g., responses to outgoing requests). This ensures bidirectional communication works.

---

### **4. Allow SSH from 10.0.0.0/24**
```bash
sudo iptables -A INPUT -s 10.0.0.0/24 -p tcp --dport 22 -j ACCEPT
```
- **Explanation**:
  - `-s 10.0.0.0/24`: Restricts SSH access to clients in the `10.0.0.0/24` subnet.
  - `-p tcp --dport 22`: Targets TCP traffic on port 22 (SSH).
  - `-j ACCEPT`: Allows this traffic.

---

### **5. Allow HTTP (80) and HTTPS (443) from Anywhere**
```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```
- **Explanation**: Opens ports 80 (HTTP) and 443 (HTTPS) for incoming connections from **any IP address**.

---

### **6. Make Rules Persistent**
1. **Install `iptables-persistent`**:
   ```bash
   sudo apt-get update
   sudo apt-get install iptables-persistent
   ```
   - This package saves rules at boot time.

2. **Save Current Rules**:
   ```bash
   sudo netfilter-persistent save
   # OR
   sudo /etc/init.d/netfilter-persistent save
   ```
   - Rules are saved to `/etc/iptables/rules.v4` and reloaded on reboot.

---

### **Final Rules Explained**
- **Default Drop**: All traffic is blocked unless explicitly allowed.
- **Established Connections**: Ensures responses to outgoing requests are accepted.
- **SSH Access**: Limited to `10.0.0.0/24` for security.
- **Web Traffic**: Opens HTTP/HTTPS globally (e.g., for public websites).
- **Outgoing Traffic**: Fully unrestricted (e.g., for accessing external services).

---

### **Verification**
```bash
sudo iptables -L -v
```
- Check the active rules and ensure no unintended traffic is allowed.

---

### **Notes**
- **ICMP (Ping)**: Not allowed by default. To enable ping, add:
  ```bash
  sudo iptables -A INPUT -p icmp -j ACCEPT
  ```
- **IPv6**: Requires separate `ip6tables` configuration. Use `ip6tables-save`/`ip6tables-restore` for persistence.

**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
### Differences Between Mock, Stub, and Fake in Software Testing

| **Concept** | **Purpose** | **Key Characteristics** | **Example Use Case** |
|------------|------------|-------------------------|----------------------|
| **Stub** | Simulate a dependency with predefined responses | Simplistic, non-functional, no behavior beyond fixed output | Mocking an API that returns static data for a test |
| **Fake** | Replace a real component with a simplified implementation | Functional, mimics real behavior but simplified | In-memory database for testing instead of a real database |
| **Mock** | Verify interactions between components | Records interactions, enforces expectations | Testing that a function calls an API with specific parameters |

### Python Example Using `unittest.mock`

```python
import unittest
from unittest.mock import patch
import requests

# Function to test that makes an HTTP call
def fetch_data(url):
    response = requests.get(url)
    return response.json()

class TestHTTPCall(unittest.TestCase):

    @patch('requests.get')
    def test_api_error_response(self, mock_get):
        # Set up mock to return a 500 error
        mock_response = unittest.mock.Mock()
        mock_response.status_code = 500
        mock_response.raise_for_status.side_effect = requests.HTTPError
        mock_get.return_value = mock_response

        # Verify the function raises an exception when API returns 500
        with self.assertRaises(requests.HTTPError):
            fetch_data('https://api.example.com/data')

        # Verify the mock was called with the correct arguments
        mock_get.assert_called_once_with('https://api.example.com/data')

if __name__ == '__main__':
    unittest.main()
```

### Explanation of the Example

1. **Mocking the External API**:
   - `@patch('requests.get')` replaces the real `requests.get` call with a mock object.
   - `mock_response` simulates an HTTP response with a `500` status code.

2. **Verifying Correct Arguments**:
   - `mock_get.assert_called_once_with('https://api.example.com/data')` ensures the URL was called correctly.

3. **Testing Error Handling**:
   - `mock_response.raise_for_status.side_effect` makes the mock raise an `HTTPError` when called.
   - The test checks that `fetch_data` propagates this exception.

This example demonstrates how mocks are used to isolate code from external dependencies while verifying interactions and error handling.

**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
Hermes mesaj kütüphanesi ile Publisher/Subscriber pattern'ini kullanarak mesajlaşma yaparken, bileşenler arasında bağımsız çalışmayı ve modüler bir yapıyı sağlırsınız. İşte detaylı açıklama ve örnek:

### 1. **Publisher Nasıl Mesaj Yayınlar?**
Publisher, belirli bir konuya (topic) mesaj yayınlar. Hermes'te bu genellikle bir `publish` veya `emit` fonksiyonu ile yapılır. Örneğin:
```python
from hermes import HermesPublisher

publisher = HermesPublisher()
publisher.publish("news", "Dünya haberi: ...")
```
- **Özellikleri**:  
  - Mesaj, belirtilen konuya (topic) göre yayınlanır.  
  - Yayın işlemi asenkron olabilir (özellikle performans için).  
  - Publisher, yayın sonucunu beklemekten veya alıcıları bilinçlendirmekten sorumlu değildir.

---

### 2. **Subscriber Nasıl Dinler?**
Subscriber, belirli konularda mesajları dinler ve alıcı olarak davranır. Hermes'te bu genellikle bir `subscribe` veya `listen` fonksiyonu ile yapılır:
```python
from hermes import HermesSubscriber

subscriber = HermesSubscriber()
subscriber.subscribe("news", callback_function)  # Mesaj geldiğinde callback çağrılır
```
- **Özellikleri**:  
  - Abone olduğu konularda mesajlar geldiğinde `callback` fonksiyonu otomatik olarak çalıştırılır.  
  - Dinleme işlemi asenkron çalışabilir.  
  - Subscriber, mesajı aldıktan sonra doğrudan Publisher'a geri bildirim yapmaz.

---

### 3. **Bu Pattern'in Fonksiyon Çağrısına Göre Avantajı**
- **Asenkron Çalışma**:  
  Publisher mesaj yayınlarken, Subscriber mesajı dinler. Bu, fonksiyon çağrıları gibi bloke edici olmaktan çıkar. Örneğin:
  ```python
  # Fonksiyon çağrısı yerine asenkron mesajlaşma:
  publisher.publish("order", "Sipariş #123")  # Anlık geri bildirim beklenmez.
  ```
- **Modülerlik ve Bağımsızlık**:  
  Publisher ve Subscriber birbirini doğrudan bilmez veya referans edemez. Hermes sağlar ki:
  - Publisher sadece mesaj yayınlar.  
  - Subscriber sadece belirli konularda dinler.  
  Bu, bileşenlerin değiştirilebilirliğini artırır.
- **Paralel İşlem Gücü**:  
  Çok sayıda Subscriber aynı mesaja abone olabilir. Her mesaj, tüm Subscriber'lar asenkron olarak işler.

---

### Örnek Uygulama
```python
from hermes import Hermes

# Hermes örneği oluşturulur
hermes = Hermes()

# Publisher mesaj yayınlar
def publish_news():
    hermes.publish("news", "Yeni makale yayınlandı!")

# Subscriber dinler
def listen_news(news):
    print(f"Alınan haber: {news}")

# Abone oluruz
hermes.subscribe("news", listen_news)

# Mesaj yayınlanır ve Subscriber asenkron olarak işlemiş olur
publish_news()  # Fonksiyon biter, mesajın işlendiği fonksiyon arka planda çalışır
```

Bu pattern, mikroservisler, event-driven arquitektürler veya büyük ölçekli uygulamalarda mesajlaşmayı otomatikleştirir. Fonksiyon çağrıları yerine asenkron mesajlaşma, performans ve skalayablilikte büyük avantajlar sunar.

**Q:** Review the following Python code and identify ALL problems (security, performance, architecture). Then rewrite it correctly:

import sqlite3

def get_user_orders(username, db_path):
    conn = sqlite3.connect(db_path)
    query = f"SELECT * FROM orders WHERE username='{username}'"
    result = conn.execute(query)
    return result.fetchall()

Please answer in English.

**A:**
Here is the analysis of the code and the corrected version:

**Security Issues:**
1. **SQL Injection Vulnerability**: Using string formatting with user input (`username`) allows attackers to inject malicious SQL code, leading to unauthorized data access or deletion.
2. **Missing Input Validation**: No checks ensure `username` or `db_path` are valid, allowing attacks like path traversal or invalid inputs.

**Performance Issues:**
1. **Resource Leak**: The database connection is not closed, leading to potential exhaustion of database connections over repeated calls.

**Architectural Issues:**
1. **Tight Coupling**: Directly connecting to the SQLite database violates the separation of concerns, making the code less modular and harder to maintain.
2. **Error Handling**: Lacks error handling for database connection failures or query execution errors.

**Corrected Code:**

```python
import sqlite3
from typing import List, Dict, Optional

class Database:
    def __init__(self, db_path: str):
        self.db_path = db_path

    def __enter__(self):
        self.conn = sqlite3.connect(self.db_path)
        return self

    def __exit__(self, exc_type, exc_val, exc_tb):
        self.conn.close()

    def get_user_orders(self, username: str) -> Optional[List[Dict]]:
        query = """SELECT * FROM orders 
                  WHERE username = :username"""
        try:
            cursor = self.conn.execute(query, {'username': username})
            columns = [col[0] for col in cursor.description]
            return [dict(zip(columns, row)) for row in cursor.fetchall()]
        except sqlite3.Error as e:
            print(f"Database error: {e}")
            return None

def main():
    db_path = "orders.db"  # Example path, should be configurable
    with Database(db_path) as db:
        orders = db.get_user_orders("test_user")
        if orders is not None:
            print(orders)

if __name__ == "__main__":
    main()
```

**Explanation of Changes:**

1. **SQL Injection Fix**: Uses parameter substitution (`:username`) with the `sqlite3` parameterized query to prevent injection attacks.
2. **Database Connection Management**: Implements a context manager (`Database` class) to ensure the connection is properly closed using `__exit__`.
3. **Error Handling**: Wraps the query execution in a try-except block to handle potential database errors gracefully.
4. **Modular Architecture**: Separates the database logic into a dedicated class, making the code more modular and testable. The `get_user_orders` method now returns a list of dictionaries for better readability.
5. **Input Handling**: The `username` is passed directly to the parameterized query without validation, but in a real-world scenario, additional validation might be needed.
6. **Resource Management**: The context manager ensures the connection is closed even if an error occurs, preventing resource leaks.
