## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
 ### Understanding the Key Differences Between SQL and NoSQL Databases

In the world of database management, the choice between SQL (Structured Query Language) and NoSQL (Not Only SQL) databases can significantly impact the performance, flexibility, and scalability of applications. Each type of database offers unique advantages and is better suited to different types of data management tasks. Understanding these differences is crucial for developers who need to select the appropriate database for their projects.

#### Data Structure and Schema Flexibility

SQL databases are based on a relational model, where data is structured in tables with predefined schemas. This means that the structure of the database is strictly defined at the time of creation, and tables must adhere to these rules. This rigidity is beneficial for applications where the data model is well-defined and predictable. SQL databases are excellent for handling complex queries, transactions, and business logic that requires structured data.

On the other hand, NoSQL databases are designed for greater flexibility in data structure and schema. They support various data models, including key-value, document, columnar, and graph-based structures. This flexibility allows NoSQL databases to adapt to the dynamic nature of applications that require the ability to scale horizontally across multiple servers. They are ideal for applications with rapidly changing data requirements, large volumes of unstructured data, and real-time data processing.

#### Scalability and Performance

Scalability is another critical factor in database selection. SQL databases, particularly those that support ACID (Atomicity, Consistency, Isolation, Durability) properties, are known for their ability to handle high transaction volumes and ensure data integrity. This is crucial in mission-critical applications where data consistency and reliability are paramount. However, traditional SQL databases often struggle with horizontal scalability, as they are designed to scale vertically by improving the performance of a single database server.

NoSQL databases, by contrast, are designed for horizontal scalability. They can distribute data across multiple servers, which is particularly beneficial for applications with high traffic and massive amounts of data. This distributed architecture allows NoSQL databases to handle large-scale, distributed data workloads effectively. For instance, a NoSQL database might be the preferred choice for a social media application where the data volume and write frequency are extremely high.

#### Use Cases

The choice between SQL and NoSQL databases also depends on the specific use case. SQL databases are typically the default choice for enterprise applications, where complex business logic and structured data are essential. Banks, large corporations, and organizations with stringent regulatory requirements often use SQL databases to manage their data due to the inherent data integrity features and support for complex querying.

NoSQL databases are preferred in scenarios where data needs to be processed in real-time and where the schema is not static. Start-ups, in particular, favor NoSQL databases for their ease of development and deployment in dynamic environments. The ability to quickly adapt to changing data requirements without significant upfront schema design makes NoSQL databases a popular choice for rapid prototyping and MVP (Minimum Viable Product) development.

In conclusion, while SQL databases excel in scenarios requiring complex transactions and structured data, NoSQL databases shine in environments demanding flexibility, scalability, and real-time processing capabilities. Developers must carefully consider these factors to select the most appropriate database technology for their projects, ensuring that the chosen database aligns with their application's data management needs.

**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
 To configure iptables on your Ubuntu server to match the rules you provided, you can use the following commands:

1. Open Terminal and type:

```bash
sudo nano /etc/iptables/rules.v4
```

2. After opening the file, add the following lines:

```bash
*filter
:INPUT DROP [0:0]
:FORWARD ACCEPT [0:0]
:OUTPUT ACCEPT [0:0]
-A INPUT -i lo -j ACCEPT
-A INPUT -d 127.0.0.0/8 -p icmp --icmp-type echo-request -j DROP
-A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT
-A INPUT -p tcp ! --syn -m state --state NEW -m tcp -j DROP
-A INPUT -i eth0 -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT
-A INPUT -i eth0 -p tcp --dport 80 -j ACCEPT
-A INPUT -i eth0 -p tcp --dport 443 -j ACCEPT
-A INPUT -j DROP
-A FORWARD -j DROP
-A OUTPUT -j ACCEPT
COMMIT
```

3. Save and close the file.

4. Apply the new rules:

```bash
sudo iptables-restore < /etc/iptables/rules.v4
```

Explanation of each line:

- *filter: This is a chain where the rules will be applied.
- :INPUT DROP [0:0]: This rule sets the INPUT chain to DROP all incoming traffic.
- :FORWARD ACCEPT [0:0]: This rule sets the FORWARD chain to ACCEPT all incoming traffic (since the packet is already inside the network, no need to drop).
- :OUTPUT ACCEPT [0:0]: This rule sets the OUTPUT chain to ACCEPT all outgoing traffic.
- -A INPUT -i lo -j ACCEPT: This rule allows traffic from the loopback interface (lo).
- -A INPUT -d 127.0.0.0/8 -p icmp --icmp-type echo-request -j DROP: This rule drops ICMP echo-request packets coming from the 127.0.0.0/8 subnet.
- -A INPUT -m state --state RELATED,ESTABLISHED -j ACCEPT: This rule allows traffic where an existing connection is being reused.
- -A INPUT -p tcp ! --syn -m state --state NEW -m tcp -j DROP: This rule drops new incoming connections that are not TCP SYN packets.
- -A INPUT -i eth0 -p tcp --dport 22 -s 10.0.0.0/24 -j ACCEPT: This rule allows incoming SSH traffic from the specified subnet (10.0.0.0/24).
- -A INPUT -i eth0 -p tcp --dport 80 -j ACCEPT: This rule allows incoming HTTP traffic (port 80).
- -A INPUT -i eth0 -p tcp --dport 443 -j ACCEPT: This rule allows incoming HTTPS traffic (port 443).
- -A INPUT -j DROP: This rule drops all other incoming traffic.
- -A FORWARD -j DROP: This rule sets the FORWARD chain to DROP all incoming traffic.
- -A OUTPUT -j ACCEPT: This rule sets the OUTPUT chain to ACCEPT all outgoing traffic.
- COMMIT: This command commits the current ruleset to memory.

5. Finally, to make the rules persistent, you can add them to the default iptables rules. To do this, create a new file in the /etc/iptables directory with the following command:

```bash
sudo iptables-save > /etc/iptables/rules.v4
```

This command generates the same iptables rules as the previous example and saves them in the /etc/iptables/rules.v4 file. This will persist the rules even after a system reboot.

Now your iptables rules are set as per your requirements, and they will persist across reboots.

**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
 In software testing, "mocks", "stubs", and "fakes" are all terms used to describe artificial tools that are used to simulate the behavior of complex components or systems.

- **Mock**: A mock is an object that replaces the real object during testing. It's used to simulate how a system would interact with another system. A mock doesn't have to do real work; it just has to act as the real object would in a given scenario.

- **Stub**: A stub is a simple object that simulates the behavior of the real object. It typically has a predefined response for specific calls, and can be used to test how code responds to known inputs.

- **Fake**: A fake is an object or a program that pretends to be another one. It can be used to test how code handles inputs and outputs, but it doesn't need to simulate the real object's behavior.

Now, here's an example using Python's unittest.mock module to mock an external HTTP API call:

```python
import unittest
from unittest.mock import Mock, patch, AsyncMock
from example_module import external_api_call  # replace with actual module and function

class TestExternalAPICall(unittest.TestCase):

    @patch('example_module.requests')
    def test_external_api_call(self, mock_requests):
        # Mock the response from the API
        mock_response = Mock()
        mock_response.status_code = 200
        response_data = {'key': 'value'}
        mock_response.json.return_value = response_data

        # Set up the mock
        mock_requests.get.return_value = mock_response

        # Call the function under test
        result = external_api_call('http://example.com')

        # Verify the mock was called with the correct arguments
        mock_requests.get.assert_called_once_with('http://example.com')

        # Verify the result
        self.assertEqual(result, response_data)

    @patch('example_module.requests')
    def test_external_api_call_with_error(self, mock_requests):
        # Mock the response from the API with an error
        mock_response = Mock()
        mock_response.status_code = 500
        response_data = {'error': 'internal server error'}
        mock_response.json.return_value = response_data

        # Set up the mock
        mock_requests.get.return_value = mock_response

        # Call the function under test
        with self.assertRaises(RuntimeError) as context:
            external_api_call('http://example.com')

        # Verify the mock was called with the correct arguments
        mock_requests.get.assert_called_once_with('http://example.com')

        # Verify the error was raised as expected
        self.assertTrue(str(context.exception), 'error message from the API error')

if __name__ == '__main__':
    unittest.main()
```

In this example, the functions `external_api_call` are being tested. The test cases use `unittest.mock` to create mocks for the `requests.get` method. The first test verifies that the mock is called with the correct arguments and that the API returns a successful response. The second test creates a mock response with an error status code and verifies that the function raises a `RuntimeError` with the expected error message.
