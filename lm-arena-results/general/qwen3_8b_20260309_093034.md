## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
SQL and NoSQL databases represent two distinct paradigms for data storage and management, each tailored to specific use cases and architectural needs. SQL databases, such as MySQL and PostgreSQL, are relational and rely on structured query language to manage data organized in tables with predefined schemas. This rigid structure ensures data consistency and supports complex queries through joins and transactions, making them ideal for applications requiring strong data integrity, like financial systems or enterprise resource planning (ERP). In contrast, NoSQL databases—such as MongoDB, Cassandra, and Redis—are non-relational and schema-less, allowing for flexible, semi-structured data formats like JSON, key-value pairs, or wide-column stores. This adaptability enables rapid development and scalability for applications with evolving data models, such as content management systems or real-time analytics platforms.

Scalability differs significantly between SQL and NoSQL databases, influenced by their architectural design. SQL databases traditionally scale vertically by upgrading hardware, which can become cost-prohibitive for large-scale applications. While some modern SQL systems support horizontal scaling through sharding or replication, this often requires complex configurations. NoSQL databases, however, are inherently designed for horizontal scalability, distributing data across multiple nodes to handle massive datasets and high traffic loads. This distributed architecture makes NoSQL particularly effective for cloud-native applications, big data processing, and scenarios where read/write throughput is critical, such as social media platforms or IoT data pipelines. The choice between vertical and horizontal scaling often hinges on the specific performance and cost requirements of the application.

Use cases further distinguish SQL and NoSQL databases based on their strengths. SQL databases excel in environments requiring ACID compliance, complex transactions, and precise data relationships, such as banking systems, inventory management, or customer relationship management (CRM) platforms. NoSQL databases, with their flexibility and scalability, are better suited for applications dealing with unstructured or semi-structured data, such as recommendation engines, log processing, or mobile app backends. For instance, a real-time analytics dashboard might leverage NoSQL for rapid data ingestion, while a legacy enterprise application with strict data consistency needs would rely on SQL. Developers must evaluate these trade-offs to select the database that aligns with their application’s performance, scalability, and data model requirements.

**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
Here's how to configure `iptables` on an Ubuntu server according to your rules, with explanations for each line:

---

### **1. Set default policies**
```bash
sudo iptables -P INPUT DROP
sudo iptables -P OUTPUT ACCEPT
```

**Explanation:**
- `iptables -P INPUT DROP`:  
  Sets the default policy for the **INPUT** chain to **DROP**. This means all incoming traffic is blocked unless explicitly allowed by a rule.
- `iptables -P OUTPUT ACCEPT`:  
  Sets the default policy for the **OUTPUT** chain to **ACCEPT**. This allows all outgoing traffic by default, as per your requirement.

---

### **2. Allow SSH (port 22) from 10.0.0.0/24**
```bash
sudo iptables -A INPUT -s 10.0.0.0/24 -p tcp --dport 22 -j ACCEPT
```

**Explanation:**
- `-A INPUT`: Appends the rule to the **INPUT** chain.
- `-s 10.0.0.0/24`: Specifies the source IP range (10.0.0.0 to 10.0.0.255).
- `-p tcp`: Matches TCP traffic.
- `--dport 22`: Targets port 22 (SSH).
- `-j ACCEPT`: Accepts the traffic.

This allows SSH access only from devices in the `10.0.0.0/24` subnet.

---

### **3. Allow HTTP (port 80) and HTTPS (port 443) from anywhere**
```bash
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT
```

**Explanation:**
- `-p tcp`: Matches TCP traffic.
- `--dport 80`: Targets port 80 (HTTP).
- `-j ACCEPT`: Accepts the traffic.
- Repeat for port 443 (HTTPS).

These rules allow web traffic (HTTP/HTTPS) from **any** source IP.

---

### **4. Make the rules persistent**
```bash
sudo apt install iptables-persistent -y
sudo netfilter-persistent save
```

**Explanation:**
- `apt install iptables-persistent`: Installs the package to save `iptables` rules to a file (`/etc/iptables/rules.v4`).
- `netfilter-persistent save`: Saves the current rules to persist across reboots. This ensures your configuration remains active after a system restart.

---

### **Summary of Rules**
| Chain     | Rule                                                                 |
|-----------|----------------------------------------------------------------------|
| **INPUT** | Drop all traffic by default.                                       |
| **INPUT** | Allow SSH (port 22) from 10.0.0.0/24.                             |
| **INPUT** | Allow HTTP (port 80) from any IP.                                  |
| **INPUT** | Allow HTTPS (port 443) from any IP.                                |
| **OUTPUT**| Allow all outgoing traffic by default.                             |

---

### **Important Notes**
1. **Testing**: After applying these rules, test connectivity to ensure:
   - SSH works from the `10.0.0.0/24` subnet.
   - HTTP/HTTPS works from external IPs.
   - Outgoing traffic is unrestricted.
2. **Security**: This configuration blocks all incoming traffic except for explicitly allowed services. Ensure your firewall rules align with your security requirements.
3. **Persistence**: The `iptables-persistent` package ensures rules survive reboots. If you remove it, the rules will be lost.

**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
### Difference Between Mock, Stub, and Fake in Software Testing

1. **Mock**  
   A mock is a test double that simulates the behavior of a real object. It is used to **verify interactions** (e.g., checking if a method was called with specific arguments). Mocks often include **expectations** and **assertions** to ensure the system under test behaves correctly. For example, a mock might verify that a method was called with the right parameters.

2. **Stub**  
   A stub is a test double that provides **predefined responses** to method calls. It is used to **supply known data** for testing, without simulating real behavior. Stubs are typically **stateless** and focus on **input validation** rather than interaction verification. For example, a stub might return a fixed value for a method call.

3. **Fake**  
   A fake is a **working implementation** that mimics the behavior of a real object but is not the actual implementation. Fakes are often used when you need a **real-like object** but with simplified or mock data. For example, a fake database might store data in memory instead of connecting to a real database.

---

### Python Example Using `unittest.mock`

This example mocks an external HTTP API call, verifies the arguments, and tests the behavior when the API returns an error.

```python
import unittest
from unittest import mock
import requests

def fetch_data(url):
    try:
        response = requests.get(url)
        response.raise_for_status()
        return response.json()
    except requests.exceptions.RequestException as e:
        return {"error": str(e)}

class TestAPICall(unittest.TestCase):
    def test_api_call(self):
        # Mock the requests.get call
        with mock.patch('requests.get') as mock_get:
            # Set up the mock to return a sample response
            mock_response = mock.MagicMock()
            mock_response.status_code = 200
            mock_response.json.return_value = {'data': 'test'}
            mock_get.return_value = mock_response

            # Call the function under test
            result = fetch_data('https://api.example.com/data')

            # Verify the mock was called with the correct URL
            mock_get.assert_called_once_with('https://api.example.com/data')

            # Check the result
            self.assertEqual(result, {'data': 'test'})

    def test_api_error(self):
        # Mock the requests.get call
        with mock.patch('requests.get') as mock_get:
            # Set up the mock to return a 404 response
            mock_response = mock.MagicMock()
            mock_response.status_code = 404
            mock_get.return_value = mock_response

            # Call the function under test
            result = fetch_data('https://api.example.com/data')

            # Verify the mock was called with the correct URL
            mock_get.assert_called_once_with('https://api.example.com/data')

            # Check the error response
            self.assertIn("404 Client Error", result["error"])

if __name__ == '__main__':
    unittest.main()
```

---

### Key Points in the Example

1. **Mocking the API Call**  
   The `requests.get` method is mocked using `unittest.mock.patch`. This replaces the real HTTP call with a mock object.

2. **Verifying Arguments**  
   The `mock_get.assert_called_once_with(...)` line checks that the mock was called with the correct URL.

3. **Testing the Error Case**  
   The mock returns a 404 response, which triggers the `response.raise_for_status()` exception. The `fetch_data` function catches this exception and returns an error message. The test verifies that the error message includes the expected text.

This example demonstrates how to use mocks to simulate external dependencies, verify interactions, and test error handling in a controlled environment.

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
### 1. **Publisher nasıl mesaj yayınlar?**  
Hermes kütüphanesiyle publisher (yayımcı), bir mesajı bir **topic** (konu) üzerinden yayınlar. Bu işlem için aşağıdaki adımlar takip edilir:  
- **Producer instance** oluşturulur.  
- Mesaj, `publish` fonksiyonu ile belirli bir topic'e gönderilir.  
- Hermes, mesajı arka planda dağıtır.  

**Kod Örneği:**
```python
from hermes import Hermes

# Hermes instance'ı başlat
hermes = Hermes()

# Publisher (yayımcı) mesaj gönderir
topic = "user_events"
message = "Kullanıcı kayıt oldu: 12345"
hermes.publish(topic, message)
```

---

### 2. **Subscriber nasıl dinler?**  
Subscriber (alıcı), belirli bir topic'ten mesajları dinler. Bu işlem için aşağıdaki adımlar takip edilir:  
- **Consumer instance** oluşturulur.  
- `subscribe` fonksiyonu ile bir topic'e abone olunur.  
- Hermes, mesajları otomatik olarak tüketiciye iletir.  

**Kod Örneği:**
```python
from hermes import Hermes

# Hermes instance'ı başlat
hermes = Hermes()

# Subscriber (alıcı) mesaj dinler
def on_message(topic, message):
    print(f"Alındı: {topic} - {message}")

hermes.subscribe("user_events", on_message)
```

---

### 3. **Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?**  
Publisher/Subscriber pattern, doğrudan fonksiyon çağrısına göre **asenkron, dağıtık ve esnek** bir iletişim sağlar. Avantajları şunlardır:  
- **Bağımsız çalışma:** Publisher ve subscriber birbirinden bağımsız çalışabilir.  
- **Ölçeklenebilirlik:** Bir topic'e birçok subscriber bağlanabilir.  
- **Kesintisizlik:** Bir subscriber çöktüğünde, mesajlar başka bir subscriber tarafından işlenebilir.  
- **Distribüte yapı:** Mesajlar farklı sunucular arasında dağıtılmak üzere tasarlanır.  

**Örnek:**  
Bir kullanıcı kayıt olduğunda, publisher "user_events" topic'ine mesaj gönderir. Bu mesaj, bir e-posta servisi (subscriber) ve bir analitik servisi (subscriber) tarafından farklı zamanlarda işlenebilir. Bu, doğrudan fonksiyon çağrısından farklıdır çünkü her servis kendi işlevini bağımsız olarak yerine getirir.
