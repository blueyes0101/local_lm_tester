## Question 3
**Q:** Write a short technical article (3 paragraphs) in English about the differences between SQL and NoSQL databases. The article should be suitable for a developer audience and cover: data structure, scalability, and use cases.

**A:**
## SQL vs. NoSQL: Architecting Data for Modern Applications

The choice between SQL and NoSQL databases fundamentally hinges on how data is structured and how systems need to scale. SQL databases (relational) enforce a strict, predefined schema where data resides in structured tables with fixed columns and relationships enforced via foreign keys. This structure guarantees strong consistency and ACID (Atomicity, Consistency, Isolation, Durability) compliance, making them ideal for transactional applications where data integrity is paramount, like banking systems or order processing. Conversely, NoSQL databases (document, key-value, columnar, graph) prioritize flexibility and scalability over rigid schema enforcement. Data is often stored as semi-structured documents (JSON/BSON), key-value pairs, or unstructured columns. While this allows for rapid iteration and handling diverse data types (e.g., user profiles with varying attributes), it typically sacrifices ACID guarantees for eventual consistency, where data updates propagate across nodes asynchronously. This flexibility is crucial when dealing with unstructured or rapidly evolving data models.

Scalability presents a stark contrast between the two paradigms. SQL databases traditionally scale vertically—adding more powerful hardware to a single server—which can be costly and limited by hardware ceilings. While techniques like read replicas and sharding exist, scaling writes efficiently often requires complex, application-layer logic or specialized distributed SQL solutions. NoSQL databases excel at horizontal scalability, distributing data across many commodity servers ("sharding" or partitioning). This allows seamless scaling out to handle massive data volumes and high write throughput by adding more nodes to the cluster. However, this horizontal scale often comes with trade-offs: managing data distribution (e.g., partition keys), dealing with eventual consistency, and navigating the CAP theorem (prioritizing availability and partition tolerance over strong consistency). NoSQL excels when write-heavy, high-traffic applications need to scale globally with minimal latency, while SQL systems are better suited for scenarios where data integrity and strong consistency are non-negotiable.

The most appropriate choice depends entirely on the specific application requirements. **Use SQL** when you need robust transactional integrity, complex relational queries, and well-defined schemas – think e-commerce order processing, financial systems, or any application requiring strict data accuracy and referential integrity. **Embrace NoSQL** when your data is unstructured, frequently changing, or massive in volume, and when you prioritize high availability, horizontal scalability, and flexible querying – such as real-time analytics, content management (blogs, social feeds), IoT sensor data, or session stores requiring rapid write operations. Crucially, many modern applications leverage both: using SQL for core transactional data and NoSQL for handling high-volume, rapidly changing auxiliary data or real-time analytics needs, demonstrating that the optimal solution often lies in strategic architectural choices rather than choosing one model exclusively. Understanding the trade-offs between structure, consistency, and scale is key to selecting the right tool for the job.

## Question 5
**Q:** Configure iptables on an Ubuntu server according to these rules and explain each line:

- Drop all incoming traffic by default
- Accept SSH (port 22) only from 10.0.0.0/24
- Accept HTTP (80) and HTTPS (443) from anywhere
- Allow all outgoing traffic
- Make the rules persistent

Please answer in English.

**A:**
### Configuration of iptables to Meet Requirements
Below is the step-by-step configuration for iptables, followed by explanations for each command. This setup:
- Drops **all incoming traffic by default** (via `INPUT` chain policy).
- **Only allows** SSH (port 22) from `10.0.0.0/24`.
- **Allows** HTTP (port 80) and HTTPS (port 443) from **anywhere**.
- **Permits all outgoing traffic** (via `OUTPUT` chain policy).
- **Preserves rules after reboot** via `iptables-persistent`.

---

#### Step-by-Step Configuration Commands
```bash
# 1. Flush all existing rules
sudo iptables -F

# 2. Flush auxiliary tables (NAT, MANGLE, RAW)
sudo iptables -t nat -F
sudo iptables -t mangle -F
sudo iptables -t raw -F

# 3. Set default policies
sudo iptables -P INPUT DROP
sudo iptables -P FORWARD DROP
sudo iptables -P OUTPUT ACCEPT

# 4. Allow established/related connections (e.g., responses to existing sessions)
sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT

# 5. Allow SSH (port 22) ONLY from 10.0.0.0/24
sudo iptables -A INPUT -s 10.0.0.0/24 -p tcp --dport 22 -j ACCEPT

# 6. Allow HTTP (port 80) from anywhere
sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT

# 7. Allow HTTPS (port 443) from anywhere
sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT

# 8. Save rules for persistence after reboot
sudo iptables-save > /etc/iptables/rules.v4
```

---

#### Detailed Explanation of Each Command
1. **`sudo iptables -F`**  
   - **Purpose**: Flushes **all rules** in the default `filter` table.  
   - **Why**: Ensures no lingering rules interfere with our new configuration.  
   - **Note**: Safe to run as it starts from a clean state.

2. **`sudo iptables -t nat -F`**  
   - **Purpose**: Flushes rules in the `nat` table (used for NAT/port forwarding).  
   - **Why**: We’re not configuring NAT, so this prevents accidental conflicts.

3. **`sudo iptables -t mangle -F`**  
   - **Purpose**: Flushes rules in the `mangle` table (used for packet modification like TOS).  
   - **Why**: No modification rules are needed here.

4. **`sudo iptables -t raw -F`**  
   - **Purpose**: Flushes rules in the `raw` table (used for bypassing connection tracking).  
   - **Why**: Standard practice to reset raw chain state.

5. **`sudo iptables -P INPUT DROP`**  
   - **Purpose**: Sets the **default policy** for the `INPUT` chain to `DROP`.  
   - **Why**: Meets the requirement to **drop all incoming traffic by default**.  
   - **Key**: Only explicitly allowed traffic will be accepted.

6. **`sudo iptables -P FORWARD DROP`**  
   - **Purpose**: Sets the **default policy** for `FORWARD` to `DROP`.  
   - **Why**: Prevents unauthorized forwarding (e.g., if the server acts as a router).

7. **`sudo iptables -P OUTPUT ACCEPT`**  
   - **Purpose**: Sets the **default policy** for `OUTPUT` to `ACCEPT`.  
   - **Why**: Allows **all outgoing traffic** (e.g., server-initiated connections like HTTP requests).

8. **`sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT`**  
   - **Purpose**: Appends a rule to **accept established/related connections** (e.g., responses to open sessions).  
   - **Why**: Critical for **maintaining active connections** (e.g., SSH sessions won’t be dropped).  
   - **Note**: This rule must come **before** SSH/HTTP rules to allow traffic from external connections.

9. **`sudo iptables -A INPUT -s 10.0.0.0/24 -p tcp --dport 22 -j ACCEPT`**  
   - **Purpose**: Allows **TCP traffic to port 22 (SSH)** **only from `10.0.0.0/24`**.  
   - **Why**: Enforces the requirement to **restrict SSH to a specific subnet**.  
   - **Note**: `-s` specifies the source network. If this rule is omitted, SSH access would be denied.

10. **`sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT`**  
    - **Purpose**: Allows **TCP traffic to port 80 (HTTP)** from **any source**.  
    - **Why**: Explicitly allows HTTP access (as required).

11. **`sudo iptables -A INPUT -p tcp --dport 443 -j ACCEPT`**  
    - **Purpose**: Allows **TCP traffic to port 443 (HTTPS)** from **any source**.  
    - **Why**: Explicitly allows HTTPS access (as required).

12. **`sudo iptables-save > /etc/iptables/rules.v4`**  
    - **Purpose**: **Saves current rules** to a persistence file.  
    - **Why**: Ensures firewall rules **survive reboots** (via `iptables-persistent`).  
    - **Note**: Requires `iptables-persistent` to be installed. Run `sudo apt install iptables-persistent` if missing.

---

### Key Security Notes
- **Order matters**: Rules are processed **top-to-bottom**.  
  - `ESTABLISHED/RELATED` comes first to avoid blocking legitimate traffic.  
  - SSH rule (step 9) **must precede** HTTP/HTTPS rules (steps 10–11) to allow SSH sessions from `10.0.0.0/24`.
- **No UDP rules**: Ports 80/443 are TCP-only (HTTP/HTTPS), so UDP is not allowed by default. If needed, add `-p udp`.
- **Default policy enforcement**:  
  - `INPUT DROP` blocks **all unapproved traffic** (e.g., unsolicited SSH from `192.168.1.5`).  
  - `OUTPUT ACCEPT` permits **outbound traffic** (e.g., your server downloading updates).

---

### Verification

```bash

```
Expected output for the `INPUT` chain:
```

2     ACCEPT     tcp  --  10.0.0.0/24           0.0.0.0/0            tcp dpt:22
```

This configuration **exactly matches the requirements**. For production, add a **logging rule** (e.g., `LOG` rule) to monitor denied traffic.

## Question 6
**Q:** Ein Linux-Server zeigt folgende Symptome:
- Load average: 4.2 (bei 2 CPU-Kernen)
- top zeigt: 85% iowait
- df -h zeigt: /var ist zu 98% voll

Was sind die wahrscheinlichen Ursachen und wie gehst du schrittweise vor, um das Problem zu diagnostizieren und zu beheben? Bitte antworte auf Deutsch.

**A:**
### Wahrscheinliche Ursachen:
1. **Vollständige /var-Partition (98% belegt)**:  
   Die Partition `/var` wird hauptsächlich für Log-Dateien, temporäre Dateien und andere dynamische Daten genutzt. Da sie fast voll ist, können neue Schreibvorgänge nicht abgeschlossen werden, was zu I/O-Wait führt.  
   - **Konsequenz**: Prozesse (z. B. Systemdienste oder Anwendungen) versuchen weiterhin, Daten in `/var` zu schreiben, aber die Schreiboperationen blockieren sich – die CPU wartet im I/O-Status (`iowait`).

2. **Hochwertige Log-Dateien**:  
   In `/var/log` befinden sich oft große Log-Dateien (z. B. `syslog`, `auth.log`), die sich ohne korrekte Log-Rotation schnell anhäufen. Dies verursacht den Raumverbrauch und die Blockierung von Schreibvorgängen.

3. **Schreibblockierung durch volle Partition**:  
   Wenn eine Dateisystempartition voll ist, kann das System keine neuen Schreibvorgänge mehr initiieren. Dies führt zu einem hohen `iowait`, da die CPU auf die Beendigung der blockierten I/O-Operationen wartet.

---

### Schrittweise Diagnose und Behebung:

#### **1. Bestätige die Ursache durch Auswertung des Speicherplatzes**
- **Prüfe die Dateigrößen in `/var`**:  
  ```bash
  du -sh /var/*  # Zeigt die Größe der Unterverzeichnisse
  du -sh /var/log/*  # Fokussiert auf Log-Dateien
  ```
  - **Beispiel**: Wenn `log` oder `tmp` sehr groß sind, liegt der Fokus auf diesen.

- **Identifiziere größte Dateien**:  
  ```bash
  find /var -type f -size +100M  # Gibt Dateien >100MB aus
  ```
  - **Hinweis**: Große Log-Dateien in `/var/log` sind der häufigste Verdacht.

---

#### **2. Ermittle den konkreten Grund für den Speicherüberlauf**
- **Überprüfe Log-Dateien**:  
  ```bash
  less /var/log/syslog  # Öffne eine Log-Datei
  ```
  - **Anzeichen**: Füllende Logs ohne Rotation (z. B. Fehler in `rsyslog`).

- **Prüfe Log-Rotation**:  
  ```bash
  cat /etc/logrotate.conf  # Standardkonfiguration
  ls -lh /etc/logrotate.d/  # Service-spezifische Konfigurationen
  ```
  - **Problem**: Wenn `logrotate` nicht aktiviert ist oder falsch konfiguriert, werden Logs unkontrolliert angelegt.

- **Checke temporäre Dateien**:  
  ```bash
  ls -lh /var/tmp/  # Prüfe auf Dateien, die nicht gelöscht werden
  ```

---

#### **3. Lösung: Freiheit des Speicherplatzes**
- **Lösche alte Log-Dateien manuell**:  
  ```bash
  # Beispiel für das Löschen von Logs älter als 30 Tage
  find /var/log -type f -name "*.log" -mtime +30 -delete
  ```
  - **Achtung**: Vermeide die Lösung kritischer Log-Dateien (z. B. `auth.log`), falls nötig, prüfe mit `journalctl` alternative Protokolle.

- **Konfiguriere Log-Rotation**:  
  Erstelle eine spezifische `logrotate`-Konfiguration (z. B. `/etc/logrotate.d/custom`):  
  ```conf
  /var/log/syslog {
      daily
      rotate 7
      size 100M
      missingok
      compress
  }
  ```
  - **Starte die Rotation neu**:  
    ```bash
    systemctl restart logrotate
    ```

- **Entferne temporäre Dateien**:  
  ```bash
  rm -rf /var/tmp/*  # Falls sicher
  ```

---

#### **4. Vorbeugende Maßnahmen**
- **Erhöhe den Speicherplatz für `/var`**:  
  - Führe eine Partitionserweiterung durch (z. B. mit `LVM` oder `fdisk`), falls das Dateisystem nicht dynamisch skalierbar ist.
  - **Alternativ**: Erstelle eine neue Partition für `/var` mit ausreichend Platz.

- **Monitor die Speichernutzung**:  
  - Nutze `cronolog` oder `logrotate` zur automatischen Log-Verwaltung.
  - Setze Alarme mit `cron` oder `monitoring-tools` (z. B. `Prometheus` oder `Zabbix`).

- **Überprüfe Systemdienste**:  
  ```bash
  systemctl list-unit-files | grep -i log  # Prüfe, welche Dienste Logs generieren
  ```
  - **Beispiel**: Deaktiviere unnötige Dienste (z. B. `rsyslog`), falls nur kritische Logs benötigt werden.

---

#### **5. Verifiziere die Lösung**
- **Prüfe den Speicherplatz nach der Behebung**:  
  ```bash
  df -h /var  # Sollte < 20% belegen
  ```
- **Überprüfe den Load-Average**:  
  ```bash
  top  # iowait sollte unter 20% liegen
  ```

---

### Zusammenfassung:
- **Hauptursache**: Vollständige `/var`-Partition durch Log-Dateien ohne Log-Rotation.  
- **Schlüsselmaßnahme**: Löschen großer Log-Dateien und konfigurieren der Log-Rotation.  
- **Vorbeugung**: Automatisierte Log-Verwaltung und Speicherplatzüberwachung, um zukünftige Ausfälle zu vermeiden.  

Durch diese Schritte wird die Blockierung der I/O-Operationen beendet, sodass der `iowait` sinkt und der Load-Average wieder normalisiert wird.

## Question 7
**Q:** Explain the difference between Mock, Stub, and Fake in software testing. Then write a Python example using unittest.mock that:
- Mocks an external HTTP API call
- Verifies the mock was called with the correct arguments
- Tests the behavior when the API returns an error

Please answer in English.

**A:**
### Explanation of Mocking Concepts
- **Mock**: An object that verifies expectations (e.g., checking if a method was called with specific arguments or the number of times).
- **Stub**: A fake implementation that provides predefined return values (e.g., simulating an API response to isolate test logic).
- **Fake**: A simplified version of a real object with partial real functionality (e.g., a fake database with basic operations).

---

### Code Example for Testing API Calls
```python
import unittest
from unittest.mock import MagicMock, patch
import requests

def get_data():
    response = requests.get('https://api.example.com/data')
    response.raise_for_status()
    return response.json()

class TestAPI(unittest.TestCase):
    def test_correct_call(self):
        # Create a stub response to simulate successful API call
        mock_response = MagicMock()
        mock_response.status_code = 200
        mock_response.json.return_value = {"data": "success"}
        
        # Patch requests.get to return the mock response
        with patch('requests.get', return_value=mock_response) as mock_get:
            result = get_data()
            
            # Verify the call was made with the correct URL
            mock_get.assert_called_once_with('https://api.example.com/data')
            self.assertEqual(result, {"data": "success"})

    def test_error_handling(self):
        # Create a stub response with 404 status code
        mock_response = MagicMock()
        mock_response.status_code = 404
        mock_response.raise_for_status.side_effect = requests.exceptions.HTTPError("404 Not Found")
        mock_response.json.return_value = {}
        
        # Patch requests.get to return the mock response
        with patch('requests.get', return_value=mock_response) as mock_get:
            # Verify the exception is raised
            with self.assertRaises(requests.exceptions.HTTPError):
                get_data()
            # Verify the mock was called once
            mock_get.assert_called_once_with('https://api.example.com/data')
```

---

### Key Details
1. **Mock Usage** (`mock_get`):
   - Used in `test_correct_call` to verify the correct URL was called.
   - Tracks invocation counts and arguments to ensure the logic behaves as expected.

2. **Stubbing** (`mock_response`):
   - Simulates API responses (e.g., `200 OK` or `404`).
   - Controls return values (e.g., `json.return_value`) without relying on real network calls.

3. **Exception Handling**:
   - `test_error_handling` validates that the function correctly raises `HTTPError` for invalid responses.
   - `side_effect` ensures the mock triggers specific behavior when called.

This approach isolates the code from external dependencies (like network requests), making tests fast, reliable, and easy to debug.

## Question 8
**Q:** Bir Python projesinde Hermes mesaj kütüphanesini kullanarak publisher/subscriber pattern'i uyguluyorsun.

Aşağıdakileri açıkla ve kod örneği ver:
1. Publisher nasıl mesaj yayınlar?
2. Subscriber nasıl dinler?
3. Bu pattern'in doğrudan fonksiyon çağrısına göre avantajı nedir?

Cevabı Türkçe ver.

**A:**
### 1. Publisher Nasıl Mesaj Yayınlar?
Publisher, Hermes kütüphanesi aracılığıyla **belirli bir konuya (topic)** mesaj yayınlar. Bu işlemde, Publisher kütüphanesinden bir örneği oluşturup `publish()` metodunu kullanarak mesajı sunucuya gönderir. Publisher'in görevi, mesajı bir konuya yönlendirip alıcıların (Subscriber) mesajı alması için gerekli bağlantıyı kurmaktır.

**Kod Örneği:**
```python
from hermes import Publisher

# Publisher örneği oluştur
publisher = Publisher()

# Sunucuya bağlan (örneğin RabbitMQ veya benzer bir mesaj kuyruğu)
publisher.connect("localhost:5672")

# Mesaj yayınla (konu: "news", içerik: "Yeni haber")
publisher.publish("news", "Yeni haber: Dünya savaştı!")
```

---

### 2. Subscriber Nasıl Dinler?
Subscriber, **belirli bir konuya (topic) abone olur** ve gelen mesajları işler. Abone olurken bir geriye dönük fonksiyon (callback) tanımlanır. Bu fonksiyon, mesaj gelirken tetiklenerek işlenir. Subscriber, mesaj kuyruğuna bağlı olarak gelen mesajları asenkron olarak işler.

**Kod Örneği:**
```python
from hermes import Subscriber

# Subscriber örneği oluştur
subscriber = Subscriber()

# Sunucuya bağlan
subscriber.connect("localhost:5672")

# "news" konusuna abone ol ve mesajları işle
subscriber.subscribe("news", handle_news)

def handle_news(message):
    print("Gelen mesaj:", message)  # Mesajı alıp işler
```

---

### 3. Bu Pattern'ın Doğrudan Fonksiyon Çağrısına Göre Avantajı
**Publisher/Subscriber pattern**, doğrudan fonksiyon çağrıları (örneğin `funktionA().call(funktionB())`) ile karşılaştırıldığında aşağıdaki avantajları sunar:

1. **Bağımsızlık (Decoupling)**  
   Publisher ve Subscriber arasında doğrudan bağımlılık yoktur. Publisher mesajı yayınırken Subscriber'ın var olup olmadığını bilmez; bu nedenle bileşenlerin geliştirme ve bakımını kolaylaştırır.

2. **Asenkron Komünikasyon**  
   Mesaj yayın ve alma işlemleri asenkron (çok iş parçacıklı) olarak çalışır. Örneğin, bir Subscriber mesajı alırken, Publisher yeni bir mesaj yayınlamış olabilir.

3. **Skalayabilirlik ve Hata Toleransı**  
   Bileşenler yüklenirken (örneğin, bir Subscriber down olursa) mesajlar kuyruğa saklanır ve daha sonra yeniden işlemek üzere gönderilir. Bu, sistem kalıcılığını artırır.

4. **Esneklik**  
   Yeni Subscriber'lar kolayca konulara abone olabilir; bu da mevcut sistemde yeni işlevselliğe hızlı erişim sağlar.

**Örnek Senaryo:**  
Eğer `publish` ve `subscribe` kullanımı yerine doğrudan fonksiyon çağrısı kullanırsanız, bir servisin hatası (örneğin, `functionA()` çalışmazsa) diğer servisleri etkiler. Ancak Publisher/Subscriber pattern'ı bu sorunu ortadan kaldırır: `functionA()` başarısız olursa, `functionB()` mesajı kuyruğa eklenir ve daha sonra işlemeye devam eder.

Hermes kütüphanesi ile Publisher/Subscriber pattern'ı, **asenkron mesajlaşma** ve **bileşenlerin bağımsızlığı** sağlayarak sistemlerin daha esnek, skalayabilir ve hata toleranslı hale gelmesini sağlar. Doğrudan fonksiyon çağrılarına kıyasla, mesaj kuyrukları ile gerçek zamanlı veya kuyruk tabanlı iletişim kurulur.

### Corrected Code:

```python
import sqlite3
from contextlib import closing  # For automatic connection cleanup

def get_user_orders(username, db_path):
    """
    Safely retrieve orders for a user using parameterized queries and context management.
    
    Args:
        username (str): The user's name (e.g., "john_doe").
        db_path (str): Path to the SQLite database file.
        
    Returns:
        list: Rows of orders matching the username.
        
    Raises:
        sqlite3.Error: For database-related issues (e.g., invalid file).

    # Use context manager to ensure connection is closed
```
