**A:**
1. POST /auth/refresh: This endpoint is used to refresh an expired access token using the refresh token stored in the cookie. The following steps should be performed:
   - Fetch the refresh token from the cookie.
   - Check if the refresh token exists in the Redis store.
   - If it does, issue a new access token and refresh token.
   - Update the refresh token in the Redis store with the new refresh token and mark the old one as invalid.
   - Set the new refresh token in the cookie.

2. POST /auth/logout: This endpoint is used to log out a user, thus invalidating their refresh token. The following steps should be performed:
   - Fetch the refresh token from the cookie.
   - Check if the refresh token exists in the Redis store.
   - If it does, set its status as invalid.
   - Clear the refresh token from the cookie.

3. Implement refresh token rotation with a Redis-based token store. This involves storing refresh tokens in a Redis database and having an associated expiration time (TTL). 6. The difference between token blacklisting and token rotation lies in the way tokens are handled after they expire or are revoked.

   - Token rotation involves regularly generating new tokens and invalidating the old ones. This helps prevent token hijacking and can offer a certain level of security as the old tokens are immediately discarded.
   
   - Token blacklisting involves storing invalid or expired tokens in a separate data structure (like a blacklist in Redis) and checking them against incoming requests. This can offer a higher level of security as the tokens are immediately discarded after they expire or are revoked, but it can also increase the complexity of the system as tokens must be checked against the blacklist on every request.

Implement this in React using hooks only (no external state library). Show:
1. Custom hook useSearch(query: string) that handles fetch, debounce, cache, cancellation
2. The SearchPage component using this hook
3. URL sync implementation
4. Why is cancelling in-flight requests important here? What bug does it prevent?

**A:**
Here's a basic example of how you might implement this in React using hooks:

```javascript
import React, { useState, useEffect, useRef } from 'react';
import { useSearchParams } from 'react-router-dom';

// Custom hook to handle fetching, debouncing, caching, and cancellation
export function useSearch(query) {
  const [results, setResults] = useState([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);
  const cancelToken = useRef();
  const [params, setParams] = useSearchParams();

  useEffect(() => {
    setResults([]);
    setLoading(true);
    setError(null);

    // Set the query parameter in the URL
    setParams({ q: query });

    // Create a new cancel token source
    const source = axios.CancelToken.source();
    cancelToken.current = source;

    axios.get(`/api/search?q=${query}`, {
      cancelToken: source.token,
    })
    .then(response => {
      setResults(response.data);
    })
    .catch(error => {
      if (!axios.isCancel(error)) {
        setError(error);
      }
    })
    .finally(() => {
      setLoading(false);
    });

    // Clean up function to cancel the in-flight request
    return () => {
      source.cancel();
    };
  }, [query, setParams]);

  return { results, loading, error };
}

// The SearchPage component using the custom hook
export default function SearchPage() {
  const [search, setSearch] = useState('');
  const { results, loading, error } = useSearch(search);

  return (
    <div>
      <input
        type="search"
        value={search}
        onChange={e => setSearch(e.target.value)}
      />
      {loading && <div>Loading...</div>}
      {error && (
        <div>
          Error: {error.message}
          <button onClick={() => setSearch(search)}>Retry</button>
        </div>
      )}
      {!loading && !error && results.length === 0 && <div>No results</div>}
      {!loading && !error && results.length > 0 && (
        <ul>
          {results.map(result => (
            <li key={result.id}>{result.name}</li>
          ))}
        </ul>
      )}
    </div>
  );
}
```

The custom hook `useSearch` handles fetching, debouncing, caching, and cancellation. It sets up state variables for results, loading, and error. It also sets up an axios cancel token source, which is used to cancel the in-flight request if a new one is started.

The `SearchPage` component uses the `useSearch` hook to fetch and display search results. It also handles retries on errors and sets the search input value based on the query parameter in the URL.

Cancelling in-flight requests is important here because it prevents bugs like race conditions, where the server responds with the results of a previous search. This can occur if a new search request is started before the previous one has finished. By canceling the in-flight request when a new one is started, we ensure that only the most recent search results are displayed to the user.

## Question 9
**Q:** You are building a collaborative document editor (like Google Docs, simplified):
- Multiple users can edit the same document simultaneously
- Each user sees others' cursors in realtime
- Changes are broadcast to all connected clients
- If a user disconnects and reconnects, they get the latest document state

Implement the server side in Node.js using the ws library:
1. Connection handling — how do you associate a WebSocket connection with a user and document?
2. Message protocol — define the message types (JSON) for: join document, text change, cursor move, user left
3. Broadcasting — when user A makes a change, how do you send it to all other users in the same document but NOT back to user A?
4. Reconnection — how do you handle a user rejoining after disconnect?
5. What happens if two users edit the exact same position simultaneously? You don't need to implement OT/CRDT — just explain the problem and name the solution approaches.

Write the core server implementation (connection map, message routing, broadcast logic).

**A:**
Here is a basic implementation of the document collaboration system using Node.js and the ws library:

```javascript
const WebSocket = require('ws');

const clients = new Map();
const documents = new Map();

const broadcast = (documentId, message, senderId) => {
  const clientsInDoc = clients.get(documentId) || [];
  for (const client of clientsInDoc) {
    if (client.id !== senderId) {
      client.send(JSON.stringify(message));
    }
  }
};

const wss = new WebSocket.Server({ port: 8080 });

wss.on('connection', (ws) => {
  ws.on('message', (message) => {
    const { type, documentId, senderId, ...rest } = JSON.parse(message);
    switch (type) {
      case 'JOIN':
        if (!documents.has(documentId)) {
          documents.set(documentId, new Set());
        }
        documents.get(documentId).add(ws);
        ws.documentId = documentId;
        ws.id = senderId;
        break;
      case 'CHANGE':
        broadcast(documentId, { type: 'CHANGE', ...rest }, senderId);
        break;
      case 'CURSOR_MOVE':
        broadcast(documentId, { type: 'CURSOR_MOVE', ...rest }, senderId);
        break;
      case 'USER_LEFT':
        broadcast(documentId, { type: 'USER_LEFT', ...rest }, senderId);
        break;
      default:
        break;
    }
  });

  ws.on('close', () => {
    if (ws.documentId) {
      const clientsInDoc = documents.get(ws.documentId);
      clientsInDoc.delete(ws);
      if (clientsInDoc.size === 0) {
        documents.delete(ws.documentId);
      }
    }
  });
});
```

In this code, we keep track of all connected clients in a `Map` called `clients` and each document's clients in a `Map` called `documents`. When a client joins, we add it to the corresponding document set. When a message is received, we check the message type and act accordingly. If it's a text change or cursor movement, we broadcast the message to all other clients in the same document. If it's a user leaving, we notify the other clients. If the client disconnects, we remove it from the document set.

This basic implementation does not handle simultaneous edits for a single position. If two users try to edit the exact same position simultaneously, the last change received by the server will overwrite the previous one. To prevent this, a more complex conflict resolution strategy would be necessary, such as Operational Transformation (OT) or Conflict-free Replicated Data Types (CRDTs). This would involve tracking the sequence of operations for each client, and then merging changes in a way that avoids conflicts. This, however, is beyond the scope of this basic implementation.
