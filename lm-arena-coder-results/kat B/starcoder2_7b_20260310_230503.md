## What is the difference between token blacklisting and token rotation? When would you use each?

**Blacklisting** is when you store all the tokens in memory that have been used. When a token is used, it is added to this blacklist and if it is found in the blacklist then you know that it is already used. This means that you have to keep track of the tokens to know when it has been used already and if it's been used already, you can't use it again and have to reject the request.

**Rotation** is when the tokens aren't stored in memory and instead you use an external storage service like Redis or Memcache to store the tokens. You don't have to keep track of which tokens have already been used and when it has been used and you can just store all the tokens in Redis or Memcache and when someone wants to use a token, you just look to see if it's been used before and reject the request if it has been used before. 

I would use **token rotation** when I know for sure that the tokens will be used once and then it would be wasteful to keep them in memory because you would have to check to see if it has already been used and this is more resource intensive. If a token would be used multiple times and you wanted to ensure that it is only used once then you would use **token blacklisting**.

**Token blacklisting** would be useful for verifying email addresses and sending confirmation emails and if the token is blacklisted then you don't need to verify it again because it is already verified and you don't need to send the confirmation email again because it is already sent.

**Token rotation** would be useful for logging in. If you have an endpoint for logging in, you would first check to see if the token is blacklisted. If it is, then you don't want to allow the user to log in because the token has already been used and you would send them an error saying that the token has already been used. If the token isn't blacklisted then you know that it hasn't been used yet so you would allow them to log in and issue a new access token and a new refresh token and then invalidate the refresh token. 

A user changing their password would be useful for token rotation because the user could be logging in with the old password and then changing their password and now they want to log in with their new password. If they are using a token rotation system then they would have to log in with their new password and then issue a new access token and a new refresh token. If they are using a token blacklisting system then they would have to log in with their new password and then it would be blacklisted and they couldn't use it until they changed their password again and then they would have to log in with their new password again and then it would be blacklisted and then they wouldn't be able to use it until they changed their password again and so on and so on. That is why it is a good idea to implement token rotation so that if a user is trying to use a token twice they can't use it twice. If they are trying to use it twice, then it is a good idea to send them an error saying that they tried to use a token twice and to ask them to try again because they are not allowed to use it twice.
