**Q:** Implement the following endpoint in Node.js + Express + TypeScript:

- Validate that assignee_id (if provided) is a member of the workspace
- Return 400 with field-level validation errors if validation fails
- Return 409 if trying to set status to 'cancelled' when task has open subtasks

**A:**

```typescript
import { Request, Response, NextFunction } from 'express';
import { z } from 'zod';

// Validation schema
const taskSchema = z.object({
  title: z.string().min(1).max(200),
  status: z.enum(['todo', 'in_progress', 'done', 'cancelled']),
  assignee_id: z.number().nullable(),
  due_date: z.string().iso8601().nullable(),
});

// Error middleware
const validationErrorMiddleware = (err: any, req: Request, res: Response, next: NextFunction) => {
  if (err instanceof z.ZodError) {
    return res.status(400).json({ errors: err.issues });
  }
  next(err);
};

// PATCH /workspaces/:workspaceId/tasks/:taskId
const patchTask = async (req: Request, res: Response) => {
  // Validate workspaceId and taskId
  const workspaceId = parseInt(req.params.workspaceId);
  const taskId = parseInt(req.params.taskId);

  // Validate task
  const task = taskSchema.safeParse(req.body);
  if (!task.success) {
    return res.status(400).json({ errors: task.error.issues });
  }

  // Check if task belongs to workspace
  const taskExists = await Task.findOne({ workspaceId, _id: taskId });
  if (!taskExists) {
    return res.status(404).json({ error: 'Task not found' });
  }

  // Validate assignee_id
  if (task.data.assignee_id) {
    const assigneeExists = await User.findOne({ _id: task.data.assignee_id, workspaceId });
    if (!assigneeExists) {
      return res.status(400).json({ errors: [{ path: 'assignee_id', message: 'Assignee not found' }] });
    }
  }

  // Check for open subtasks
  if (task.data.status === 'cancelled' && taskExists.subtasks.some(subtask => subtask.status !== 'done')) {
    return res.status(409).json({ error: 'Cannot cancel task with open subtasks' });
  }

  // Update task
  await Task.findByIdAndUpdate(taskId, task.data, { new: true });

  // Return success
  res.status(200).json(task.data);
};

export default patchTask;
```

**Q:** Continuing from the previous auth system:

1. POST /auth/refresh — use the refresh token from cookie to issue a new access token. Rotate the refresh token (issue new one, invalidate old one).

**A:**
**Refresh Token Rotation with Redis:**

**Redis Data Structure:**

```
HSET refresh_tokens user_id token_value ttl
```

* `user_id`: Unique identifier of the user.
* `token_value`: The refresh token.
* `ttl`: Time-to-live for the token in seconds.

**TTL Strategy:**

* Set the TTL of the refresh token in Redis to a fixed value (e.g., 30 days).
* When a new refresh token is issued, update the TTL of the previous token in Redis.

**Token Reuse Detection:**

* When a user presents a refresh token, check if it exists in Redis.
* If it does, check if the token value matches the value stored in Redis.
* If they don't match, it indicates a token reuse attempt.

**Invaliding All User Sessions on Password Change:**

* When a user changes their password, invalidate all refresh tokens associated with that user in Redis.
* You can achieve this by iterating over the refresh token entries in Redis and checking the user ID associated with each token.

**Token Blacklisting vs. Token Rotation:**

**Token Blacklisting:**

* Store the refresh token in a blacklist (e.g., Redis set).
* When a user tries to use a blacklisted token, deny access.

**Token Rotation:**

* Issue a new refresh token and invalidate the old one.
* This approach prevents token reuse but requires additional logic to manage token expiry and Redis TTLs.

**Use Cases:**

* **Token blacklisting:** Suitable for scenarios where security is the primary concern and token reuse is a rare event.
* **Token rotation:** More complex but provides better security against token leakage or reuse.

**Conclusion:**

Implementing refresh token rotation with Redis is an effective approach for securing user sessions. By detecting token reuse and invalidating tokens upon password change, you can enhance the security of your authentication system.

```
import { useState, useEffect, useRef } from 'react';

const useSearch = (query) => {
  const [results, setResults] = useState([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState(null);
  const cancelToken = useRef(null);

  useEffect(() => {
    if (!query) {
      return;
    }

    const timeoutId = setTimeout(() => {
      cancelToken.current = new AbortController().signal;
      fetch(`/api/search?q=${query}`, { signal: cancelToken.current })
        .then(res => res.json())
        .then(data => {
          setResults(data);
          setError(null);
        })
        .catch(err => {
          if (err.name !== 'AbortError') {
            setError(err);
          }
        })
        .finally(() => {
          setLoading(false);
        });
    }, 300);

    return () => clearTimeout(timeoutId);
  }, [query]);

  return { results, loading, error, cancelToken };
};

export default useSearch;
```

**A:**
## Notification System Design

**1. Transport Mechanism:**

* **WebSocket:** Real-time updates, low latency, efficient for continuous communication between server and client.
* **SSE (Server-Sent Events):** Similar to WebSocket, but simpler and more lightweight.
* **Polling:** Less efficient, requires frequent client requests for updates.
* **Push Notifications:** Can be used as a fallback if WebSocket or SSE are unavailable.

**Justification:** WebSocket and SSE are the best options due to their low latency and real-time capabilities. They allow users to be notified in real-time when there are updates to their tasks.

**2. Database Schema:**

```
Notifications Table:
- user_id (foreign key)
- task_id (foreign key)
- notification_type (assigned, commented, deadline_approaching)
- message
- timestamp
```

**3. Handling Offline Users:**

* Store missed notifications in the database.
* When a user comes back online, send them a list of missed notifications.

**4. Deadline Check:**

* **Event-driven approach:**
    * Schedule a cron job to run every hour.
    * Use an event queue to store tasks with upcoming deadlines.
    * Process events in the queue and send notifications to users.

**5. Scaling for 100,000 Users:**

* Use a cloud-based notification service that can handle large numbers of concurrent users.
* Implement a message queue to distribute notification tasks across multiple servers.
* Scale the database and server infrastructure as needed.

**Additional Considerations:**

* Error handling and logging for notification delivery failures.
* User preferences for notification settings (email, SMS, in-app).
* Security measures to prevent unauthorized access to notifications.

**Conclusion:**

By using WebSocket and an event-driven approach, we can design a scalable and efficient notification system for the project management tool. This system ensures that users are notified in real-time about important updates to their tasks and deadlines.
