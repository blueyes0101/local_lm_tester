### **4. Error Response Format**
All errors return a consistent structure:
```json
{
  "error": {
    "code": "VALIDATION_ERROR",  // e.g., "NOT_FOUND", "FORBIDDEN", "CONFLICT"
    "message": "Task due date cannot be in the past.",
    "details": {
      "field": "dueDate",
      "issue": "must be a future date"
    },
    "timestamp": "2026-03-11T12:00:00Z"
  }
}
```

**Common Status Codes:**
- `200 OK`: Successful GET/PATCH.
- `201 Created`: Successful POST.
- `204 No Content`: Successful DELETE.
- `400 Bad Request`: Client-side validation error.
- `401 Unauthorized`: Missing/invalid auth token.
- `403 Forbidden`: User lacks permissions.
- `404 Not Found`: Resource doesn’t exist.
- `409 Conflict`: Optimistic concurrency failure (e.g., version mismatch).
- `422 Unprocessable Entity`: Semantic validation error (e.g., invalid status transition).

---

### **5. Pagination**
All list endpoints use **offset-based pagination** (simpler for most SaaS tools):
```json
{
  "data": [...],  // Array of resources
  "pagination": {
    "total": 100,  // Total items (without filters)
    "limit": 20,   // Items per page
    "offset": 0,   // Current offset
    "pages": 5,    // Total pages
    "next": "/v1/ws123/tasks?offset=20",  // Optional: Next page URL
    "prev": "/v1/ws123/tasks?offset=0"    // Optional: Previous page URL
  }
}
```

**Alternatives Considered:**
- **Cursor-based pagination**: Better for infinite scroll but harder to implement with filters.
- **Key-based pagination**: Useful for time-series data (e.g., `after: task123`).

---

### **6. Key Design Decisions**
1. **Tenant Isolation**: All endpoints are scoped to `{tenantId}` (workspace ID) in the URL.
2. **Optimistic Locking**: Tasks include a `version` field to handle concurrent updates.
3. **Filtering**: Tasks support filtering by `assignee`, `status`, and `dueDate` (range queries).
4. **Partial Updates**: `PATCH /tasks/{id}/status` for atomic status changes.
5. **Links**: All resources include `links` for self-referencing and related resources.
6. **Error Consistency**: Standardized error format across all endpoints.

---
### **7. Example Workflow**
1. **Create a Task**:
   `POST /v1/ws123/tasks` → Returns `task123`.
2. **List Tasks for User**:
   `GET /v1/ws123/tasks?filters[assigneeId]=user789`.
3. **Update Status**:
   `PATCH /v1/ws123/tasks/task123/status` → Returns updated task.
4. **Add Comment**:
   `POST /v1/ws123/tasks/task123/comments` → Returns `comment456`.

### **4. Token Reuse Detection**
**Problem:** If an attacker steals a refresh token, they can reuse it until it expires.
**Solution:** Track token usage with a `last_used_at` field and block after 1 reuse.

**Implementation:**
```python
# Redis key: rt:user123:abc123:usage
SET rt:user123:abc123:usage 1 EX 300  # TTL: 5 minutes (prevent replay)

# On refresh:
usage_count = redis.incr("rt:user123:abc123:usage")
if usage_count > 1:
    raise HTTPException(status_code=403, detail="Token reused")
```

**Alternative:** Use a **sliding window** (e.g., track last 3 uses in a sorted set):
```bash
# On refresh:
ZADD rt:user123:abc123:window 1234567890 "usage1"
ZREM rt:user123:abc123:window (0..1234567880)  # Remove old entries
if ZCARD rt:user123:abc123:window > 1:
    raise HTTPException(status_code=403, detail="Token reused")
```

### **6. Token Blacklisting vs. Rotation**
| Feature               | **Token Rotation**                          | **Token Blacklisting**                     |
|-----------------------|--------------------------------------------|--------------------------------------------|
| **Mechanism**         | Issue new token, invalidate old one.       | Maintain a list of invalid tokens.         |
| **Storage Overhead**  | Low (only current token stored).           | High (grow with invalidated tokens).       |
| **Replay Protection** | Limited (requires TTL + usage tracking).   | Strong (explicitly block known tokens).    |
| **Use Case**          | Stateless systems (JWT).                   | Stateful systems (e.g., session tokens).   |
| **Performance**       | O(1) lookup (Redis).                       | O(n) lookup (blacklist scan).              |
| **Scalability**       | Excellent (no blacklist growth).           | Poor (blacklist bloat).                    |

**When to Use:**
- **Rotation:** Prefer for JWT-based auth (stateless, no server-side storage).
- **Blacklisting:** Use for session tokens (e.g., Flask-Session) or when rotation is impractical.

**Hybrid Approach:**
- Use **rotation** for refresh tokens (stateless).
- Use **blacklisting** for short-lived access tokens (if stateful).

---

### **Security Hardening**
1. **Refresh Token Storage:**
   - Store in `HttpOnly`, `Secure`, `SameSite=Strict` cookies.
   - Bind to IP/user-agent (optional but recommended).

2. **Token Expiry:**
   - Access tokens: 15 mins.
   - Refresh tokens: 30 days (but rotate every use).

3. **Rate Limiting:**
   - Limit refresh attempts to 3/hour per IP.

4. **Logging:**
   - Log all refresh/token invalidation events.

5. **Password Change:**
   - Force re-login after password change (invalidate all sessions).

### 2. Complete Component Implementation

```tsx
import React, { useState, useMemo, useCallback } from 'react';
import { DataTableProps } from './DataTable.types';

const DataTable = <T, U extends keyof T>({
  columns,
  data,
  selectedRows,
  onRowSelect,
  onSelectAll,
  sortColumn,
  sortDirection = 'none',
  pageSize,
  onPageSizeChange,
  onPageChange,
  defaultPageSize = 10,
  defaultPage = 0,
  onSort,
  className = '',
  tableClassName = '',
  headerClassName = '',
  rowClassName = '',
}: DataTableProps<T, U>) => {
  // Memoized sorted data
  const sortedData = useMemo(() => {
    if (!sortColumn) return data;

    return [...data].sort((a, b) => {
      const aValue = a[sortColumn];
      const bValue = b[sortColumn];

      // Handle different types
      if (typeof aValue === 'string' && typeof bValue === 'string') {
        return sortDirection === 'asc'
          ? aValue.localeCompare(bValue)
          : bValue.localeCompare(aValue);
      }

      if (typeof aValue === 'number' && typeof bValue === 'number') {
        return sortDirection === 'asc'
          ? aValue - bValue
          : bValue - aValue;
      }

      return 0;
    });
  }, [data, sortColumn, sortDirection]);

  // Memoized paginated data
  const paginatedData = useMemo(() => {
    const start = pageSize * defaultPage;
    return sortedData.slice(start, start + pageSize);
  }, [sortedData, pageSize, defaultPage]);

  // Memoized total pages
  const totalPages = Math.ceil(sortedData.length / pageSize);

  // Handle sort column click
  const handleSort = useCallback((column: U) => {
    if (!columns.find(c => c.key === column)) return;

    const newDirection: SortDirection =
      sortColumn === column && sortDirection === 'asc' ? 'desc' :
      sortColumn === column && sortDirection === 'desc' ? 'none' :
      sortDirection === 'none' ? 'asc' : 'none';

    onSort?.(column, newDirection);
  }, [sortColumn, sortDirection, onSort]);

  // Handle page size change
  const handlePageSizeChange = useCallback((e: React.ChangeEvent<HTMLSelectElement>) => {
    const size = parseInt(e.target.value);
    onPageSizeChange(size);
  }, [onPageSizeChange]);

  // Handle page change
  const handlePageChange = useCallback((page: number) => {
    onPageChange(page);
  }, [onPageChange]);

  // Check if row is selected
  const isRowSelected = useCallback((row: T) => {
    return selectedRows.some(r => r === row);
  }, [selectedRows]);

  // Handle row selection
  const handleRowSelect = useCallback((row: T, selected: boolean) => {
    onRowSelect(row, selected);
  }, [onRowSelect]);

  // Handle select all
  const handleSelectAll = useCallback((selected: boolean) => {
    onSelectAll(selected);
  }, [onSelectAll]);

  // Calculate select all state
  const selectAllChecked = useMemo(() => {
    if (paginatedData.length === 0) return false;
    return paginatedData.every(row => isRowSelected(row));
  }, [paginatedData, isRowSelected]);

  return (
    <div className={`datatable-container ${className}`}>
      <table className={`datatable ${tableClassName}`}>
        <thead>
          <tr className={headerClassName}>
            <th style={{ width: '50px' }}>
              <input
                type="checkbox"
                checked={selectAllChecked && paginatedData.length > 0}
                onChange={(e) => handleSelectAll(e.target.checked)}
                disabled={paginatedData.length === 0}
              />
            </th>
            {columns.map((column) => (
              <th
                key={column.key as string}
                style={{ width: column.width, textAlign: column.align || 'left' }}
                onClick={() => column.sortable && handleSort(column.key as U)}
              >
                {column.header}
                {column.sortable && (
                  <span>
                    {sortColumn === column.key && (
                      sortDirection === 'asc' ? ' ↑' : sortDirection === 'desc' ? ' ↓' : ''
                    )}
                  </span>
                )}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {paginatedData.length > 0 ? (
            paginatedData.map((row, rowIndex) => (
              <tr key={`${rowIndex}-${JSON.stringify(row)}`} className={rowClassName}>
                <td style={{ width: '50px' }}>
                  <input
                    type="checkbox"
                    checked={isRowSelected(row)}
                    onChange={(e) => handleRowSelect(row, e.target.checked)}
                  />
                </td>
                {columns.map((column) => (
                  <td
                    key={`${column.key}-${rowIndex}`}
                    style={{
                      width: column.width,
                      textAlign: column.align || 'left',
                      whiteSpace: 'nowrap',
                      overflow: 'hidden',
                      textOverflow: 'ellipsis'
                    }}
                  >
                    {column.renderCell ? (
                      column.renderCell(row)
                    ) : (
                      <span>{String(row[column.key])}</span>
                    )}
                  </td>
                ))}
              </tr>
            ))
          ) : (
            <tr>
              <td colSpan={columns.length + 1} style={{ textAlign: 'center' }}>
                No data available
              </td>
            </tr>
          )}
        </tbody>
      </table>

      <div className="datatable-pagination">
        <div className="page-size-selector">
          <label>Rows per page:</label>
          <select
            value={pageSize}
            onChange={handlePageSizeChange}
          >
            <option value="10">10</option>
            <option value="25">25</option>
            <option value="50">50</option>
          </select>
        </div>

        <div className="page-navigation">
          <button
            onClick={() => handlePageChange(Math.max(0, defaultPage - 1))}
            disabled={defaultPage === 0}
          >
            Previous
          </button>
          <span>Page {defaultPage + 1} of {totalPages}</span>
          <button
            onClick={() => handlePageChange(Math.min(totalPages - 1, defaultPage + 1))}
            disabled={defaultPage >= totalPages - 1}
          >
            Next
          </button>
        </div>
      </div>
    </div>
  );
};

export default DataTable;
```

### 1. Custom Hook: `useSearch`

```tsx
import { useState, useEffect, useRef, useCallback } from 'react';

type SearchResult = {
  id: string;
  title: string;
  // Add other result properties as needed
};

type SearchError = {
  message: string;
};

export const useSearch = (query: string) => {
  const [results, setResults] = useState<SearchResult[]>([]);
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState<SearchError | null>(null);
  const [cache, setCache] = useState<Record<string, { data: SearchResult[]; timestamp: number }>>({});
  const controllerRef = useRef<AbortController | null>(null);
  const lastQueryRef = useRef(query);

  const fetchResults = useCallback(async (q: string) => {
    if (!q.trim()) {
      setResults([]);
      return;
    }

    // Check cache first
    const cached = cache[q];
    if (cached && Date.now() - cached.timestamp < 5 * 60 * 1000) {
      setResults(cached.data);
      return;
    }

    setIsLoading(true);
    setError(null);

    // Cancel previous request if it exists
    controllerRef.current?.abort();

    controllerRef.current = new AbortController();
    const { signal } = controllerRef.current;

    try {
      const response = await fetch(`/api/search?q=${encodeURIComponent(q)}`, { signal });

      if (!response.ok) throw new Error('Network response was not ok');

      const data: SearchResult[] = await response.json();

      // Update cache
      setCache(prev => ({
        ...prev,
        [q]: { data, timestamp: Date.now() }
      }));

      setResults(data);
    } catch (err) {
      if (err.name !== 'AbortError') {
        setError({ message: 'Failed to fetch results. Please try again.' });
      }
    } finally {
      setIsLoading(false);
    }
  }, [cache]);

  useEffect(() => {
    // Only proceed if query has changed
    if (query !== lastQueryRef.current) {
      lastQueryRef.current = query;
      fetchResults(query);
    }
  }, [query, fetchResults]);

  const retrySearch = useCallback(() => {
    if (error) {
      fetchResults(query);
    }
  }, [error, query, fetchResults]);

  return {
    results,
    isLoading,
    error,
    retrySearch,
    clearCache: () => setCache({})
  };
};
```

### 4. Why Cancelling In-Flight Requests is Important

Cancelling in-flight requests prevents several potential issues:

1. **Performance Optimization**: When users type quickly, multiple requests might be made for similar queries. Without cancellation, you'd be:
   - Making unnecessary network requests
   - Keeping the UI in a loading state longer than needed
   - Potentially overwhelming the server with redundant requests

2. **Memory Leaks**: Without cancellation, old AbortController instances might persist, keeping references to network requests alive even after components unmount.

3. **Race Conditions**: If a new request completes after the UI has already updated with a previous response, you might show stale data.

4. **Resource Wastage**: Each new request consumes bandwidth and server resources, which could be better spent on the most recent query.

5. **UI Confusion**: The loading state might persist longer than expected, creating a poor user experience.

The specific bug cancellation prevents is the "stale data" scenario where:
- User types "react"
- First request completes with results
- User continues typing "react hooks"
- Second request completes with more relevant results
- But the UI might still show the first results because the second request was slower to complete

By cancelling the first request when the second begins, we ensure only the most recent query's results are displayed.

```javascript
  return documents.get(documentId);

    try {
      const message = JSON.parse(rawMessage);
      handleMessage(connection, message);
    } catch (err) {
      console.error('Invalid message:', err);

      handleJoinDocument(connection, message);
      break;
    case MESSAGE_TYPES.TEXT_CHANGE:
      handleTextChange(connection, message);
      break;
    case MESSAGE_TYPES.CURSOR_MOVE:
      handleCursorMove(connection, message);
      break;
    default:
      console.warn('Unknown message type:', message.type);

    ws.send(JSON.stringify({ error: 'Document ID is required' }));
    return;
```

**Q:** You are asked to add real-time notifications to the existing project management tool. Users should be notified when: a task is assigned to them, a comment is added to their task, a deadline is approaching (within 24h).

Design the notification system:

**A:**
Here’s a comprehensive design for a real-time notification system in your project management tool, addressing each of your questions:

---

### **1. Transport Mechanism: WebSocket (with Fallback)**
**Why WebSocket?**
- **Real-time bidirectional communication**: Ideal for instant notifications (task assignment, comments, deadlines).
- **Low latency**: Unlike polling (e.g., every 30s), WebSocket keeps the connection open, reducing delays.
- **Efficient**: Only sends data when needed (unlike SSE, which streams continuously).
- **Scalable**: Can be load-balanced with tools like Redis pub/sub or Pusher.

**Fallback for Offline Users**:
- If WebSocket is unavailable, use **Server-Sent Events (SSE)** for browser-based users.
- For mobile apps, use **push notifications** (FCM for Android, APNs for iOS).
- **Polling** (e.g., every 5 minutes) as a last resort for critical notifications.

**Comparison Table**:
| Mechanism       | Pros                          | Cons                          | Best For                     |
|-----------------|-------------------------------|-------------------------------|------------------------------|
| **WebSocket**   | Low latency, bidirectional    | Complex to scale               | Primary real-time channel    |
| **SSE**         | Simple, unidirectional        | No browser control            | Fallback for browsers        |
| **Push Notifs** | Works offline, instant         | Platform-specific (FCM/APNs)  | Mobile apps                  |
| **Polling**     | Works everywhere              | High server load, delayed     | Rare cases (e.g., offline)   |

---

### **2. Database Schema for Notifications**
**Tables**:
1. **`notifications`** (Stores all notifications):
   ```sql
   CREATE TABLE notifications (
     id BIGSERIAL PRIMARY KEY,
     user_id BIGINT NOT NULL,  -- Target user
     task_id BIGINT,           -- Related task (NULL for system-wide)
     type ENUM('task_assigned', 'comment_added', 'deadline_approaching', 'system') NOT NULL,
     message TEXT NOT NULL,
     is_read BOOLEAN DEFAULT FALSE,
     created_at TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
     metadata JSONB,           -- Extra data (e.g., {"assigner_id": 123, "deadline": "2026-03-12"})
     expires_at TIMESTAMP WITH TIME ZONE  -- For cleanup (e.g., 30 days old)
   );
   ```

2. **`notification_preferences`** (User-specific settings):
   ```sql
   CREATE TABLE notification_preferences (
     user_id BIGINT PRIMARY KEY,

     sse_notifications BOOLEAN DEFAULT TRUE,
     last_seen_notification_id BIGINT  -- For pagination
   );
   ```

3. **`notification_subscriptions`** (WebSocket/SSE connections):
   ```sql
   CREATE TABLE notification_subscriptions (
     id UUID PRIMARY KEY,
     user_id BIGINT NOT NULL,
     connection_type ENUM('websocket', 'sse', 'push') NOT NULL,
     device_token VARCHAR(255),  -- For push notifs
   ```

**Indexing**:
- `notifications(user_id, is_read, created_at)` for fast queries.
- `notification_subscriptions(user_id, connection_type)` for routing.

### **3. Handling Offline Users**
**Approach**:
1. **Queue Notifications**: Store notifications in `notifications` table even if the user is offline.
2. **Retry Mechanism**:
   - For WebSocket/SSE: Reconnect and send queued notifications when the user comes back online.
   - For Push Notifications: Retry delivery via FCM/APNs (exponential backoff).
3. **Expiration**:
   - Delete notifications older than 30 days (or mark as `expires_at`).
   - For critical notifications (e.g., deadlines), send an email if push fails.

**Example Workflow**:
```
User goes offline → Notification is queued in DB.
User returns online → System checks `notification_subscriptions` and sends queued notifs.
```

---

### **4. Deadline Check Implementation**
**Approach**: **Event-Driven + Cron Job Hybrid**
- **Cron Job**: Run hourly to check tasks due within 24h (e.g., `0 * * * *`).
- **Event Queue**: For scalability, use a queue (e.g., RabbitMQ, AWS SQS) to process tasks asynchronously.

**Implementation Steps**:
1. **Cron Job**:
   ```bash
   # Run every hour
   0 * * * * /usr/bin/curl -X POST http://backend-api/check-deadlines
   ```
   - Fetches tasks due in <24h from the DB.
   - Enqueues notification events for each affected user.

2. **Queue Worker**:
   - Consumes events from the queue (e.g., `deadline_approaching`).
   - Sends notifications via WebSocket/SSE/push.
   - Updates `notifications` table.

**Why Not Just Cron?**
- Cron alone is synchronous and can’t handle spikes (e.g., 100K users with deadlines).
- Queue decouples the check from notification delivery.

**Alternative**: Use a **database trigger** (if DB supports it) to fire events when deadlines are near, but queues are more flexible.

---

### **5. Scaling to 100K Concurrent Users**
**Challenges**:
- WebSocket connections (memory/CPU).
- Notification queue backlog.
- Database load (reads/writes).

**Solutions**:
1. **WebSocket Scaling**:
   - Use a **WebSocket gateway** (e.g., Pusher, Ably, or custom with Redis pub/sub).
   - Horizontal scaling: Deploy multiple app servers behind a load balancer.
   - Connection pooling: Limit concurrent WebSocket connections per user.

2. **Queue Scaling**:
   - Distribute queue workers across multiple machines (e.g., 10 workers for 100K users).
   - Use a distributed queue (e.g., Kafka, AWS SQS with FIFO).

3. **Database**:
   - Shard notifications by user ID (e.g., `notifications_1` to `notifications_10`).
   - Read replicas for `notification_preferences` queries.

4. **Caching**:
   - Cache frequent queries (e.g., "notifications for user X") with Redis.
   - Use Redis pub/sub to broadcast notifications to active users.

5. **Monitoring**:
   - Track queue depth, WebSocket connection drops, and notification latency.
   - Auto-scale workers based on queue length.

**Architecture Diagram**:
```
[Users] → [Load Balancer] → [App Servers (WebSocket)]
       ↓
[Cron Job] → [Queue (RabbitMQ)] → [Notification Workers]
       ↓
[Database (Sharded)] ←→ [Redis (Cache/PubSub)]
```
