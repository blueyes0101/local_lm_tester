**Q:** Continuing from the previous auth system:

**A:**
### 1. **POST /auth/refresh — Refresh Token Rotation**

**Implementation Steps:**
- **Extract the refresh token** from the cookie (e.g., `refresh_token`).
- **Validate the refresh token** by checking its existence in Redis.
- **Generate a new access token** (short-lived, e.g., 1 hour) and a **new refresh token** (longer-lived, e.g., 7 days).
- **Invalidate the old refresh token** by deleting its entry from Redis.
- **Store the new refresh token** in Redis with the same user ID and TTL.
- **Return the new access token** to the client and update the cookie with the new refresh token.

**Example Redis Data Structure:**
```plaintext
# Key: refresh_token:{token}
# Value: {user_id, expiration_time}
```

**Redis TTL Strategy:**
- **Refresh tokens:** 7 days (e.g., `EX 604800`).
- **Access tokens:** 1 hour (e.g., `EX 3600`).

---

### 2. **POST /auth/logout — Invalidate Refresh Token**

**Implementation Steps:**
- **Delete the refresh token** from Redis (e.g., `DEL refresh_token:{token}`).
- **Clear the refresh token cookie** on the client side to prevent future use.

---

### 3. **Redis-Based Token Store with TTL Strategy**

**Redis Data Structures:**
- **Refresh Token Mapping:**
  ```plaintext
  # Key: refresh_token:{token}
  # Value: {user_id, expiration_time}
  ```
- **User Refresh Tokens (Optional for password change):**
  ```plaintext
  # Key: user:{user_id}:refresh_tokens
  # Value: Set of refresh tokens
  ```

**TTL Strategy:**
- **Refresh tokens:** 7 days (ensure they expire after a reasonable time).
- **Access tokens:** 1 hour (short-lived for security).

---

### 4. **Token Reuse Detection (Refresh Token Rotation)**

**Mechanism:**
- **Token rotation** ensures that each refresh token is valid **only once**.
- When a refresh token is used:
  1. Check if it exists in Redis.
  2. If valid, delete it from Redis (invalidates the old token).
  3. Generate and store a new refresh token.
- **Reuse detection:** If a refresh token is used again, it will no longer exist in Redis, so the request is denied.

**Example:**
```plaintext
# First use: refresh_token:old_token exists → delete it, store new_token.
# Second use: refresh_token:old_token does not exist → invalid.
```

---

### 5. **Invalidate All Sessions on Password Change**

**Implementation Steps:**
- **When a user changes their password:**
  1. Retrieve all refresh tokens associated with the user from Redis (e.g., `user:{user_id}:refresh_tokens`).
  2. **Delete each refresh token** from Redis (invalidates all active sessions).
  3. **Clear the user’s refresh token set** in Redis (optional, to clean up data).

**Example:**
```plaintext
# Key: user:123:refresh_tokens
# Value: {refresh_token1, refresh_token2, ...}
```

**Redis Commands:**
```plaintext
# For each token in the set:
DEL refresh_token:{token}
```

---

### 6. **Token Blacklisting vs. Token Rotation**

**Differences:**

| **Aspect**               | **Token Rotation**                          | **Token Blacklisting**                      |
|--------------------------|---------------------------------------------|---------------------------------------------|
| **Mechanism**            | Replace old token with a new one on each use. | Add invalid tokens to a blacklist.          |
| **Security**             | More secure (invalidates tokens proactively). | Less secure (tokens may be used until blacklisted). |
| **Implementation**       | Requires managing token replacements.       | Requires maintaining a blacklist.           |
| **Use Case**             | Preferred in modern systems for security.   | Used in legacy systems or when rotation is not feasible. |

**When to Use:**
- **Token Rotation:** Use in high-security systems where immediate token invalidation is critical (e.g., after password change, logout).
- **Token Blacklisting:** Use in systems with limited resources or legacy architectures where token rotation is not feasible. However, it is less secure and may require additional measures (e.g., short TTLs for tokens). 

---

### Summary

- **Refresh token rotation** ensures that each token is only valid once, preventing reuse.
- **Redis** is used to store and manage tokens with appropriate TTLs.
- **Password change** requires invalidating all sessions by deleting associated refresh tokens.
- **Token rotation** is more secure than blacklisting, but both have use cases depending on system requirements.

### 3. **N+1 Problem and Fix with DataLoader**

#### **Problem Example**
If a query fetches 10 recipes and their authors:

    title

    }
  }

#### **Fix with DataLoader**
Use a `DataLoader` to batch author lookups:

```javascript
const authorLoader = new DataLoader(async (recipeIds) => {
  const authors = await context.db.query(
    `SELECT * FROM authors WHERE id IN (${recipeIds.join(',')})`
  );
  return recipeIds.map(id => authors.find(a => a.id === id) || null);
});

// In the resolver for Recipe.author:
const authorResolver = (recipe) => {
  return authorLoader.load(recipe.authorId);
};
```

### 1. **Transport Mechanism: WebSocket**  
**Why?**  
WebSocket provides **full-duplex communication**, allowing the server to push notifications to clients in real-time without requiring clients to poll. This is ideal for scenarios like task assignments, comments, or deadline alerts, where immediacy is critical.  
- **Advantages over alternatives**:  
  - **SSE (Server-Sent Events)**: Unidirectional (server-to-client), which is sufficient for notifications but lacks bidirectional communication (e.g., client feedback).  
  - **Polling**: Inefficient for large user bases (100k+ users) due to high latency and bandwidth usage.  
  - **Push Notifications (e.g., FCM/FCP)**: Limited to mobile apps and require additional infrastructure.  
WebSocket is the **best fit for real-time, bidirectional updates** in a web-based tool.

### 2. **Database Schema for Notifications**  
**Schema Design**  
```sql
CREATE TABLE notifications (
    id UUID PRIMARY KEY,
    user_id UUID NOT NULL REFERENCES users(id),
    task_id UUID NOT NULL REFERENCES tasks(id),
    type ENUM('task_assigned', 'comment_added', 'deadline_approaching') NOT NULL,
    message TEXT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    is_read BOOLEAN DEFAULT FALSE,
    read_at TIMESTAMP,
    metadata JSONB -- Optional: Store additional context (e.g., task title, assigner)
);
```
**Key Fields**:  
- `user_id`: The user who should receive the notification.  
- `task_id`: Links the notification to a specific task.  
- `type`: Categorizes the notification (e.g., deadline approaching).  
- `is_read`: Tracks whether the user has seen the notification.  
- `metadata`: Stores optional details (e.g., task name, assigner, deadline time).

### 3. **Handling Offline Users**  
**Approach**:  
- **Queue Pending Notifications**: When a notification is generated (e.g., task assigned), it is **stored in the database** if the user is offline.  
- **Offline Detection**: Track user online/offline status using a `last_seen` timestamp in the `users` table.  
- **Reconnect Handling**:  
  - On reconnection, the client sends a request (e.g., `GET /notifications?since=last_seen`) to fetch all unacknowledged notifications.  
  - The server queries the `notifications` table for records where `created_at > last_seen` and `is_read = FALSE`, then sends them via WebSocket.  
- **Guaranteed Delivery**: Use a **message queue** (e.g., Redis) to buffer notifications for offline users. Workers can retry sending notifications until the user reconnects.

### 4. **Deadline Check Implementation**  
**Approach**: Use a **distributed task scheduler** (e.g., **Celery + Redis** or **Airflow**) to run the deadline check every hour.  
**Steps**:  
1. **Task**: A background job runs hourly to query all tasks where:  
   ```sql
   deadline >= NOW() AND deadline <= NOW() + INTERVAL '24 HOURS'
   ```  
2. **Avoid Duplicates**: Track the last notification time for each task in a `task_notifications` table:  
   ```sql
   CREATE TABLE task_notifications (
       task_id UUID PRIMARY KEY,
       last_deadline_notification TIMESTAMP
   );
   ```  
3. **Generate Notifications**: For tasks meeting the deadline condition and not yet notified, insert a new record into `notifications` and update `task_notifications`.  
4. **Scalability**: Use a **message queue** (e.g., Kafka) to distribute the workload across multiple workers.

### 5. **Scaling to 100,000 Concurrent Users**  
**Strategies**:  
1. **WebSocket Scaling**:  
   - Use a **WebSocket proxy** (e.g., **Nginx with WebSockets support**) to distribute connections across multiple backend servers.  
   - Implement **sticky sessions** to ensure users connect to the same server for consistency.  
2. **Message Queue for Notifications**:  
   - Use **Redis** or **Kafka** to buffer notifications. Workers process messages in parallel and send them via WebSocket.  
   - **Decouple** notification generation (e.g., task assignment) from delivery.  
3. **Database Sharding**:  
   - Shard the `notifications` table by `user_id` or `task_id` to handle high write/read throughput.  
   - Use **read replicas** for querying notifications.  
4. **Distributed Task Scheduler**:  
   - Use **Celery with Redis** or **Apache Airflow** for the hourly deadline check.  
5. **Caching**:  
   - Cache frequently accessed data (e.g., task deadlines) in **Redis** to reduce database load.
