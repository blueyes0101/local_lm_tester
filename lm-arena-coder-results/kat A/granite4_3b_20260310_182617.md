3. **Refresh Token Rotation with Redis**

To rotate the refresh token and store it in a Redis database, we can use a Redis hash data structure with a TTL (Time to Live). When a new refresh token is issued, it is stored in a Redis hash with a key that includes the user's ID and an expiration time.

```javascript
const redis = require('redis');

const rotateRefreshToken = async (userId) => {
  const redisClient = redis.createClient();
  const key = `refresh_token:${userId}`;

  // Generate a new refresh token
  const newRefreshToken = generateNewRefreshToken();

  // Store the refresh token in Redis with TTL
  redisClient.hset(key, 'refresh_token', newRefreshToken, 'expires', Math.floor(Date.now() / 1000) + TOKEN_TTL_SECONDS, (err) => {
```

6. **Token Blacklisting vs Token Rotation**

- **Token Blacklisting**: This involves maintaining a list of revoked or invalid tokens. When a token is blacklisted, it cannot be used for any requests. This method is simple but can be less efficient as it requires a check for every request against the blacklist.

- **Token Rotation**: This involves issuing a new token whenever a user authenticates or refreshes their access token. The old token is invalidated immediately, and the new token is stored securely, often in a fast-access store like Redis.

**Use Cases**:
- **Token Blacklisting**: Use when you need to quickly and easily revoke access for a specific token, such as in case of a user logging out or a compromised token.
- **Token Rotation**: Use for long-lived tokens or when you need to ensure that even if a token is compromised, the attacker cannot use it for a long period. This is particularly useful in high-security environments.

In general, token rotation is preferred for most scenarios due to its proactive approach to security, while token blacklisting is useful for immediate revocation scenarios.

### 2. Complete Component Implementation

Here is the complete DataTable component implementation:

```typescript
import React, { useMemo } from 'react';

const DataTable = <T>({ columns, data, pageSizeOptions, isSortable, isRowSelectable, onRowSelectAllChange, onPageSizeChange }: DataTableProps<T>) => {
  const [pageNumber, setPageNumber] = React.useState<number>(0);
  const [selectedRows, setSelectedRows] = React.useState<T[]>([]);
  const [sortBy, setSortBy] = React.useState<string | boolean>(false);

  const sortedData = useMemo(() => {
    if (!sortBy) return data;
    
    const sorted = [...data].sort((a, b) => {
      const col = columns.find(col => col.key === sortBy);
      if (!col) return 0;
      
      if (col.sortable) {
        if (col.sortable === 'asc') {
          return a[col.key] > b[col.key] ? -1 : 1;
        } else if (col.sortable === 'desc') {
          return a[col.key] < b[col.key] ? -1 : 1;
        }
      }
      return 0;
    });

    return sorted;
  }, [data, columns, sortBy]);

  const handlePageChange = (newPage: number) => {
    setPageNumber(newPage);
  };

  const handleSelectAllChange = (event: React.ChangeEvent<HTMLInputElement>) => {
    const isChecked = event.target.checked;
    setSelectedRows(isChecked ? data : []);
    onRowSelectAllChange?.(isChecked);
  };

  const handleRowSelect = (row: T) => {
    const newSelected = [...selectedRows, row];
    setSelectedRows(newSelected);
    onRowSelectAllChange?.(selectedRows.length === data.length);
  };

  const handleSortChange = (key: string | boolean) => {
    if (key === sortBy) {
      setSortBy(isSortable ? (sortBy === 'asc' ? 'desc' : 'asc') : false);
    } else {
      setSortBy(key);
    }
  };

  const paginatedData = useMemo(() => {
    return sortedData.slice((pageNumber - 1) * pageSize, pageNumber * pageSize);
  }, [sortedData, pageNumber]);

  return (
    <div>
      <select onChange={(e) => onPageSizeChange(e.target.value)}>
        {pageSizeOptions.map((option) => (
          <option key={option} value={option}>
            {option}
          </option>
        ))}
      </select>
      <table>
        <thead>
          <tr>
            {columns.map(col => (
              <th key={col.key} onClick={() => handleSortChange(col.key)}>
                {col.header}
                <div style={{ marginLeft: col.sortable ? '10px' : '0' }}>
                  {(col.sortable && sortBy === col.key) ? (col.sortable === 'asc' ? '▲' : '▼') : ''}
                </div>
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {paginatedData.map((row) => (
            <tr key={JSON.stringify(row)}>
              {columns.map(col => (
                <td key={col.key}>
                  {isSortable && col.sortable ? (
                    <span>
                      {col.renderCell?.(row) || JSON.stringify(row[col.key])}
                    </span>
                  ) : col.renderCell?.(row) || JSON.stringify(row[col.key])}
                </td>
              ))}
              <td>
                <input type="checkbox" onChange={(e) => handleRowSelect(row)} />
              </td>
            </tr>
          ))}
        </tbody>
      </table>
    </div>
  );
};
```

Canceling the in-flight requests ensures that the application always provides the most current data based on the user's latest input, preventing these kinds of bugs and ensuring a smoother user experience.

3. Broadcasting — when user A makes a change, how do you send it to all other users in the same document but NOT back to user A?
4. Reconnection — how do you handle a user rejoining after disconnect?

Here's a basic implementation of a collaborative document editor server using Node.js and the ws library:

```javascript
const WebSocket = require('ws');

// Create a WebSocket server
const wss = new WebSocket.Server({ port: 8080 });

// Store connected users and their document associations
const users = {};

// Message types
const MESSAGE_JOIN = 'JOIN';
const MESSAGE_TEXT_CHANGE = 'TEXT_CHANGE';
const MESSAGE_CURSOR_MOVE = 'CURSOR_MOVE';
const MESSAGE_USER_LEFT = 'USER_LEFT';

// Broadcast a message to all connected clients
function broadcast(message) {
  for (const user of users.values()) {
    if (user.readyState === WebSocket.OPEN) {
      user.send(JSON.stringify(message));
    }
  }
}

// Join a document
function joinDocument(ws, documentId) {
  users[ws] = { documentId };
  broadcast({ type: MESSAGE_JOIN, documentId });
}

// Text change
function textChange(ws, text) {
  users[ws].textChange = text;
  broadcast({ type: MESSAGE_TEXT_CHANGE, documentId: users[ws].documentId, text });
}

// Cursor move
function cursorMove(ws, position) {
  users[ws].cursorMove = position;
  broadcast({ type: MESSAGE_CURSOR_MOVE, documentId: users[ws].documentId, position });
}

// User left
function userLeft(ws) {
  const documentId = users[ws].documentId;
  delete users[ws];
  broadcast({ type: MESSAGE_USER_LEFT, documentId });
}

// Handle incoming messages
wss.on('connection', (ws) => {
  console.log('Client connected');
  ws.on('message', (message) => {
    const { type } = JSON.parse(message);
    switch (type) {
      case MESSAGE_JOIN:
        joinDocument(ws, JSON.parse(message).documentId);
        break;
      case MESSAGE_TEXT_CHANGE:
        textChange(ws, JSON.parse(message).text);
        break;
      case MESSAGE_CURSOR_MOVE:
        cursorMove(ws, JSON.parse(message).position);
        break;
      case MESSAGE_USER_LEFT:
        userLeft(ws);
        break;
    }
  });

  ws.on('close', () => {
    console.log('Client disconnected');
  });
});
```

This server handles the following functionality:

1. Connection handling: It uses a `users` object to map WebSocket connections (`ws`) to users and their associated document IDs.
2. Message protocol: It defines four message types: `JOIN`, `TEXT_CHANGE`, `CURSOR_MOVE`, and `USER_LEFT`.
3. Broadcasting: It broadcasts messages to all connected clients, except for the sender.
4. Reconnection: It automatically handles reconnections when a user disconnects, as it listens for the `close` event on the WebSocket connection and removes the user from the `users` object.
5. Conflict resolution: If two users edit the exact same position simultaneously, the server will receive both text change messages. The client-side implementation should handle these conflicts by choosing one version over the other or by merging the changes. Some common approaches are Operational Transformation (OT) or Conflict-free Replicated Data Types (CRDT).

This server implementation can be further enhanced with features like authentication, authorization, and persistence of document state.

**Q:** You are asked to add real-time notifications to the existing project management tool. Users should be notified when: a task is assigned to them, a comment is added to their task, a deadline is approaching (within 24h).

Design the notification system:
1. What transport mechanism would you use and why? (WebSocket, SSE, polling, push notifications — compare and justify)
2. How do you store notifications? Design the database schema.
3. How do you handle a user who is offline — do they get missed notifications when they come back online?
4. The deadline check needs to run every hour for all tasks. How do you implement this? (cron job, queue, event-driven?) — show the implementation approach.
5. How do you scale this if there are 100,000 concurrent users?

**A:**
1. **Transport Mechanism**: WebSocket is the most suitable transport mechanism for this system. Unlike traditional HTTP requests which are stateless and require a new connection for each request, WebSocket is a persistent connection that allows real-time, bidirectional communication between the client and the server. This makes it ideal for delivering real-time notifications to users as they are assigned to tasks, receive comments, or when deadlines are approaching. Additionally, WebSockets are more efficient in terms of resource usage than polling or long-polling techniques, as they only require a single TCP connection per user.

2. **Database Schema**: The database schema should be designed to efficiently store and retrieve notifications. A simple design could include a `notifications` table with the following fields: `id` (primary key), `user_id` (foreign key to the `users` table), `type` (e.g., task_assigned, comment_added, deadline_upcoming), `task_id` (foreign key to the `tasks` table if applicable), `timestamp` (when the notification was generated), and `read` (a boolean flag indicating whether the user has read the notification).

```sql
CREATE TABLE notifications (
  id INT AUTO_INCREMENT PRIMARY KEY,
  user_id INT,
  type VARCHAR(255),
  task_id INT,
  timestamp TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  read BOOLEAN DEFAULT FALSE
);
```

3. **Handling Offline Users**: For users who are offline, the system should buffer notifications locally and then deliver them once the user comes back online. This ensures that no notifications are missed. The system can achieve this by storing notifications in the user's `notifications` table with the `read` flag set to `FALSE`. When the user connects back to the server, the system can push these notifications to the user.

4. **Deadline Check Implementation**: The deadline check can be implemented using a combination of a scheduled task (cron job) and an event-driven system. The cron job can run every hour and check for any tasks that have deadlines within the next 24 hours. These tasks are then stored in a temporary queue and processed by the event-driven system. When the user checks their notifications, the system can retrieve these tasks from the queue and push them as notifications.

5. **Scaling for 100,000 Concurrent Users**: Scaling a WebSocket-based notification system for 100,000 concurrent users can be achieved by using a combination of load balancers, microservices, and a distributed database. The load balancers distribute incoming connections to multiple WebSocket servers, which can then handle multiple connections concurrently. Each WebSocket server can be stateless, which allows for horizontal scaling (adding more servers as needed). The database should also be horizontally scaled using sharding or partitioning to handle the increased load. Additionally, using a message broker like RabbitMQ or Kafka for the notification queue can help distribute the load evenly across the servers. The use of CDN can also help with reducing latency for users geographically distributed.
