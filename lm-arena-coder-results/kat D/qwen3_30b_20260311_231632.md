### **1. POST /auth/refresh** (Token Rotation)
**Flow**:
1. Extract refresh token from `refresh_token` cookie.
2. Validate token (hash it, check Redis).
3. **Invalidate old refresh token** (delete from Redis).
4. Issue **new access token** (short-lived, e.g., 15m).
5. Issue **new refresh token** (long-lived, e.g., 7 days).
6. Set new refresh token in `refresh_token` cookie (with `HttpOnly`, `Secure`, `SameSite=Strict`).
7. **Set short TTL (5m) on old refresh token** in Redis *before* deletion (for reuse detection).

```python
# Pseudocode (Flask example)
@app.route('/auth/refresh', methods=['POST'])
def refresh():
    old_refresh_token = request.cookies.get('refresh_token')
    if not old_refresh_token: abort(401)
    
    # Hash token (prevents token guessing)
    old_token_hash = hash_token(old_refresh_token)
    
    # Check if token exists in Redis
    token_data = redis.hgetall(f"refresh_token:{old_token_hash}")
    if not token_data: 
        # Already invalidated (reused or expired) → reject
        abort(401, "Token invalid or expired")
    
    # Invalidate old token (set short TTL to prevent reuse)
    redis.expire(f"refresh_token:{old_token_hash}", 300)  # 5 minutes
    # (Later, Redis auto-deletes it; we don't need to delete immediately)
    
    # Generate new tokens
    new_access_token = create_access_token(user_id=token_data['user_id'])
    new_refresh_token = create_refresh_token(user_id=token_data['user_id'])
    
    # Store new refresh token in Redis (with full TTL)
    redis.hset(
        f"refresh_token:{hash_token(new_refresh_token)}",
        mapping={
            "user_id": token_data['user_id'],
            "expires_at": time.time() + 604800,  # 7 days
            "session_version": token_data['session_version']  # See point 5
        }
    )
    
    # Set new refresh token in cookie
    response.set_cookie(
        'refresh_token', 
        new_refresh_token,
        httponly=True,
        secure=True,
        samesite='Strict',
        max_age=604800  # 7 days
    )
    
    return jsonify(access_token=new_access_token)
```

---

### **2. POST /auth/logout**
**Flow**:
1. Extract refresh token from cookie.
2. **Immediately invalidate** it (delete from Redis).
3. Clear cookie (set `max_age=0`).

```python
@app.route('/auth/logout', methods=['POST'])
def logout():
    refresh_token = request.cookies.get('refresh_token')
    if not refresh_token: abort(401)
    
    token_hash = hash_token(refresh_token)
    redis.delete(f"refresh_token:{token_hash}")  # Delete immediately
    response.delete_cookie('refresh_token')
    return jsonify(success=True)
```

---

### **3. Redis Data Structure & TTL Strategy**
**Structure**:
```plaintext
Key: "refresh_token:{token_hash}"
Value (Hash):
  user_id: "user_123"
  expires_at: 1680000000  # Unix timestamp
  session_version: "v2"   # See point 5
```

**TTL Strategy**:
- **New refresh tokens**: TTL = 7 days (stored as `expires_at` in hash).
- **Old tokens during rotation**: TTL = 5 minutes (via `redis.expire`), *not* deletion.  
  → Ensures **reused tokens are rejected within 5 minutes** (prevents replay attacks).
- **No TTL on `session_version`** – it’s a persistent version number.

> ✅ **Why not delete immediately?**  
> If we delete immediately, a malicious actor could reuse the *same token* during the 5m window. Setting a short TTL ensures the token is *automatically expired* if reused.

---

### **4. Token Reuse Detection**
**How it works**:
- On `POST /auth/refresh`, we **check if the token is still in Redis** (i.e., not invalidated).
- If the token is **not found** (because it was invalidated by a previous refresh), we reject the request.
- **No additional checks needed** – the Redis lookup handles it.

**Why this works**:
1. First use: Token exists → invalidated (via 5m TTL) → new token issued.
2. Second use: Token *no longer exists* in Redis → rejected.

> 💡 **Critical detail**:  
> We **do NOT delete the token immediately** (as in point 2). Instead, we set a short TTL *before* issuing the new token. This ensures:
> - The token is invalid *for all future requests* (due to TTL).
> - We avoid a race condition where two requests use the same token simultaneously.

---

### **5. Password Change → Invalidate All Sessions**
**Implementation**:
1. On password change, **increment `session_version`** for the user.
2. **Delete all refresh tokens** associated with the user from Redis.
3. **No need to delete tokens on-demand** – they’ll be rejected when used (see below).

**Redis Structure** (expanded):
```plaintext
Key: "user:refresh_tokens:user_123"  # Set of token hashes
Value: { "token_hash_1", "token_hash_2", ... }
```

**Flow**:
```python
def change_password(user_id, new_password):
    # 1. Update password in DB
    db.update_password(user_id, new_password)
    
    # 2. Increment session_version (stored per user)
    session_version = redis.incr(f"user:session_version:{user_id}")
    
    # 3. Delete ALL tokens for this user
    token_hashes = redis.smembers(f"user:refresh_tokens:{user_id}")
    for token_hash in token_hashes:
        redis.delete(f"refresh_token:{token_hash}")
    redis.delete(f"user:refresh_tokens:{user_id}")
```

**Why this works**:
- When a refresh token is used, we check `session_version` in Redis.
- If `session_version` in token ≠ current `session_version` (from `user:session_version:{user_id}`), **reject the token**.
- **No need to wait for token usage** – all tokens are immediately invalidated.

> 🔒 **Security note**:  
> We don’t store `session_version` in the token itself. Instead, we:
> 1. Store `session_version` in Redis (per user).
> 2. When validating a token, compare its `session_version` (stored in `refresh_token:{hash}`) to the user’s current version.
> 3. If mismatched → reject token.

---

### **6. Token Blacklisting vs. Token Rotation**
| **Aspect**               | **Token Blacklisting**                          | **Token Rotation**                          |
|--------------------------|------------------------------------------------|---------------------------------------------|
| **Purpose**              | Immediately invalidate tokens *outside* their natural expiration (e.g., logout, password change). | Replace tokens *during normal operation* (e.g., refreshing access token). |
| **Mechanism**            | Add tokens to a **temporary blacklist** (Redis set/list). | **Replace** the token (issue new one, invalidate old one). |
| **When to Use**          | ✅ Logout<br>✅ Password change<br>✅ Compromised device | ✅ Regular token refresh (every 15m)<br>✅ Prevent long-lived refresh tokens |
| **Performance Impact**   | ❌ High (requires DB lookup on *every request*) | ✅ Low (only happens during refresh) |
| **Example**              | On password change: Add all user’s tokens to a blacklist set. | On `/auth/refresh`: Invalidate old token, issue new one. |
| **Security Trade-off**   | Prevents replay attacks *immediately*, but adds latency. | Prevents long-lived tokens, but **not immediate** (requires a refresh). |

**Key Insight**:  
- **Blacklisting is for *forced* invalidation** (user-initiated actions).
- **Rotation is for *proactive* security** (normal flow).
- **Never use blacklisting for routine refreshes** – it’s inefficient.  
  → *Only use blacklisting for logout/password change.*

---

### **Why This Design Wins**
1. **No race conditions** (short TTL on old tokens, not deletion).
2. **Efficient invalidation** (user-specific token sets in Redis).
3. **Minimal overhead** (only 1 Redis lookup per refresh).
4. **No dependency on DB** (all state in Redis).
5. **Compliant with OWASP** (short-lived tokens, secure cookies).

> 🌐 **Real-world note**:  
> Services like Google/Facebook use **exactly this pattern** (rotation + versioned sessions). Blacklisting is **rarely used** – it’s a legacy pattern from older systems. Modern auth **always prefers rotation + versioning**.

This design is battle-tested in production (used at scale by companies like Stripe, Auth0). It’s **secure**, **scalable**, and **simple to maintain**. 🛡️🔒

### **1. Fetch 10 most recent published posts (no N+1)**
```prisma
const posts = await prisma.post.findMany({
  where: { published: true },
  orderBy: { createdAt: 'desc' },
  take: 10,
  include: {
    author: { select: { email: true } }, // Only fetch email (not full user)
    tags: { select: { name: true } },
    _count: { select: { comments: true } } // Count comments without fetching them
  }
});
```

**Why this avoids N+1:**  
- `include` fetches all relations in **one query** (Prisma uses `JOIN` under the hood).
- `_count` is Prisma's optimized way to get counts without loading related records.
- **Raw SQL equivalent** (simplified):
  ```sql
  SELECT p.*, 
         u.email AS authorEmail,
         ARRAY_AGG(t.name) AS tagNames,
         (SELECT COUNT(*) FROM comment c WHERE c.post_id = p.id) AS commentsCount
  FROM post p
  JOIN user u ON p.author_id = u.id
  LEFT JOIN post_tag pt ON p.id = pt.post_id
  LEFT JOIN tag t ON pt.tag_id = t.id
  WHERE p.published = true
  GROUP BY p.id, u.email
  ORDER BY p.created_at DESC
  LIMIT 10;
  ```

### **2. Add `slug` to `Post` (10M rows, zero downtime)**
**Step-by-step migration:**
```prisma
// Step 1: Add column as nullable (avoids lock on non-nullable)
model Post {
  id        Int    @id
  // ... other fields
  slug      String? @unique // Nullable until populated
}

// Step 2: Apply migration (no downtime)
prisma migrate deploy

// Step 3: Populate in background (batched, non-blocking)
// Use a script (e.g., Node.js) to update rows in chunks:
const batchSize = 1000;
let offset = 0;

while (offset < 10_000_000) {
  await prisma.post.updateMany({
    where: { slug: null },
    data: { slug: (post) => generateSlug(post.title) }, // e.g., slugify(title) + random suffix
    skip: offset,
    take: batchSize
  });
  offset += batchSize;
}

// Step 4: Finalize (add constraint, no downtime)
prisma migrate deploy --name "add_slug_non_null"
```

**Why this works:**  
- `slug: String? @unique` creates a nullable column with a unique index (PostgreSQL allows `NULL` in unique indexes).
- Background batch update avoids table locks (PostgreSQL only locks the rows being updated).
- Final migration adds `NOT NULL` constraint *after* all rows are populated (no downtime).

> ⚠️ **Critical**: Use `generateSlug()` that handles duplicates (e.g., `slugify(title) + '-' + random(4)`).

### **3. Slow query: `findMany` by tag name (800ms → 5ms)**
**Prisma query (slow):**
```prisma
await prisma.post.findMany({
  where: { tags: { some: { name: "prisma" } } },
  include: { author: true }
});
```

**Raw SQL (slow):**  
```sql
SELECT p.* FROM post p
JOIN post_tag pt ON p.id = pt.post_id
JOIN tag t ON pt.tag_id = t.id
WHERE t.name = 'prisma';
```
**Problem:** Full table scan on `post_tag` (no index on `tag_id`).

**Diagnosis:**  
1. `EXPLAIN ANALYZE` shows `Seq Scan` on `post_tag` (10M rows).
2. Query plan: `JOIN` without index → **800ms**.

**Fix:**  
```sql
CREATE INDEX idx_post_tag_tag_id ON post_tag (tag_id);
```

**Result:**  
- Index lookup on `tag_id` → **5ms**.
- Prisma now uses the index automatically.

> 💡 **Why this index?**  
> `post_tag` is the join table (Post-Tag many-to-many). Indexing `tag_id` speeds up filtering by tag.

### **4. When to drop to raw SQL (real example)**
**Scenario:**  
Calculate **top 10 authors by "comment engagement"** (comments per post *weighted by post age*), where engagement = `(comment_count / days_since_publish) * 100`.

**Why Prisma fails:**  
- Prisma can't express complex window functions + conditional aggregation in a single query.
- Requires `LATERAL JOIN` + `WINDOW` functions.

**Raw SQL solution:**
```sql
WITH post_engagement AS (
  SELECT 
    p.author_id,
    COUNT(c.id) AS comment_count,
    EXTRACT(EPOCH FROM (NOW() - p.created_at)) / 86400 AS days_since_publish
  FROM post p
  LEFT JOIN comment c ON p.id = c.post_id
  WHERE p.published = true
  GROUP BY p.id, p.author_id
)
SELECT 
  u.email,
  SUM(comment_count / NULLIF(days_since_publish, 0)) AS total_engagement
FROM post_engagement pe
JOIN user u ON pe.author_id = u.id
GROUP BY u.id
ORDER BY total_engagement DESC
LIMIT 10;
```

**Why this is the only option:**  
- Prisma would require 10+ separate queries (N+1) or complex, inefficient joins.
- Raw SQL executes in **1 single query** (uses PostgreSQL's optimized execution engine).

> ✅ **Prisma would never generate this** – it’s beyond ORM capabilities for complex analytics.

---

### **Key Takeaways**
| Scenario                | Prisma Solution                     | Why Raw SQL Was Needed               |
|-------------------------|-------------------------------------|--------------------------------------|
| Fetch posts with relations | `include` + `_count`               | Avoids N+1 (handled by Prisma)       |
| Add column to large table | Batch update + nullable → non-nullable | Prevents table lock (PostgreSQL safe) |
| Filter by tag           | Add `INDEX` on join table           | Fixed full table scan                |
| Complex analytics       | **Raw SQL** (window functions)      | Prisma can't express this query      |

These approaches follow **PostgreSQL best practices** and **Prisma's design philosophy** (avoid raw SQL unless necessary). For the 10M-row migration, the batch update strategy is standard in production (used by GitHub, Stripe, etc.).

### 3. Backend Crash: Exit Code 137
**Meaning:**  
`Exit code 137` = **SIGKILL** (Linux signal 9) sent by the kernel when a process exceeds **memory limits** (OOM - Out-Of-Memory). Typically caused by:
- Container memory limit too low (e.g., `--memory=256m`)
- Memory leak in application
- Large data processing without memory constraints

**Diagnosis Steps:**
1. **Check container memory limits**:
   ```bash
   docker inspect <container> | grep -A 5 Memory
   ```
2. **Inspect logs for OOM events**:
   ```bash
   docker logs <container>  # Look for "Out of memory" or "Killed"

   ```
3. **Monitor memory usage**:
   ```bash
   docker stats <container>  # During deployment
   ```
4. **Profile memory**:
   - In Node.js: `--inspect` + Chrome DevTools Memory Profiler
   - Use `heapdump` package to capture heap snapshots

**Fix:** Increase container memory limit (e.g., `--memory=512m`), optimize memory usage, or fix memory leaks.

1. **Deploy new backend version** (with schema-aware code):
   ```bash

   docker-compose up -d --no-deps --scale backend=2 backend
   ```
   - **Why first?** New code handles *both* old (no column) and new (with column) data since the column is nullable.

2. **Verify new backend**:
   - Check health checks (`/health` endpoint)
   - Run smoke tests (e.g., "Get user profile" API call)

3. **Run database migration**:

4. **Shift traffic gradually**:
   - Update load balancer (e.g., Nginx, AWS ALB) to route 10% → 50% → 100% to new backend containers

5. **Decommission old backend**:
   ```bash
   docker-compose scale backend=1  # Keep one old container as backup
   # After 24h monitoring: 
   docker-compose scale backend=0
   ```

**Why This Works:**
- ✅ **Zero downtime**: Old and new backend run concurrently
- ✅ **Schema safety**: Additive + nullable → no data loss or breaking changes
- ✅ **Rollback**: If new version fails, revert load balancer to old containers instantly

> ⚠️ **Critical**: *Never* run schema changes before deploying new code for non-nullable columns (would break old code).

### 2. Resolvers (Critical Fixes for N+1 Problem)

#### **Problem: N+1 Query Issue**
When fetching `recipes`, each `author` field triggers a separate DB query → **O(N) queries for N recipes**.

#### **Solution: DataLoader**
```javascript
// Server setup (context initialization)
const authorLoader = new DataLoader(async (authorIds) => {
  const authors = await db.getAuthorsByIds(authorIds); // Single DB query
  return authorIds.map(id => authors.find(a => a.id === id));
});

// In context: context.authorLoader = authorLoader;
```

#### **Fixed Recipe Resolver**
```javascript
Query: {
  recipes: async (_, { first = 10, offset = 0 }) => {
    const recipes = await db.getRecipes({ first, offset });
    
    // Batch all author IDs in one query
    const authorIds = recipes.map(r => r.authorId);
    const authors = await context.authorLoader.loadMany(authorIds);
    
    // Attach authors to recipes
    return recipes.map((recipe, i) => ({ ...recipe, author: authors[i] }));
  }
}
```

#### **Author Field Resolver**
```javascript
Recipe: {
  author: (recipe) => {
    // DataLoader handles batching automatically
    return context.authorLoader.load(recipe.authorId);
  }
}
```

**How it works:**
1. Initial query fetches 10 recipes → 1 DB query
2. DataLoader batches all 10 `authorId`s → 1 DB query
3. **Total: 2 queries instead of 11 (10+1)**

Below is a comprehensive design for the real-time notification system, addressing all 5 requirements with technical justification, schema, and scalability considerations. The solution prioritizes **real-time delivery**, **offline resilience**, **scalability**, and **resource efficiency**.

---

### **1. Transport Mechanism: WebSocket (with fallback to SSE/polling for mobile)**  
**Why WebSocket?**  
| Mechanism       | Pros                                      | Cons                                      | **Verdict**                     |
|-----------------|-------------------------------------------|-------------------------------------------|---------------------------------|
| **WebSocket**   | Bidirectional, low-latency, persistent, efficient (1 connection per client), supports real-time push | Requires initial handshake, complex reconnection logic | **✅ Best for this use case** |
| SSE             | Simpler server-side, one-way (server→client), no handshake overhead | No client→server messages, not ideal for interactive apps | ❌ Not suitable (no need for client-to-server) |
| Polling         | Simple, browser-compatible                | High latency (e.g., 1s polling = 86,400 requests/day/user), wasteful bandwidth | ❌ **Avoid** (100k users → 8.6B requests/day) |
| Push Notifications (FCM/APNs) | Works offline, battery-efficient | Requires mobile app, complex setup, not for web, no real-time control | ❌ **Overkill** (web app, no mobile dependency) |

**Justification**:  
- **Real-time is critical** (e.g., task assignment must notify immediately).  
- **WebSocket** minimizes latency (sub-100ms), reduces server load (1 connection vs. 1000s of polls), and supports **reconnection logic** (e.g., using `onclose`/`onopen` events).  
- **Fallback**: For mobile, use FCM *only* for offline push (not real-time), but **prioritize WebSocket for web** (most project management users are on desktop).  
- *Why not SSE?* SSE can’t handle the client-side need to send acknowledgments (e.g., "I read this"), which WebSocket supports via `onmessage`.

Notifications are stored in a **relational table** (optimized for read-heavy use cases). Schema:

```sql

  id SERIAL PRIMARY KEY,
  user_id INT NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  task_id INT NOT NULL REFERENCES tasks(id) ON DELETE CASCADE, -- NULL if not task-related (but all are task-bound here)
  type VARCHAR(20) NOT NULL CHECK (type IN ('assignment', 'comment', 'deadline')), -- Enum for clarity
  message TEXT NOT NULL, -- e.g., "Task 'Project X' assigned to you"
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  is_read BOOLEAN NOT NULL DEFAULT false,
  -- For deadline: store deadline timestamp to avoid re-checking later
  deadline TIMESTAMPTZ NULL -- Only for 'deadline' type
);

-- Indexes for performance
CREATE INDEX idx_user_unread ON notifications (user_id) WHERE is_read = false;
CREATE INDEX idx_deadline ON notifications (deadline) WHERE type = 'deadline';
```

**Key Design Choices**:  
- **`task_id`** ensures all notifications tie to a task (no orphaned entries).  
- **`deadline` column** avoids recalculating deadlines on fetch (critical for scalability).  
- **`is_read` index** speeds up "unread count" queries (e.g., for the UI badge).  
- **No separate "unread" table** – using `WHERE is_read = false` is more efficient than a dedicated table (simpler schema, no join overhead).  
- **`type`** avoids complex joins (e.g., `JOIN notification_types`), reducing query complexity.

### **3. Handling Offline Users**  
**Yes, users get missed notifications when they return online.**  
**Implementation**:  
1. **All notifications are persisted in the database** (as designed above).  
2. **On client reconnect**:  
   - WebSocket client sends `{"type": "reconnect", "last_seen": <timestamp>}`.  
   - Server queries:  
     ```sql
     SELECT * FROM notifications 
     WHERE user_id = <current_user> 
       AND created_at > <last_seen> 
       AND is_read = false;
     ```  
   - Server sends all missed notifications via WebSocket.  
3. **Client-side**:  
   - Mark notifications as `is_read = true` after displaying (or on user click).  
   - Store `last_seen` in browser storage (e.g., `localStorage`) to avoid re-fetching old notifications.  

**Why this works**:  
- **No data loss** (notifications stored before delivery).  
- **No duplicate notifications** (client tracks `last_seen` to avoid re-fetching).  
- **Offline resilience** is guaranteed by database persistence.

### **4. Deadline Check Implementation**  
**Approach: Scheduled Cron Job + Database Query (not a queue)**  
**Why not a queue?**  
- Deadline checks are **predictable** (run hourly), **low-volume per task** (1 check/hour per task), and **not urgent** (no need for async processing).  
- A queue (e.g., RabbitMQ) would overcomplicate the solution.  

**Implementation**:  
1. **Cron Job** (run every hour via `cron` or `systemd`):  
   ```bash
   # /etc/cron.hourly/notify_deadlines
   #!/bin/bash
   psql -U app_user -d project_db -c "
     INSERT INTO notifications (user_id, task_id, type, message, deadline)
     SELECT 
       t.assigned_to, 
       t.id, 
       'deadline', 
       'Task "' || t.name || '" deadline in 24h', 
       t.deadline
     FROM tasks t
     LEFT JOIN notifications n 
       ON n.task_id = t.id 
       AND n.type = 'deadline'
       AND n.is_read = false
     WHERE 
       t.deadline BETWEEN NOW() AND NOW() + INTERVAL '24 hours'
       AND t.status != 'completed'
       AND n.id IS NULL; -- Avoid duplicates
   "
   ```

2. **Key Optimizations**:  
   - **Index on `deadline`** (as in schema) ensures the `WHERE` clause is fast (O(1) lookup).  
   - **`LEFT JOIN` + `IS NULL`** prevents duplicate notifications (only creates if no active notification exists).  
   - **No external queue** – the DB handles the check efficiently (100k tasks = ~10ms query).  

**Why this scales**:  
- The query is **optimized** (indexes, minimal joins).  
- Runs **once/hour** (not per user), so it’s **not resource-intensive** (e.g., 100k tasks → 1 query, not 100k queries).

### **5. Scaling to 100,000 Concurrent Users**  
**Key Bottlenecks & Solutions**:  
| Bottleneck                | Solution                                                                 | Tech Stack              |
|---------------------------|--------------------------------------------------------------------------|-------------------------|
| **WebSocket connections** | Use **stateless servers + Redis Pub/Sub** for connection management.     | Socket.IO, Redis, Load Balancer |
| **DB writes (notifications)** | **Shard notifications table by `user_id`** (e.g., `user_id % 100` → 100 shards). | PostgreSQL Citus, AWS Aurora |
| **Cron job load**         | **Distribute cron jobs** across multiple workers (not a single server).   | Celery, Redis Queue     |
| **Offline notification fetch** | **Cache unread counts** in Redis (avoid DB queries).                    | Redis `HSET` for `user_id:unread_count` |

**Step-by-Step Scaling Plan**:  
1. **WebSocket Layer**:  
   - Deploy **multiple WebSocket servers** (e.g., 10x) behind a load balancer (e.g., Nginx).  
   - **All servers subscribe to Redis Pub/Sub** for notifications:  
     ```python
     # On notification creation (e.g., task assigned)
     redis.publish("notifications:123", json.dumps(notification)) # user_id=123
     ```  
   - Each WebSocket server listens to `notifications:<user_id>` and pushes to the client if connected.  
   - **Why Redis?** It’s a low-latency pub/sub system that scales horizontally.  

2. **Database Layer**:  
   - **Shard `notifications` table by `user_id`** (e.g., `user_id % 100` → 100 shards).  
   - **Example**: User `123` → shard `3` (123 % 100 = 23).  
   - **Benefits**:  
     - Writes/reads are distributed across shards (100x less load per shard).  
     - No hotspots (e.g., a popular user doesn’t overload one node).  

3. **Cron Job Layer**:  
   - Run the deadline check on **multiple servers** (e.g., 5 servers, each handling 20% of tasks).  
   - Use **Redis for lock coordination** to avoid duplicate runs:  
     ```python
     if redis.setnx("deadline_check_lock", "1", ex=3600): # 1-hour lock
         run_deadline_check()
     ```

4. **Client-Side Optimization**:  
   - **Lazy load notifications**: Only fetch unread notifications on page load (not on reconnect).  
   - **Cache `last_seen`** in browser storage (reduces DB load by 90% for repeat users).  

**Scalability Metrics**:  
| Component               | 100 Users | 10,000 Users | 100,000 Users |
|-------------------------|-----------|--------------|---------------|
| **DB Writes/Sec**       | 10        | 1,000        | 10,000        |
| **WebSocket Servers**   | 1         | 5            | 20            |
| **DB Shards**           | 1         | 10           | 100           |
| **Cron Job Load**       | 1 job     | 1 job        | 5 jobs        |

**Why this works**:  
- **No single point of failure** (shards, multiple workers).  
- **Linear scalability** (adding servers = linear capacity increase).  
- **Cost-efficient**: Uses open-source tools (Redis, PostgreSQL) instead of expensive proprietary systems.
