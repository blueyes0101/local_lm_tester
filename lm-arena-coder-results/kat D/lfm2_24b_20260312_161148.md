6. What happens if someone sends an expired token? Show the exact error response.

```typescript
// auth.controller.ts
import express, { Request, Response, NextFunction } from 'express';
import bcrypt from 'bcrypt';
import jwt from 'jsonwebtoken';
import { getRepository } from 'typeorm';
import User from '../models/User';

const JWT_SECRET = process.env.JWT_SECRET || 'your-secret-key';
const REFRESH_TOKEN_SECRET = process.env.REFRESH_TOKEN_SECRET || 'refresh-secret';
const ACCESS_TOKEN_EXPIRES = '15m';
const REFRESH_TOKEN_EXPIRES = '7d';

// 1. Register endpoint
export const register = async (req: Request, res: Response) => {
  const { email, password } = req.body;
  if (!email || !password) {
    return res.status(400).json({ error: 'Email and password required' });
  }

  const userRepository = getRepository(User);
  const existingUser = await userRepository.findOne({ where: { email } });
  if (existingUser) {
    return res.status(400).json({ error: 'Email already registered' });
  }

  const hashedPassword = await bcrypt.hash(password, 12);
  const user = userRepository.create({
    email,
    password: hashedPassword,
    role: 'user' // Default role
  });
  await userRepository.save(user);

  // Generate tokens
  const accessToken = jwt.sign(
    { userId: user.id, email: user.email, role: user.role, iat: Date.now() },
    JWT_SECRET,
    { expiresIn: ACCESS_TOKEN_EXPIRES }
  );
  const refreshToken = jwt.sign(
    { userId: user.id },
    REFRESH_TOKEN_SECRET,
    { expiresIn: REFRESH_TOKEN_EXPIRES }
  );

  res.json({ user: { id: user.id, email: user.email, role: user.role } });
};

// 2. Login endpoint
export const login = async (req: Request, res: Response) => {
  const { email, password } = req.body;
  const userRepository = getRepository(User);
  const user = await userRepository.findOne({ where: { email } });

  if (!user || !(await bcrypt.compare(password, user.password))) {
    return res.status(401).json({ error: 'Invalid credentials' });
  }

  const accessToken = jwt.sign(
    { userId: user.id, email: user.email, role: user.role, iat: Date.now() },
    JWT_SECRET,
    { expiresIn: ACCESS_TOKEN_EXPIRES }
  );
  const refreshToken = jwt.sign(
    { userId: user.id },
    REFRESH_TOKEN_SECRET,
    { expiresIn: REFRESH_TOKEN_EXPIRES }
  );

  res.cookie('refreshToken', refreshToken, {
    httpOnly: true,
    secure: process.env.NODE_ENV === 'production', // HTTPS in production
    sameSite: 'strict',
    maxAge: REFRESH_TOKEN_EXPIRES * 1000
  });

  res.json({ accessToken });
};

// 3. Protected route
export const getMe = async (req: Request, res: Response) => {
  res.json({ user: req.user });
};

// 4. Auth middleware
export const authMiddleware = (req: Request, res: Response, next: NextFunction) => {
  const token = req.headers.authorization?.replace('Bearer ', '');
  if (!token) {
    return res.status(401).json({ error: 'Missing token' });
  }

  try {
    const decoded = jwt.verify(token, JWT_SECRET) as { userId: number; email: string; role: string; iat: number };
    req.user = decoded;
    next();
  } catch (error) {
    res.status(401).json({ error: 'Invalid or expired token' });
  }
};

// 5. Error response for expired token
```

**Security Explanation:**

1. **Expired Token Response:**
   When an expired token is sent, the server returns:
   ```json
   {
     "error": "Invalid or expired token"
   }
   ```
   The JWT library throws an error which we catch and return a standardized response.

2. **Why httpOnly Cookies for Refresh Tokens?**
   - **XSS Protection:** HttpOnly cookies cannot be accessed by client-side JavaScript, preventing XSS attacks from stealing tokens.
   - **CSRF Mitigation:** When using SameSite=Strict/Lax, cookies are not sent with cross-site requests.
   - **Secure Transmission:** Cookies with Secure flag (in production) ensure HTTPS-only transmission.
   - **LocalStorage Risks:** Storing tokens in localStorage exposes them to XSS vulnerabilities, allowing attackers to steal tokens via script injection.

**Key Security Practices:**
- Passwords are hashed with bcrypt (cost factor 12)
- Access tokens have short expiration (15 minutes)
- Refresh tokens have longer expiration (7 days) and are stored securely
- JWT payload contains minimal necessary data
- Secure cookie flags (HttpOnly, SameSite, Secure in production)

This implementation follows OAuth 2.0 best practices for token-based authentication while balancing security and usability.

### 1. **POST /auth/refresh — Refresh Token Logic**

- **Input:** Refresh token from cookie.
- **Process:**
  - Validate the refresh token (check if it exists in Redis and is not expired).
  - If valid, generate a new access token (short-lived).
  - Rotate the refresh token: store a new refresh token in Redis, invalidate the old one.
  - Set the new refresh token in the cookie.
- **Response:** New access token and (optionally) new refresh token.

---

### 2. **POST /auth/logout — Invalidate Refresh Token**

- **Action:** Remove the refresh token from Redis.
- **Cookie:** Clear the refresh token cookie.

---

### 3. **Redis Data Structure & TTL Strategy**

**Redis Key Structure:**
```plaintext
refresh_token:<hash]:<user_id>  ->  { "user_id": "...", "exp": <timestamp> }
```
- **Key:** `refresh_token:<hash>:<user_id>`
- **Value:** JSON object with user ID and expiration timestamp.

**TTL Strategy:**
- Set expiration on Redis key to match refresh token lifetime (e.g., 7 days).
- When refreshing, issue a new token with a new TTL.

**Example:**
```python
# On refresh
redis.setex(f"refresh_token:{refresh_token_hash}", 60*60*24*7, json.dumps({"user_id": user_id}))
```

---

### 4. **Handling Duplicate Refresh Token Usage**

- **Detection:** On each refresh request, check if the provided refresh token exists in Redis.
- **If found:**
  - If it's already used (i.e., already rotated), reject and return error.
  - If not used, proceed and rotate.
- **Implementation:**
  - Store a `used` flag or a `version` number in Redis for each token.
  - On refresh, compare the provided token's version with stored version.

**Redis Example:**
```plaintext
refresh_token:<hash>:<user_id> -> { "user_id": "...", "version": 3, "exp": ... }
```
- On refresh, increment version and compare.

---

### 5. **Invalidate All Sessions on Password Change**

- **Approach:** On password change, invalidate all refresh tokens for that user.
- **Implementation:**
  - Store a `session_id` or `user_refresh_version` in Redis.
  - On password change, increment user's session version or delete all refresh tokens for that user.
  - On each refresh, check if the refresh token's user version matches current.

**Redis Example:**
```plaintext
user:<user_id>:refresh_version -> 5
refresh_token:<hash>:<user_id> -> { "user_id": "...", "version": 5, ... }
```
- On password change: increment `user:<user_id>:refresh_version`.

|--------------------------|--------------------------------------------------------------------------------|-------------------------------------------------------------------------------------|
| **Definition**           | Always issue a new token; old token becomes invalid.                            | Mark a token as invalid; old

### 4. Memoization Strategy

To avoid unnecessary re-renders:

1. **`React.memo` for Rows**: Wrap `renderRow` in `React.memo` if your `renderCell` is pure (no side effects). This prevents re-rendering rows unless their props change.

2. **`useCallback` for Handlers**: We already used `useCallback` for `handleSort`, `handleRowSelect`, and `handlePageChange`. This ensures that the function references don't change on every render, preventing child re-renders.

3. **`useMemo` for Derived Data**: `visibleData` is memoized with `useMemo`. This ensures it only recalculates when `data`, `currentPage`, or `pageSize` changes.

4. **Controlled vs Uncontrolled**: If `isControlled` is `false`, the component relies on props (`selectedRowIds`, `onRowSelect`, etc.) and does not manage its own state. This avoids internal state updates that would trigger re-renders.

5. **Key Prop**: Ensure each row has a stable `key` (e.g., `row.id`). This helps React efficiently update the DOM.

**Summary**: The key to performance here is ensuring that the `DataTable` component itself doesn't re-render unnecessarily. By memoizing handlers, using stable keys, and memoizing derived values, you achieve optimal performance.

```tsx
 */
export const useSearch = (query: string) => {
  const [results, setResults] = useState<any[]>([]);
  const [isLoading, setIsLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);
  const [retry, setRetry] = useState(() => {}, []); // Reset retry function on new query

  // 1. Caching: Store results with timestamp
  const cache = useRef<Map<string, { data: any[], timestamp: number }>>(new Map());

  // 2. Debounce Timer
  const timerRef = useRef<NodeJS.Timeout | null>(null);

  // 3. Cancellation Logic
  const controller = useRef<AbortController | null>(null);

  // 4. Effect to handle the search logic
  useEffect(() => {
    // Clear previous timer on new query
    if (timerRef.current) {
      timerRef.current.cancel();
    }

    // Check cache first (within 5 minutes)
    const cached = cache.current.get(query);
    const now = Date.now();
    if (cached && (now - cached.timestamp) < 5 * 60 * 1000) {
      setResults(cached.data);
      setError(null);
      return;
    }

    // If there's an in-flight request, cancel it
    if (controller.current) {
      controller.current.abort(); // This will cause mockFetch to reject on abort
    }

    // Start new request
    const fetchData = async () => {
      setIsLoading(true);
      setError(null);

      controller.current = new AbortController();
      const { signal } = controller.current;

      try {
        const data = await mockFetch(query, signal); // Pass abort signal
        setResults(data);

        // Update cache
        cache.current.set(query, { data, timestamp: now });
      } catch (err: any) {
        // Ignore abort errors (network failure)
        if (signal.aborted) {
          return;
        }
        setError(err.message || "Failed to fetch");
      } finally {
        setIsLoading(false);
      }
    };

    // Debounce
    timerRef.current = setTimeout(fetchData, 300);

    // Cleanup on unmount
    return () => {
      if (timerRef.current) {
        timerRef.current.cancel();
      }
      if (controller.current) {
        controller.current.abort();
      }
    };
  }, [query]); // Runs when query changes

  // Retry function
  const retrySearch = useCallback(() => {
    // Reset error and trigger new fetch
    setError(null);
    if (query) {
      // Reset cache for this query to force refetch (optional)
      cache.current.delete(query);
      // Trigger the effect again (handled by useEffect)
    }
  }, [query]);

  return { results, isLoading, error, retry };
};
```

### 4. Why Cancelling In-Flight Requests is Important

Cancelling in-flight requests is crucial to prevent a specific class of bugs known as **"Stale State"** or **"Race Condition"** bugs.

**The Bug:**
Imagine a user rapidly types "apple".

1.  **Typing "a"**: The hook starts a fetch (Request A).
2.  **Typing "ap"**: The hook cancels Request A and starts Request B.
3.  **Typing "app"**: The hook cancels Request B and starts Request C.

If we did not cancel the previous requests:
*   Request A might finish after Request C has already returned data.
*   When Request C arrives, it updates the state with the *new* query ("app"), overwriting the result of the *old* query ("a") or "ap".
*   The user sees the result for "app" (which might be empty or different), but the UI might have briefly shown the result for "a" before it disappeared.

**The Prevention:**
By cancelling the previous request, we ensure that only the *latest* query is processed. This guarantees that the UI state always matches the current user input. Without cancellation, you would see flickering or incorrect data as the results jump between different search terms.

### 3. Backend Crashing with Exit Code 137

**Meaning:**  
Exit code `137` means **"Termination signal 128 + exit code"**.  
Node.js receives a `SIGTERM` (or `SIGKILL` if not stopped gracefully) and exits immediately. This usually happens when:

- The process is killed by Docker (e.g., `docker stop`).
- A fatal unhandled exception occurs (e.g., `uncaughtException` or `process.on('unhandledRejection')`).
- A dependency crashes (e.g., database connection pool exhaustion, uncaught error in async code).
- The `process.exit()` is called with a non-zero code after an uncaught error.

**Diagnosis Steps:**

1. **Check Docker logs:**  
   ```bash
   docker logs <backend_container>
   ```
   Look for stack traces or error messages.

2. **Enable unhandled rejection/error handling:**  
   ```js
   process.on('unhandledRejection', (reason, promise) => {
     console.error('Unhandled Rejection at:', promise, 'Reason:', reason);
     // Optionally exit with 1 to force Docker to kill it
     process.exit(1);
   });
   process.on('uncaughtException', (err) => {
     console.error('Uncaught Exception:', err);
     // Graceful shutdown logic here
     process.exit(1);
   });
   ```

3. **Monitor resource usage:**  
   Check CPU/Memory with `docker stats` or `htop` inside the container.

4. **Test locally:**  
   Run `docker-compose up --build` and reproduce the crash.

### 3. The N+1 Problem

**The Problem:**
The `author` field in the `Recipe` type is resolved by looking up the author by ID. If you request a list of recipes and then resolve the `author` for each one, you make one query to get the recipes, then N additional queries to get the authors (one per recipe).

**Concrete Example:**
Suppose you fetch 5 recipes. The GraphQL query might look like:
```graphql
query {
  recipes {
    title
    author {
      name
    }
    ingredients {
      name
    }
  }
}
```
- **Step 1:** Fetch recipes (1 query).
- **Step 2:** For each recipe (5), fetch author (5 queries).
- **Total:** 6 queries.

**The Fix: Using DataLoader**
DataLoader batches and caches requests. Instead of fetching each author individually, we batch the requests.

```javascript
// Inside your server setup
const DataLoader = require('dataloader');

// Create a loader that fetches authors by IDs
const authorLoader = new DataLoader(async (authorIds) => {
  // Fetch all authors in one batch
  const allAuthors = authors.filter(a => authorIds.includes(a.id));
  const map = allAuthors.reduce((acc, a) => { acc[a.id] = a; return acc; }, {});
  return authorIds.map(id => map[id] || null);
});

// Then in the resolver:
Recipe: {
  author: (parent) => authorLoader.load(parent.author.id)
}
```
Now, fetching 5 recipes results in **2 queries**: 1 for recipes, 1 for authors (batching the 5 lookups).

### 5. When to Choose GraphQL vs. REST

**Choose GraphQL when:**
- The client needs flexibility (e.g., mobile app wants only specific fields).
- You want to avoid over-fetching (fetching only what you need).
- You have complex relationships (nested data).
- You want to evolve your API without versioning (adding fields doesn't break old clients).

*   **Why SSE for Mobile?** Mobile apps often have background processes. SSE is lighter than WebSocket for simple push scenarios and is natively supported by most mobile frameworks.
